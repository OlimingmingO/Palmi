## 一、产品概述

### 1.1 产品定位

小伴运营后台是团队内部使用的Web应用，服务于产品迭代与质量监控。核心理念：

> "不能看到数据就是瞎子——你无法改进你看不到的东西。"

没有运营后台的MVP就像蒙着眼走路——做了但不知道做得怎么样。运营后台让团队能亲眼看到用户与小伴的真实交互，基于数据做产品决策，而非基于"感觉"。

### 1.2 目标用户

团队规模2-5人，包括：
- 产品经理（主要使用者）
- CEO（战略决策）
- 技术负责人（数据模型与系统健康）

### 1.3 MVP核心能力

| 能力层次 | 能力 | 解决的问题 |
|---------|------|-----------|
| L1：看得到 | 对话记录浏览 + 全文搜索 | 能看到用户跟小伴聊了什么 |
| L2：看得懂 | 意图标签 + 话题统计 + 活跃度 | 理解用户在聊什么类型话题、谁活跃谁沉默 |
| L3：能决策 | 未满足需求识别 + 周报 | 知道下一步该做什么功能 |
| L4：能搜索 | 全文搜索 | 快速定位特定场景的对话 |
| L5：看得细 | 老人详情页（单人全维度视图） | 对单个老人的完整运营状态一目了然 |

### 1.4 产品边界

- ✅ 包含：对话数据浏览、分析、搜索、报告、单老人全维度详情
- ❌ 不包含：用户端功能修改、对话引擎调参、直接与老人沟通
- ❌ 不包含：对外可见的任何界面（纯团队内部工具）

## 二、用户画像

### 2.1 主要用户：产品经理

**角色特征**：
- 需要每天了解用户与小伴的真实对话质量
- 需要发现小伴"掉链子"的地方（未满足需求）
- 需要从数据中发现产品迭代方向
- 时间有限，每天最多投入10-15分钟在后台巡检

**核心痛点**：
- 不逐条看对话就不知道小伴回得好不好
- 不知道用户最常聊什么话题、需求分布如何
- 不知道哪些功能应该优先做

**期望**：打开后台10分钟内，能掌握产品当前状态和迭代方向

### 2.2 次要用户：CEO

**角色特征**：
- 关注高层趋势：用户增长、留存、核心指标
- 关注未满足需求——决定产品战略和融资故事
- 不需要看逐条对话，需要汇总视图

**核心痛点**：
- 需要快速了解"用户最需要什么我们还没做的"
- 需要数据支撑产品方向决策

### 2.3 次要用户：技术负责人

**角色特征**：
- 关注数据模型正确性（存储是否完整、标签是否准确）
- 关注系统健康度（存储延迟、标签处理队列）
- 偶尔通过后台排查对话异常

### 2.4 典型使用场景

| 场景 | 频率 | 时长 | 操作 | 获得 |
|------|------|------|------|------|
| 日常巡检 | 每天1次 | 10分钟 | 翻看当天对话、看未满足需求列表 | 感知用户状态、发现质量问题 |
| 需求发现 | 每周2-3次 | 15分钟 | 查看未满足需求聚合、话题统计 | 产品迭代优先级 |
| 对话质量优化 | 每周1-2次 | 20分钟 | 搜索特定关键词、查看标签下对话 | Prompt优化方向 |
| 产品决策 | 按需 | 30分钟 | 综合查看活跃度、话题趋势、未满足需求 | 数据驱动的功能决策 |

### F1: 对话记录持久化存储

**来源**：US-001 | **优先级**：P0 | **Story Points**：3

#### 描述

所有用户与小伴的对话（企业微信文字消息）被完整、结构化地存储，确保团队能在后台随时回溯任何对话，且数据可被后续分析模块（F2-F8）消费。这是整个运营后台的数据基础设施。

#### 输入

| 数据来源 | 内容 | 格式 |
|---------|------|------|
| 企业微信 | 用户发送的文字消息 | 文本 |
| 企业微信 | 小伴的文字回复 | 文本 |
| 系统元数据 | 时间戳、用户ID、渠道标记、会话ID | 结构化字段 |

#### 输出

| 输出内容 | 消费方 | 格式 |
|---------|--------|------|
| 结构化对话记录 | F2对话浏览器、F3全文搜索 | 数据库记录 |
| 带元数据的消息流 | F4意图标签、F7未满足需求 | 事件/队列 |

#### 数据模型

```
Session（一次完整对话）
├── session_id: 唯一标识
├── user_id: 用户标识
├── start_time: 对话开始时间
├── end_time: 对话结束时间
├── channel: 渠道（企微）
├── turn_count: 对话轮数
│
└── Turns（一轮问答）
    ├── turn_id: 轮次标识
    ├── sequence: 轮次序号
    │
    └── Messages（单条消息）
        ├── message_id: 消息唯一标识
        ├── sender: 发送者（user/xiaoban）
        ├── content: 消息内容（文字）
        ├── timestamp: 精确时间
        ├── channel: 渠道
```

**Session划分规则**：
- 企业微信：用户最后一条消息后超过30分钟无新消息，视为Session结束
- 同一天内同一用户可有多个Session

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 存储延迟 < 5秒 | 从对话轮次完成到数据落库，不超过5秒 |
| BR-3 | 元数据完整性 | 每条消息必须包含：content、timestamp、sender、channel、session_id |
| BR-4 | 失败重试机制 | 存储失败时消息进入重试队列，DB恢复后自动补录，保证零数据丢失 |
| BR-5 | 用户体验隔离 | 存储层异常绝不影响用户侧的正常对话体验（异步解耦） |
| BR-6 | 数据不可变 | 对话记录一旦写入不可修改（仅标签等衍生数据可追加） |
| BR-7 | 渠道标识必填 | 每条消息必须标明来自企微渠道 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 数据库不可达 | 消息进入本地重试队列，按指数退避策略重试，恢复后按时间顺序补录 |
| 超长单条消息（>10000字） | 正常存储，不做截断（MVP阶段几乎不会出现） |
| 同一秒内多条消息 | 按接收顺序存储，用消息ID保证顺序性 |
| 数据保留期限 | MVP阶段无自动删除机制，永久保留 |
| 用户首次对话 | 自动创建用户记录 + 首个Session |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 用户通过企业微信发送文字消息并收到小伴回复，When 该轮对话结束，Then 用户消息+小伴回复+完整元数据在5秒内写入存储 |
| AC-3 | Given 存储服务异常（DB不可达），When 新对话产生，Then 消息进入重试队列且不丢失，恢复后自动按序补录 |
| AC-4 | Given 存储层发生异常，When 用户正在与小伴对话，Then 用户侧对话体验不受任何影响 |
| AC-5 | Given 任意已存储的对话记录，When 查询该记录，Then 所有必要元数据字段（content/timestamp/sender/channel/session_id）完整无缺失 |

### F2: 对话记录浏览器

**来源**：US-002 | **优先级**：P0 | **Story Points**：5

#### 描述

Web界面供团队浏览完整的用户对话历史。以"用户列表 → 日期时间线 → 完整对话"的三层结构呈现，支持快速定位到任意用户任意日期的对话内容，判断对话质量和用户状态。

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 团队成员访问Web页面 | 浏览器（公网远程访问 Web Console (React)） | 简单密码认证后进入 |
| 对话记录数据 | F1存储层 | 查询已存储的结构化对话数据 |

#### 输出

| 输出 | 说明 |
|------|------|
| 用户列表视图 | 所有用户卡片，含关键元数据 |
| 日期时间线视图 | 选中用户后，按天分组展示对话摘要 |
| 对话详情视图 | 展开后的完整对话内容，含发送者和渠道标识 |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 用户列表信息 | 显示：昵称/称呼、最后活跃时间、总对话次数（Session数）、本周对话次数 |
| BR-2 | 默认排序 | 用户列表按最近活跃时间降序排列（最近活跃的在最前） |
| BR-3 | 日期分组 | 进入用户详情后，对话按自然日分组，每天显示：日期、当日Session数、当日总轮数、简要摘要（首轮对话的前20字） |
| BR-4 | 对话呈现 | 展开某天对话后，按时间正序显示全部消息，每条消息标注：发送者标识（用户/小伴）、精确时间 |
| BR-5 | 无互动日展示 | 某用户当天无任何对话记录时，显示"当日无互动"文案，不显示空白 |
| BR-6 | 对话时间线连续性 | 日期时间线应展示从用户首次对话到今天的所有日期（包含无互动日），确保完整性 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 新用户尚未产生对话 | 在用户列表中显示，状态栏标记"尚未开始对话"，点击进入后显示空状态提示 |
| 单日对话超长（>100轮） | 分页加载，每页显示20轮，支持"加载更多"，不一次性全部渲染 |
| 用户量增长（>50人） | 用户列表支持按昵称搜索筛选 |
| 对话正在进行中（实时） | 不要求实时更新浏览页面，刷新后可见最新数据即可 |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 后台有多个用户的对话数据，When 打开对话浏览页，Then 看到用户列表（显示昵称、最后活跃时间、总对话数），按最近活跃排序 |
| AC-2 | Given 选中某个用户，When 点击进入，Then 看到该用户的对话时间线按天分组，每天显示对话轮数和简要摘要 |
| AC-3 | Given 选中某天的对话，When 展开查看，Then 看到完整对话内容，按时间顺序排列，清晰区分消息来源（用户/小伴）和渠道（企微/电话） |
| AC-4 | Given 某用户某天无对话记录，When 查看该天，Then 显示"当日无互动"而非空白 |
| AC-5 | Given 单日对话超过100轮，When 展开该天对话，Then 内容分页加载，交互流畅无卡顿 |

### F3: 对话全文搜索

**来源**：US-003 | **优先级**：P1 | **Story Points**：3

#### 描述

支持团队通过关键词搜索所有用户的对话记录，快速定位包含特定内容的对话片段。解决"想知道有多少用户提到过买菜"这类需求场景。

#### 输入

| 输入 | 说明 |
|------|------|
| 搜索关键词 | 用户输入的中文/英文/数字文本（1个或多个词） |
| 搜索范围 | MVP阶段固定为全部用户、全部时间 |

#### 输出

| 输出 | 说明 |
|------|------|
| 搜索结果列表 | 匹配的对话片段，含上下文、用户归属、时间信息 |
| 结果总数 | 匹配的对话条目数量 |
| 跳转链接 | 点击结果可跳转到完整对话上下文 |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 结果展示格式 | 每条结果显示：匹配文本（关键词高亮）+ 前后各1-2句话作为上下文 + 所属用户昵称 + 时间戳 |
| BR-2 | 结果排序 | 按时间倒序排列（最新的在前） |
| BR-3 | 结果跳转 | 点击任一结果，跳转到F2对话浏览器中该对话的完整上下文，搜索关键词在原文中高亮标记 |
| BR-4 | 空结果处理 | 无匹配结果时显示"未找到相关对话"，并给出建议："请尝试更换关键词或使用更短的词组" |
| BR-5 | 搜索范围 | MVP阶段搜索覆盖全部用户、全部历史时间段的对话 |
| BR-6 | 搜索对象 | 同时搜索用户消息和小伴回复的内容 |
| BR-7 | 结果分页 | 每页展示20条结果，支持翻页 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 单字搜索（如"买"） | 允许，可能结果较多，分页展示，不做限制 |
| 特殊字符搜索 | 过滤SQL注入风险字符，对正常标点符号支持搜索 |
| 超长关键词（>50字） | 截取前50字进行搜索，提示"已截取前50字进行搜索" |
| 中文分词 | 支持中文全文检索，确保"买菜"能匹配"帮我买点菜" |
| 搜索性能 | MVP数据量（<50用户、<10000条对话）下响应时间 < 3秒 |
| 并发搜索 | MVP阶段不需要考虑高并发（同时在线1-3人） |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 输入关键词"买菜"并执行搜索，When 系统中存在包含"买菜"的对话，Then 返回所有匹配片段，每条含上下文、用户归属、时间信息 |
| AC-2 | Given 搜索结果列表，When 点击某条结果，Then 跳转到该对话的完整上下文页面，关键词高亮显示 |
| AC-3 | Given 搜索关键词在全部对话中无匹配，When 执行搜索，Then 显示"未找到相关对话"及换词建议 |
| AC-4 | Given MVP数据量范围内（<10000条对话），When 执行任意关键词搜索，Then 结果返回时间 < 3秒 |

### F4: 对话意图自动标签

**来源**：US-004 | **优先级**：P0 | **Story Points**：5

#### 描述

利用AI自动对每一轮用户对话进行意图/话题分类，从预定义标签集中选择一个或多个标签。使团队无需逐条阅读对话即可掌握用户的对话类型分布，同时为F5话题统计和F7未满足需求识别提供数据基础。

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 对话轮次数据 | F1存储完成事件触发 | 包含用户消息 + 小伴回复的完整一轮对话 |
| 对话上下文 | F1存储层 | 当前Session中该轮之前的对话（用于理解语境） |
| 预定义标签集 | 系统配置 | 当前可用的意图标签列表 |

#### 输出

| 输出 | 消费方 | 说明 |
|------|--------|------|
| 意图标签（1个或多个） | F2浏览器展示、F5话题统计 | 从标签集中选择的分类标签 |
| 置信度分数 | F4自身（低置信度触发人工审核） | 0-1之间的打分 |
| 标签修改日志 | 模型优化 | 记录人工修正操作 |

#### 标签集（初始版本 v1）

| 标签 | 含义 | 举例 |
|------|------|------|
| 闲聊 | 日常寒暄、随意聊天 | "今天天气不错""吃饭了吗" |
| 情感倾诉 | 表达情绪、倾诉心事 | "最近心情不好""想我女儿了" |
| 健康相关 | 健康话题、症状描述、就医 | "膝盖又疼了""要不要去医院" |
| 购物需求 | 购买物品的请求或讨论 | "帮我买鸡蛋""超市有什么打折的" |
| 出行需求 | 出门、打车、导航相关 | "帮我叫个车""公园怎么走" |
| 信息查询 | 查询天气、新闻、知识等 | "明天冷不冷""今天有什么新闻" |
| 任务委托 | 明确请求小伴执行某事 | "提醒我3点吃药""帮我跟小明说" |
| 社交相关 | 涉及人际关系、社交活动 | "王姐约我明天去公园""儿子又没打电话" |
| 其他 | 以上标签均不适用 | — |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 异步处理 | 对话存储完成后异步触发标签任务，不阻塞主流程 |
| BR-2 | 时效要求 | 标签结果须在对话存储后5分钟内生成 |
| BR-3 | 最少标签数 | 每轮对话至少产出1个标签 |
| BR-4 | 多标签支持 | 一轮对话可同时拥有多个标签（如"购物需求"+"任务委托"） |
| BR-5 | 标签集可配置 | 管理员可新增/停用标签（停用后历史数据保留，新对话不再使用该标签） |
| BR-6 | 人工修正 | 运营人员可在后台编辑任一轮对话的标签，修改立即生效 |
| BR-7 | 修正日志 | 人工修正须记录：修正人、修正时间、原标签、新标签，供模型优化参考 |
| BR-8 | 低置信度处理 | 置信度 < 0.6 的标注结果标记为"待归类"，在运营后台单独列出供人工审核 |
| BR-9 | 准确率目标 | 人工抽检标签准确率 ≥ 80% |
| BR-10 | 上下文感知 | 标签引擎须参考对话上下文而非仅看当前轮单条消息 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 极短消息（"嗯""好的""哈哈"） | 结合该轮对话上下文（前后消息）判断意图，不孤立标注 |
| 新话题超出标签集覆盖范围 | 标记为"其他"并设置低置信度，进入"待归类"队列 |
| "其他"标签占比过高（>40%） | 系统在后台提示"建议扩展标签集"（见F5面板） |
| 多轮连续对话中意图切换 | 每轮独立标注，允许同一Session内不同轮次有不同标签 |
| AI服务不可用 | 标签任务进入重试队列，不影响对话存储和浏览；超过1小时未处理标记为"标注异常" |
| 批量历史数据补标 | 支持对已存储但未标注的历史对话进行批量异步标注 |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 一轮新对话存入系统，When 存储完成触发标签任务，Then 5分钟内该对话被打上至少1个意图标签 |
| AC-2 | Given 随机抽取50轮已标注对话进行人工校验，When 逐条对比AI标签与人工判断，Then 准确率 ≥ 80% |
| AC-3 | Given 运营人员发现标签错误，When 在后台手动修改标签，Then 修改立即生效并产生修正日志记录 |
| AC-4 | Given 标签引擎遇到低置信度对话（< 0.6），When 标注完成，Then 该条标记为"待归类"并出现在人工审核列表中 |
| AC-5 | Given 用户发送"嗯"等极短消息，When 标签引擎处理，Then 基于对话上下文给出合理标签，而非简单标为"其他" |

### F5: 高频话题统计面板

**来源**：US-005 | **优先级**：P1 | **Story Points**：5

#### 描述

以可视化图表展示各意图标签在指定时间范围内的出现频次和变化趋势，帮助团队快速发现"用户最常聊什么"以及"哪些需求在增长"，为产品决策提供数据支撑。

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 已标注的对话数据 | F4意图标签模块 | 全部已完成标注的对话轮次 |
| 时间范围选择 | 用户操作 | 用户选择的统计时间区间 |

#### 输出

| 输出 | 说明 |
|------|------|
| 标签频次柱状图 | 各标签在选定时间范围内的出现次数，降序排列 |
| 趋势标记 | 环比增长显著的标签标记为"上升趋势🔥" |
| 钻取对话列表 | 点击标签后展示的具体对话列表 |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 默认视图 | 打开面板默认展示最近7天的标签频次柱状图，按频次降序排列 |
| BR-2 | 时间范围选项 | 支持切换：今天 / 最近7天 / 最近30天 / 自定义时间段 |
| BR-3 | 自定义时间段 | 日期选择器，起止日期均可选，最长跨度不限制（MVP数据量小） |
| BR-4 | 趋势检测规则 | 若某标签在当前7天的频次对比上一个7天增长超过50%，标记为"上升趋势🔥" |
| BR-5 | 钻取功能 | 点击图表中任一标签柱体，下方展开该标签下的对话列表（显示用户、时间、对话摘要前30字） |
| BR-6 | 钻取对话跳转 | 对话列表中的每条记录可点击跳转到F2对话浏览器查看完整对话 |
| BR-7 | 频次计算口径 | 一轮对话有多个标签时，每个标签各计1次（不是只计主标签） |
| BR-8 | 图表数据刷新 | 切换时间范围后图表实时刷新，无需手动点击"查询" |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 选定时间范围内标注数据不足（<10轮） | 图表正常展示，但在图表上方显示警示："数据样本较少（共X轮），仅供参考" |
| "其他"标签占比 > 40% | 在面板顶部显示提示："其他类占比较高（XX%），建议评估是否需要扩展标签集" |
| 新标签刚添加（无历史数据） | 趋势标记不适用，不显示🔥标记（需至少两个完整周期才能计算趋势） |
| 时间范围为"今天"且当天尚无数据 | 显示空图表+提示"今日暂无标注数据" |
| 标签被停用 | 停用标签的历史数据仍参与统计展示（保留完整性），但标签名称后标注"已停用" |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 系统中有已标注的对话数据，When 打开统计面板，Then 显示最近7天各标签的频次柱状图，按降序排列，数据准确 |
| AC-2 | Given 统计面板，When 切换时间范围（今天/7天/30天/自定义），Then 图表数据正确刷新为对应时间范围 |
| AC-3 | Given 统计面板，When 点击某个标签柱体（如"购物需求"），Then 下方展开该标签下的对话列表，可查看摘要并跳转完整对话 |
| AC-4 | Given 某标签在最近7天频次环比增长>50%，When 打开面板，Then 该标签旁显示"上升趋势🔥"标记 |
| AC-5 | Given 选定时间范围内标注数据<10轮，When 查看面板，Then 显示"数据样本较少，仅供参考"警示 |

### F6: 用户活跃度看板

**来源**：US-006 | **优先级**：P1 | **Story Points**：3

#### 描述

综合展示所有用户的活跃状态概览，通过关键指标卡片和带状态标签的用户列表，让团队快速判断产品整体健康度、发现流失风险用户、识别超级用户。

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 对话记录元数据 | F1存储层 | 时间戳、Session数、轮次数、发起方 |
| 用户信息 | F1存储层 | 用户ID、昵称、注册时间 |

#### 输出

| 输出 | 说明 |
|------|------|
| 关键指标卡片区 | 总用户数、DAU、WAU、平均对话频率、平均对话深度 |
| 用户活跃状态列表 | 每个用户的活跃详情和状态标签 |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 指标卡片-总用户数 | 系统中有过至少1次对话记录的用户总数 |
| BR-2 | 指标卡片-DAU | 今天有过对话互动（主动或被动）的用户数 |
| BR-3 | 指标卡片-WAU | 最近7天有过对话互动的用户数 |
| BR-4 | 指标卡片-平均对话频率 | 全体活跃用户（最近7天）平均每周对话Session数 |
| BR-5 | 指标卡片-平均对话深度 | 全体用户最近7天的平均每次Session轮数 |
| BR-6 | 用户状态-活跃 | 最近3天内有过对话互动（不区分主动/被动） |
| BR-7 | 用户状态-沉默 | 3-7天内无任何对话互动 |
| BR-8 | 用户状态-流失风险 | 超过7天无任何对话互动 |
| BR-9 | 用户列表信息 | 每行显示：昵称、最后活跃时间、本周对话次数（Session数）、本周用户主动发起次数、状态标签（活跃🟢/沉默🟡/流失风险🔴） |
| BR-10 | 列表排序 | 默认按状态优先级排序：流失风险 > 沉默 > 活跃；同状态内按最后活跃时间升序（最久未活跃的在前） |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| "互动"定义 | 包括：用户主动发消息、用户回复小伴消息、电话接听并有对话。不包括：仅收到小伴消息但未回复 |
| 新用户注册前3天 | 实施"新手保护期"：注册3天内即使无互动也不标记为"沉默"，显示"新用户"标签 |
| 用户只读不回 | 仅收到小伴消息但未回复的，不算"互动"，3天后进入"沉默"状态 |
| 用户主动发起的定义 | 用户在无小伴消息的情况下首先发送消息，或在小伴消息后超过1小时才回复的（视为新话题发起） |
| 所有用户都是活跃状态 | 正常展示，指标卡片中"流失风险"区域显示0 |
| 系统刚启动无数据 | 所有指标卡片显示"0"或"暂无数据"，用户列表为空状态 |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 系统有活跃用户数据，When 打开活跃度看板，Then 指标卡片（总用户数/DAU/WAU/平均频率/平均深度）显示正确实时数据 |
| AC-2 | Given 某用户3天内有过对话，When 查看用户列表，Then 该用户显示"活跃🟢"状态标签 |
| AC-3 | Given 某用户超过3天未互动但不超过7天，When 查看看板，Then 该用户标记为"沉默🟡" |
| AC-4 | Given 某用户超过7天未互动，When 查看看板，Then 该用户标记为"流失风险🔴" |
| AC-5 | Given 新用户注册不满3天且无互动，When 查看看板，Then 该用户显示"新用户"标签，不标记为"沉默" |

### F7: 未满足需求识别

**来源**：US-007 | **优先级**：P0 | **Story Points**：5

#### 描述

自动识别并标记用户对话中"提出了明确需求/请求但小伴未能执行"的场景。将这些未满足需求按类型聚合统计，是产品迭代的第一驱动信号——直接告诉团队下一步该做什么功能。

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 对话轮次数据 | F1存储层 | 包含用户消息和小伴回复的完整对话轮 |
| 意图标签 | F4标签引擎 | 该轮对话的意图分类（尤其关注任务委托类） |
| 小伴回复模板特征 | 系统配置 | 小伴"无法执行"类回复的特征词库 |

#### 输出

| 输出 | 说明 |
|------|------|
| 未满足需求标记 | 特定对话轮被标记为"未满足需求" |
| 需求描述提取 | 从对话中提取的具体需求文本描述 |
| 聚合统计列表 | 按类型分组的未满足需求排行（含频次、具体对话链接） |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 识别逻辑核心 | 用户表达了明确的任务请求/需求意图 + 小伴的回复表明未能执行该任务 |
| BR-2 | 小伴未执行判断依据 | 小伴回复中包含以下特征之一：①明确表示帮不了（"这个我暂时还帮不了"）②转移话题未解决③回避性回复④提供了信息但未实际执行任务 |
| BR-3 | 需求聚合分组 | 相似需求按类型聚合（如"购物类需求 x 23次"、"打车类需求 x 8次"），而非逐条罗列 |
| BR-4 | 聚合排序 | 按频次降序排列，最高频未满足需求排在最前 |
| BR-5 | 钻取查看 | 每个聚合分组可展开查看具体的对话实例列表 |
| BR-6 | 成功执行排除 | 小伴成功处理的需求（如天气查询、提醒设置）不计入未满足需求 |
| BR-7 | 误标修正 | 运营人员可标记某条为"误标"，标记后从统计中移除并记入反馈日志 |
| BR-8 | 去重计数 | 同一用户重复提出的同类需求（如多次说"帮我买菜"）计为1个独立需求（频次不虚增） |
| BR-9 | 处理时效 | 随F4标签处理一并完成，存储后5分钟内产出结果 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 随口感叹 vs 真实请求 | "要是能帮我买菜就好了"（假设性感叹）vs "帮我买点鸡蛋"（明确请求）——系统应能区分，前者不计入；模棱两可时标记为低置信度供人工确认 |
| 部分满足 | 小伴帮了一部分但另一部分没做到（如查了信息但没代办）→ 未完成的部分仍计入未满足需求 |
| 用户主动放弃 | 用户说"算了没事"→ 仍计入未满足需求（用户需求是真实的，只是知道小伴做不到后放弃了） |
| 非任务型对话中的隐含需求 | "最近买菜好贵啊"——不是请求，不标记为未满足需求 |
| 同一用户同一类需求重复出现 | 同一类型只计1次频次，但记录所有对话实例供参考 |
| 需求类型超出现有聚合分类 | 新建聚合分类，不强行归入已有类 |
| AI识别服务不可用 | 进入重试队列，不阻塞其他功能 |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 用户表达明确任务请求且小伴回复表明未执行，When 识别引擎处理该对话，Then 该对话被正确标记为"未满足需求"并提取需求描述 |
| AC-2 | Given 存在多条未满足需求记录，When 打开未满足需求列表，Then 看到按频次降序的聚合分组（如"购物类 x 23次"），点击可展开具体对话 |
| AC-3 | Given 小伴成功处理了某请求（如查天气），When 识别引擎处理该对话，Then 不将其标记为未满足需求 |
| AC-4 | Given 运营人员发现误标，When 在后台标记为"误标"，Then 该条从统计频次中移除，且操作被记录到反馈日志 |
| AC-5 | Given 同一用户多次提出同类需求（如3次说"帮我买菜"），When 查看聚合统计，Then 该类型计为1个独立需求（不虚增为3次） |

### F8: 每周需求洞察报告

**来源**：US-008 | **优先级**：P2 | **Story Points**：3

#### 描述

每周自动生成一份结合数据和自然语言的"本周用户需求洞察"摘要报告，推送给团队核心成员。团队无需每天盯后台也能掌握产品关键趋势，降低运营后台的使用门槛。

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 用户活跃度数据 | F6 | 本周活跃/沉默/流失用户统计 |
| 话题标签统计 | F4 + F5 | 本周各标签频次及环比变化 |
| 未满足需求数据 | F7 | 本周新增/累计未满足需求聚合 |
| 值得关注的对话 | F1 + F4 | 深度对话、情感倾诉、异常对话等 |

#### 输出

| 输出 | 说明 |
|------|------|
| 自然语言摘要报告 | 结构化文本报告，含数据概览和趋势解读 |
| 推送消息 | 通过企业微信/邮件发送给指定接收人 |

#### 报告内容结构

```
📊 小伴运营周报（MM.DD - MM.DD）

1. 本周概览
   - 活跃用户数：X人（较上周 +/-Y%）
   - 新增用户：X人 | 流失风险：X人
   - 本周总对话数：X次，平均深度：X轮/次

2. 用户聊什么（Top 5高频话题）
   - 标签1：XX次（↑30%）
   - 标签2：XX次（持平）
   - ...
   - 新趋势：[如有上升趋势🔥的标签特别说明]

3. 小伴做不到的（Top 3未满足需求）
   - 需求类型1：XX次（举例：用户原话摘录）
   - 需求类型2：XX次
   - 需求类型3：XX次

4. 值得关注的对话
   - [1-2条深度对话或典型场景摘要]

5. 本周洞察
   - [1-2句AI生成的趋势解读或建议]
```

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 生成时间 | 每周一早上8:00自动生成上一周（周一至周日）的报告 |
| BR-2 | 推送渠道 | 通过企业微信消息和/或邮件推送给配置的接收人列表 |
| BR-3 | 接收人配置 | 后台可配置报告接收人（企业微信ID/邮箱地址），默认全团队 |
| BR-4 | 报告内容完整性 | 必须包含：活跃用户数、新增/流失对比、Top 5高频话题及变化、Top 3未满足需求、值得关注的对话摘要 |
| BR-5 | 数据不足处理 | 若该周对话总轮数 < 10轮，报告注明"本周数据样本较少，仅供参考"，不做趋势分析 |
| BR-6 | 趋势对比基准 | 与上一周同口径数据对比计算增减百分比 |
| BR-7 | 报告存档 | 每期报告自动存档，可在后台查看历史周报 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 产品上线首周（无上周对比数据） | 跳过趋势对比，报告标题注明"首周报告"，仅展示绝对数据 |
| 节假日周（数据异常） | 正常生成，但在报告中注明"本周含节假日，数据模式可能受影响" |
| 报告生成失败 | 自动重试1次（间隔10分钟）；若仍失败，通过企业微信通知技术负责人 |
| 无活跃用户（产品未启用） | 不生成报告，不推送 |
| 接收人列表为空 | 报告照常生成并存档，但不推送（后台可查看） |
| 报告内容过长 | 限制在1000字以内，超出部分以"查看完整数据请访问运营后台"结尾 |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|

### F9: 知识引擎（PKE）监控

**来源**：运营需求 | **优先级**：P1 | **Story Points**：5

#### 描述

提供对小伴个人知识引擎（PKE）运行状态的全面可视化监控，让团队能清楚看到知识的采集、编译、消费全链路状态。包含三个子视图：Raw信息浏览、Wiki知识浏览、Wiki消费日志。

#### 子视图一：Raw信息浏览

展示每位老人的知识库中已采集的原始对话片段（`vault/{elder_id}/raw/*.md`）。

| 展示字段 | 说明 |
|---------|------|
| 老人名称/ID | 关联的老人标识 |
| 文件名 | raw目录下的markdown文件名 |
| 采集时间 | 从文件名/元数据中提取的时间戳 |
| 内容预览 | 文件内容前50字 |

**操作**：
- 按老人筛选
- 按日期范围筛选
- 点击查看完整Raw内容

#### 子视图二：Wiki知识浏览

展示每位老人知识库中已编译的知识条目（`vault/{elder_id}/wiki/*.md`）。

| 展示字段 | 说明 |
|---------|------|
| 老人名称/ID | 关联的老人标识 |
| Wiki条目标题 | wiki目录下的文件名/标题 |
| 最后修改时间 | 文件last-modified时间 |
| 内容大小 | 文件字节数 |
| 新鲜度指标 | 最近编译日期 vs 今天的间隔天数 |

**操作**：
- 按老人筛选
- 点击查看完整Wiki内容
- 高亮显示超过7天未更新的条目（提示可能需要重新编译）

#### 子视图三：Wiki消费日志

记录每一次PKE查询（`pke use`）的详细日志，完整追踪知识被消费的情况。

| 展示字段 | 说明 |
|---------|------|
| 时间戳 | 查询发生的精确时间 |
| 老人ID | 查询关联的老人 |
| 用户消息（查询文本） | 触发知识查询的原始用户消息 |
| 返回内容摘要 | PKE返回的知识片段前100字 |
| 响应延迟 | 查询耗时（毫秒） |
| 命中/未命中 | PKE是否返回了非空的知识上下文 |

**操作**：
- 按老人筛选
- 按日期范围筛选
- 按关键词搜索（查询文本或返回内容）
- 统计命中率（命中次数/总查询次数）

#### 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| PKE Vault文件系统 | `/data/users/{elder_id}/raw/` 和 `/data/users/{elder_id}/wiki/` | Raw和Wiki的源数据 |
| PKE查询日志 | `pke_query_logs` 数据表 | 每次`pke use`调用的持久化记录 |
| 老人基本信息 | F1存储层（elders表） | 老人ID与名称的映射 |

#### 输出

| 输出 | 说明 |
|------|------|
| Raw信息列表页 | 分页展示所有采集的原始对话片段 |
| Wiki知识列表页 | 分页展示所有编译后的知识条目 |
| Wiki消费日志页 | 分页展示每次知识查询的详细记录 |
| 知识健康度概览 | 汇总指标：总Raw数、总Wiki数、今日查询次数、命中率 |

#### 数据模型（新增）

```
pke_query_logs（Wiki消费日志表）
├── id: 主键
├── elder_id: 关联老人ID
├── query_text: 用户原始消息（触发查询的文本）
├── result_snippet: PKE返回的知识片段（前200字）
├── hit: 布尔值（是否返回非空结果）
├── latency_ms: 查询耗时（毫秒）
├── created_at: 查询时间戳
```

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR-1 | 查询日志自动记录 | 每次对话触发的`pke use`调用，无论命中与否，均写入`pke_query_logs`表 |
| BR-2 | Raw列表实时读取 | Raw信息从文件系统实时读取，无缓存（MVP数据量小） |
| BR-3 | Wiki列表实时读取 | Wiki条目从文件系统实时读取，展示最新编译结果 |
| BR-4 | 日志保留期限 | MVP阶段无自动清理，永久保留查询日志 |
| BR-5 | 命中率计算 | 命中率 = 返回非空结果的查询次数 / 总查询次数 × 100% |
| BR-6 | 新鲜度告警 | Wiki条目超过7天未更新时，在列表中标记为"需要重新编译" |
| BR-7 | 分页规则 | 每页默认20条记录，支持翻页 |

#### 边界条件

| 条件 | 处理方式 |
|------|---------|
| 老人Vault目录不存在 | 该老人显示"知识库未初始化"状态，不报错 |
| Raw/Wiki目录为空 | 显示空列表 + 提示"暂无数据" |
| PKE查询超时（1秒） | 日志记录latency_ms为超时值，hit标记为false，result_snippet为空 |
| 文件系统读取失败 | 页面显示错误提示"文件系统不可达"，不影响其他子视图 |
| 日志表数据量过大（>10万条） | MVP阶段不处理，后续考虑分区或归档 |
| 同一秒多次查询 | 每次独立记录，按created_at精确排序 |

#### 验收标准

| AC编号 | 验收条件 |
|--------|---------|
| AC-1 | Given 某老人有Raw数据文件，When 打开Raw信息浏览页并选择该老人，Then 显示该老人所有raw文件列表（含文件名、时间、内容预览），点击可查看完整内容 |
| AC-2 | Given 某老人有Wiki知识条目，When 打开Wiki知识浏览页，Then 显示该老人所有wiki条目（含标题、修改时间、大小），超过7天未更新的标记为"需要重新编译" |
| AC-3 | Given 用户与小伴对话触发了PKE查询，When 查看Wiki消费日志，Then 能看到该次查询的完整记录（时间、老人ID、查询文本、返回摘要、延迟、命中状态） |
| AC-4 | Given 系统运行一段时间后，When 查看知识健康度概览，Then 显示总Raw数、总Wiki数、今日查询次数、命中率等汇总指标 |
| AC-5 | Given Wiki消费日志页面，When 按老人+日期范围筛选，Then 只显示符合条件的查询记录 |

#### 技术前置条件

> 注意：子视图三（Wiki消费日志）需要后端代码修改——在 `backend/app/pke/pke_service.py` 的 `query()` 方法中增加持久化日志写入逻辑，将每次查询的参数和结果写入 `pke_query_logs` 表。这是本模块的技术前置依赖。

### F10: 老人详情页（Tenant Detail）


**跨系统依赖**：
- F1 的数据来源 = 小伴老人端（企业微信对话引擎 + 电话呼叫系统）的对话输出
- F4/F7 的分析结果反哺产品决策 → 指导老人端功能优先级（如确认"购物"是最高频未满足需求后，优先开发代购功能）
- F7 的未满足需求直接决定Phase 2该做什么功能

## 八、术语表

| 术语 | 定义 |
|------|------|
| Session | 一次完整对话。企微渠道以30分钟无新消息为界，电话渠道以通话结束为界 |
| Turn | 一轮问答。包含用户的一次发言+小伴的一次回复 |
| Message | 单条消息。一个Turn通常包含2条Message（用户1条+小伴1条） |
| 意图标签 | 对一轮对话的主题/意图分类标注 |
| 未满足需求 | 用户表达了明确的任务请求，但小伴未能执行的场景 |
| 活跃用户 | 最近3天内有过对话互动的用户 |
| 沉默用户 | 3-7天内无任何对话互动的用户 |
| 流失风险用户 | 超过7天无任何对话互动的用户 |
| 新手保护期 | 用户注册后前3天的特殊状态，期间不标记为"沉默" |
| DAU | Daily Active Users，日活跃用户数 |
| WAU | Weekly Active Users，周活跃用户数 |
| PKE | Personal Knowledge Engine，个人知识引擎。基于对话记录提取、编译、检索老人的长期记忆和偏好知识 |
| Vault | 每位老人的知识库存储目录，包含raw/和wiki/两个子目录 |
| Raw | 原始对话采集片段，由capture操作写入vault/raw/目录的markdown文件 |
| Wiki | 经过daily compile编译后的结构化知识条目，存储在vault/wiki/目录 |
| Wiki消费 | 对话过程中通过pke use查询Wiki知识库的行为，每次查询记录到消费日志 |
