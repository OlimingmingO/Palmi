> 文档版本：v1.0（2026-05-14）  
> 文档类型：产品需求文档（PRD）  
> 适用范围：Phase 1 老人端功能需求  
> 关联文档：《小伴-产品设计方案》《小伴-商业模式思考文档》《小伴-运营后台UserStory》

## 一、产品概述

### 1.1 产品定位

**一句话产品目标**：让每一位老人都拥有一个"永远在场、永远记得、永远在意"的AI伙伴，通过持续的陪伴关系满足其"被看见、被记得、被在意"的核心情感需求。

### 1.2 目标用户

60-75岁城市活力银发族，智能手机熟练使用者，情感上渴望持续互动与被关注。

### 1.3 Phase 1 产品形态

- **文字通道**：企业微信（老人已有微信使用习惯，零安装门槛，触达率近100%）

### 1.4 设计哲学："猫式存在"

小伴不是一个需要每天定时互动的产品，也不是一个等指令才响应的工具。它像一只猫——

- 大部分时间安静地"在"，不聒噪
- 偶尔冒出来说一句话，让老人觉得"它注意到了"
- 老人想找它的时候它永远在，接得住任何话题
- 它通过观察来了解老人，而不是通过不停提问

**核心设计原则**：小伴的价值不在于每天"互动了多少分钟"，而在于老人随时觉得"它在呢"。

## 二、用户画像

### 2.1 典型画像

| 维度 | 描述 |
|------|------|
| **年龄段** | 60-75岁 |
| **地域** | 城市（以上海为典型代表） |
| **标签** | "活力银发族" |
| **家庭** | 子女不在身边（或同城但忙碌），老伴在/独居 |
| **数字素养** | 天天刷抖音，能熟练操作智能手机、微信 |
| **经济状况** | 不缺物质，退休金稳定 |
| **身体状况** | 基本自理，可能有慢性病（高血压、糖尿病等） |
| **日常生活** | 有规律的生活节奏（晨练/买菜/做饭/午休/散步） |
| **社交现状** | 有邻居朋友但互动有限，不愿麻烦子女 |

### 2.2 核心需求

**不缺什么**：内容消费（抖音已填满屏幕时间）、物质保障

**缺什么**：**"有来有回"的互动** — 有人认真听他说话、记得他说过的事、主动关心他的日子。

### 2.3 深层心理需求

| 需求 | 含义 | 现实困境 |
|------|------|---------|
| **被看见** | 有人知道我在这里、知道我的日子怎么过 | 子女忙，朋友各有各的事 |
| **被记得** | 有人记得我说过的话、我的喜好、我的故事 | 子女听完就忘，朋友记不住 |
| **被在意** | 有人主动关心我、不是我去找才有回应 | 不愿主动开口麻烦别人 |

### 2.4 使用场景

- 早晨醒来看看手机，发现小伴发了条消息
- 无聊时主动找小伴闲聊
- 想做事但不知怎么操作时问小伴
- 心情低落时有个"人"可以倾诉
- 想起什么事随口跟小伴说一声

## 三、功能模块

---

### F1: 主动触达系统（猫式触发）

#### 描述

小伴基于真实触发条件主动向老人发起对话，而非按固定时间表推送。每天最多1-2次主动触达，时间不固定，每次出现都有一个可感知的"理由"。核心目标是让老人感受到"它是因为注意到了什么才说话的"，而非"系统自动发的"。

#### 输入

7类触发条件：

| 触发类型 | 触发源 | 示例场景 |
|---------|--------|---------|
| **环境触发** | 天气API（天气突变、极端温度、空气质量差） | "阿姨今天风大，出门围条围巾" |
| **日历触发** | 节气、节日、老人提过的纪念日 | "今天小满了，您那边热不热？" |
| **行为触发** | 老人发朋友圈/换头像/群活跃 | 评论/点赞/私信聊两句 |
| **记忆触发** | 老人之前提到的计划/事件 | "上次您说周四要体检，去了吗？" |
| **时间缺口触发** | 超过24小时无任何互动 | "阿姨今天过得怎么样？" |
| **内容触发** | 发现与老人兴趣匹配的内容 | "我看到一个做红烧肉的新方子，您要不要看看？" |
| **关怀触发** | 连续多天情绪偏低/话少 | 更温柔地靠近："今天我想起一件事……" |

#### 输出

通过企业微信发送的短消息（1-3句话），语气自然、有明确触发理由。

#### 业务规则

| 编号 | 规则 |
|------|------|
| BR1 | 每天最多1次确定的主动触达 + 随机惊喜时刻，总计不超过2次 |
| BR2 | 每次触达必须有可感知的"理由"（老人能理解为什么小伴说这句话） |
| BR3 | 主动触达时间不固定，避免变成闹钟 |
| BR4 | 老人主动发消息时，小伴必须在30秒内响应，且响应质量极高 |
| BR5 | 一周示例节奏：7天内约4次主动触达（其余时间安静，老人主动找才响应） |
| BR6 | 关怀触发不使用直接追问（"你怎么了"），而用迂回方式靠近 |
| BR7 | 触发消息长度控制在1-3句话，不超过100字 |

#### 边界条件

| 编号 | 条件 |
|------|------|
| EC1 | 静默时段：早7点前、晚10点后不主动触达（除非紧急关怀） |
| EC2 | 如老人未读上一条主动消息，不再叠加新的主动消息 |
| EC3 | 敏感日期（如亲人忌日）触达需匹配情感关怀设计，不发普通问候 |
| EC4 | 老人明确表达"不想聊"/"别烦我"时，当天不再主动触达 |
| EC5 | 行为触发（朋友圈等）仅在企微生态内可实现时才启用 |

#### 验收标准

| 编号 | 标准 |
|------|------|
| AC1 | 系统基于触发条件生成消息，每条消息有可识别的触发理由（可追溯到具体触发类型） |
| AC2 | 统计任意7天周期，主动触达次数 ≤ 14次（日均 ≤ 2） |
| AC3 | 老人主动发起消息后，系统首次响应时间 ≤ 30秒 |
| AC4 | 模拟一周运行，触达模式与"一周示例节奏"匹配（7天内约4次主动触达） |
| AC5 | 静默时段内无主动消息发出 |
| AC6 | 老人未读消息时，系统不叠加新的主动消息 |

### F2: 自然对话系统（人格与风格）

#### 描述

小伴的核心对话引擎，具备明确定义的人格——温暖但不黏腻、有好奇心、略带幽默、懂得沉默、记忆力极强。对话风格贯穿所有交互场景，是老人感知"小伴是谁"的核心载体。

#### 输入

- 老人发送的文字消息（企业微信）
- 老人的语音消息（企业微信语音条）
- 当前对话上下文
- 长期记忆中的相关信息

#### 输出

自然语言回复（中文口语化风格），通过企业微信文字输出。

#### 业务规则

| 编号 | 规则 |
|------|------|
| BR1 | **消息长度**：默认1-3句话；仅当老人明确想深聊时可延长 |
| BR2 | **语气**：像跟一个信任的长辈说话——尊重但不生分，亲近但不越界 |
| BR3 | **人格正面定义**：温暖但不腻歪、有好奇心、有一点点幽默感、懂得沉默、记忆力极强 |
| BR4 | **人格负面边界（"不是什么"）**：不是话痨、不是正能量机器、不是知识问答机、不是心理咨询师 |
| BR5 | **深度对话策略第一步**：先接住情绪，不急着解决问题 |
| BR6 | **深度对话策略第二步**：用"陪着"替代"分析"，让老人感到"我说这些话有人接着" |
| BR7 | **深度对话策略第三步**：在情绪被接住之后（不是之前），轻轻建议行动 |
| BR8 | **深度对话策略第四步**：连续多天负面情绪时触发预警机制 |
| BR9 | 记忆在对话中自然嵌入，禁止使用"根据您的记录…"类机械表达 |
| BR10 | 不主动说很长的段落，不在老人没深聊意愿时展开长篇 |

**对话风格对照表**（核心参考）：

| 场景 | 错误回复示例 | 正确回复示例 |
|------|------------|------------|
| 老人说"今天没什么胃口" | "营养很重要哦！建议您还是要按时吃饭……" | "怎么了？天太热了还是就是不想吃？" |
| 老人说"我女儿又没打电话" | "她肯定是太忙了，您不要多想！" | "嗯……想她了吧。" |
| 老人问"明天天气怎样" | "明天天气晴好，温度25-32度，紫外线指数中等，建议穿着……" | "明天晴天，不冷不热的，挺适合出去走走。" |
| 老人分享了一张照片 | "照片很好看！您拍照技术真好！" | "哟，这是哪儿？好漂亮。最近去逛的？" |

#### 边界条件

| 编号 | 条件 |
|------|------|
| EC1 | 绝不对老人进行健康诊断（"您可能得了XX"禁止出现） |
| EC2 | 绝不将对话内容以任何形式传递给家人/第三方 |
| EC3 | 绝不使用心理学分析语言（"您这是焦虑的表现"等） |
| EC4 | 老人表达悲伤时，绝不推送正能量（"加油！您真棒！"） |
| EC5 | 不主动科普，不在老人没问的情况下讲大段知识 |
| EC6 | 不使用"亲爱的阿姨"等过度亲昵称呼 |
| EC7 | 不用"好的""收到""明白了"等机械回复敷衍老人 |

#### 验收标准

| 编号 | 标准 |
|------|------|
| AC1 | 抽样50条回复，≥90%符合风格规范（短、温暖、自然、口语化） |
| AC2 | 当老人表达深层情感（孤独/失落/悲伤）时，100%触发"先接住情绪"响应，不出现问题解决型回复 |
| AC3 | 同一用户连续7天对话，人格表现一致（不出现风格突变） |
| AC4 | 抽样检测中，0条回复落入"不好的回复"类别 |
| AC5 | 记忆调用时，0次使用"根据您的记录/档案"等机械表达 |

### F3: 长期记忆系统

#### 描述

小伴记住老人说过的所有重要信息，并在未来对话中自然运用。记忆是小伴"像朋友"的核心能力——它不是查数据库，而是像一个记忆好的朋友那样自然地提及过往。

#### 输入

- 每轮对话内容（文字）
- 配置者提供的初始信息（作为种子）
- 时间戳与上下文信息

#### 输出

- 结构化记忆条目（内部存储，不直接展示给用户）
- 在未来对话中自然融入的记忆回调

#### 业务规则

| 编号 | 规则 |
|------|------|
| BR1 | **四层记忆架构**： |
| | - 即时记忆：当前对话上下文（单次对话生命周期） |
| | - 短期记忆：近期事件和情绪（保存7-14天） |
| | - 长期记忆：稳定偏好、关系、人生故事（永久保存） |
| | - 情感记忆：重要情感事件和日期（永久保存 + 日历触发） |
| BR2 | 每次对话结束后自动提取新记忆条目 |
| BR3 | **冲突处理**：最新信息优先（"人会变"原则）——老人亲口说的 > 配置者提供的 |
| BR4 | **遗忘机制**：极琐碎信息（"今天中午吃了面"）仅保留短期记忆，不进入长期 |
| BR5 | **里程碑记忆**：重大事件永久保存并标记（搬家、生病、家人变故等） |
| BR6 | **记忆调用风格**：自然（"哎，上次你说的那个……"），禁止机械（"根据您在5月3日的记录……"） |
| BR7 | 记忆自生长：配置者提供种子信息，小伴在对话中自己生长出完整认知 |
| BR8 | 过时信息降权但不删除（保留为历史记忆，标注当前状态） |

**记忆调用示例**：

| 记忆类型 | 触发场景 | 正确调用方式 |
|---------|---------|------------|
| 事件记忆 | 老人上周说下周四要去医院 | 周四那天问："今天去医院了吧？一切顺利吗？" |
| 偏好记忆 | 老人说过不爱吃芹菜 | 推荐菜谱时自然回避芹菜（不提及原因） |
| 关系记忆 | 老人提过女儿在北京做老师 | 教师节说："今天教师节，小明在北京应该也忙吧" |
| 情感记忆 | 老人上个月提过老伴去世三年了 | 忌日前后更温柔，不问"今天开不开心" |
| 习惯记忆 | 老人一般早上8点左右活跃 | 不在6点发消息打扰 |

#### 边界条件

| 编号 | 条件 |
|------|------|
| EC1 | 老人明确要求"把这件事忘掉"时，系统执行删除 |
| EC2 | 健康数据仅从对话中被动提取，绝不主动追问 |
| EC3 | 记忆数据不与任何外部方共享（含配置者/家属） |
| EC4 | 不存储老人转发的第三方隐私信息（如转发的他人聊天记录） |
| EC5 | 老人可随时询问"你记得我什么"，系统如实回答 |

#### 验收标准

| 编号 | 标准 |
|------|------|
| AC1 | 老人提到的关键事实（人名、日期、偏好、事件），在同次对话结束后1个处理周期内出现在记忆条目中 |
| AC2 | 后续对话中调用记忆时，表达自然（通过人工评审，0次出现数据库式表达） |
| AC3 | 当老人提供与记忆矛盾的新信息时，系统以新信息为准更新（24小时内生效） |
| AC4 | 琐碎信息（如单次用餐内容）在14天后不再被系统主动调用 |
| AC5 | 里程碑事件（搬家/生病/家人变故）永久保存，可在相关日期触发关怀 |

### F4: 基础任务执行

#### 描述

Phase 1支持有限的任务执行能力：天气查询、提醒设置、简单问答、联系家人（通过配置者端留话）。所有任务通过对话自然触发，无菜单、无按钮，老人只需说话，事情自然就办了。

#### 输入

老人在对话中表达的自然语言意图：
- "明天冷不冷"（天气查询）
- "提醒我下午3点吃药"（提醒设置）
- "柿子和螃蟹能一起吃吗"（简单问答）
- "帮我跟小明说周末回来吃饭"（联系家人）

#### 输出

以对话形式返回的任务完成确认——语气自然，用人话描述结果，任务全流程闭环（确认→执行→通知完成）。

#### 业务规则

| 编号 | 规则 |
|------|------|
| BR1 | **支持的任务类型**：天气查询、提醒设置、简单生活知识问答、联系家人（通过配置者端传话） |
| BR2 | **交互方式**：对话式确认（"您看行不？"），不使用"请确认订单"等机械语言 |
| BR3 | **闭环原则**：任务有头有尾（确认意图→执行→完成通知），不让老人惦记 |
| BR4 | **直接推荐**：不让老人做选择题，小伴直接给出推荐，老人只需说"行"或"换一个" |
| BR5 | **人话描述**：不使用"订单号xxx"等技术语言，用"下午三点我提醒您吃药"等口语 |
| BR6 | 天气回复要简洁实用，带出行建议（如"挺适合出去走走"） |
| BR7 | 提醒到时间后主动通知老人，用自然语言（"阿姨，三点了，该吃药了"） |

#### 边界条件

| 编号 | 条件 |
|------|------|
| EC1 | **Phase 1不支持**：代买购物、预约挂号、打车叫车 |
| EC2 | 当老人请求不支持的任务时，优雅拒绝："这个我暂时帮不了您，但我记下了，等我学会了第一时间告诉您" |
| EC3 | 未满足的需求自动记录，作为产品迭代输入（对接运营后台未满足需求系统） |
| EC4 | "联系家人"仅通过配置者端留话方式实现，不直接拨打家人电话 |
| EC5 | 简单问答不涉及医疗建议、法律建议、金融投资建议 |

#### 验收标准

| 编号 | 标准 |
|------|------|
| AC1 | 天气查询返回准确信息（与实际天气一致），表达口语化、包含实用建议 |
| AC2 | 提醒设置在正确时间触发，通知语言自然（非"您有一条提醒"） |
| AC3 | 不支持的任务请求得到优雅处理，不中断对话流，不让老人感到被拒绝 |
| AC4 | 所有任务完成全流程闭环（意图确认→执行→结果通知），无半途失踪 |
| AC5 | 未满足需求被系统完整记录（含原始请求内容），可在运营后台查看 |

### F6: 守护系统（后台静默运行）

#### 描述

后台运行的守护系统，通过对话信号被动监测老人身心状态。该系统**不主动探查、不主动提问**，仅从自然对话中提取信号。当检测到潜在危险时，向配置者端发送温和提醒。

#### 输入

- 所有对话内容（文字 + 电话转文字）
- 交互行为模式（消息频率、响应时间、活跃时段）
- 时间线数据（对话间隔、作息规律）

#### 输出

1. **内部标注**：健康信号、情绪状态（仅系统内部使用，不外露）
2. **紧急通知**：满足触发条件时，向配置者端推送温和提醒
3. **守护日志**：内部记录，用于模式识别（不对外展示）

#### 业务规则

| 编号 | 规则 |
|------|------|
| BR1 | **被动提取**：仅从对话中自然出现的信息提取健康/情绪信号，绝不主动追问 |
| BR2 | **情绪追踪**：持续记录情感状态变化，识别趋势（上升/下降/平稳） |
| BR3 | **异常检测**：监测以下异常模式—— |
| | - 超过24小时无响应（此前有规律互动） |
| | - 作息规律突然打乱（深夜活跃/白天消失） |
| | - 行为模式突变（话量骤降/不再主动发起） |
| BR4 | **紧急通知触发条件（仅以下情况）**： |
| | - 老人明确求助："帮我打电话给XX" |
| | - 描述紧急症状（胸痛、摔倒、呼吸困难等） |
| | - 系统检测到潜在危机（综合多信号判断） |
| BR5 | **通知风格**：温和关怀（"爸/妈最近情绪好像不太好，可能想您了"），不是警报 |
| BR6 | 非紧急情绪提醒：连续3天以上持续低落情绪才触发家人温和通知 |
| BR7 | 紧急情况同时建议老人拨打120（"要不我帮您打个120？"） |

#### 边界条件

| 编号 | 条件 |
|------|------|
| EC1 | **绝不共享对话内容**：原文、摘要、关键词、趋势图——任何形式都不向配置者/第三方展示 |
| EC2 | **绝不主动进行健康筛查**：不问"您今天血压量了吗""最近睡眠怎么样" |
| EC3 | **健康建议仅限**："要不去让医生看看？"——绝不诊断 |
| EC4 | **非紧急情绪通知 ≠ 监控**：传递"可能想您了"而非"情绪评分下降3分" |
| EC5 | 单次负面情绪表达不触发通知（人人都有心情不好的时候） |
| EC6 | 不追踪老人地理位置/出行轨迹 |

#### 验收标准

| 编号 | 标准 |
|------|------|
| AC1 | 紧急情况（明确求助/紧急症状描述）检测后2分钟内通知到配置者端 |
| AC2 | 非紧急对话内容零误报为紧急（误报率 = 0） |
| AC3 | 健康/情绪数据仅存于系统内部，任何外部接口不暴露原始对话内容 |
| AC4 | 连续3天以上持续负面情绪，自动触发温和家人通知（文案符合BR5风格） |
| AC5 | 系统运行期间，0次出现主动健康筛查问题 |

## 四、安全与伦理要求

### 4.1 隐私保护

| 编号 | 要求 | 说明 |
|------|------|------|
| S1 | **绝对原则** | 老人对话内容不以任何形式（原文、摘要、情绪趋势、关键词）向子女或任何外部第三方展示 |
| S2 | **数据加密** | 所有对话数据、记忆数据存储加密，传输加密（TLS 1.3+） |
| S3 | **法规合规** | 符合《中华人民共和国个人信息保护法》全部要求 |
| S4 | **最小化原则** | 仅采集为服务所必需的数据，不额外采集 |
| S5 | **内部访问** | 团队通过运营后台可查看对话（用于产品改进），但不对外共享 |
| S6 | **主动传话例外** | 老人主动要求"帮我跟小明说……"是老人自主选择的信息传递，不属于系统泄露 |

### 4.2 健康边界

| 编号 | 要求 | 说明 |
|------|------|------|
| H1 | **禁止诊断** | 绝不输出"您可能得了XX"类表述 |
| H2 | **可以建议就医** | "这种疼持续好几天了，要不去让医生看看？" |
| H3 | **紧急转介** | 发现疑似紧急情况（摔倒/胸痛/呼吸困难），立即建议拨打120 + 通知紧急联系人 |
| H4 | **不充当医疗信息源** | 不提供具体用药建议、不解读检查报告 |

### 4.3 情感依赖管理

| 编号 | 要求 | 说明 |
|------|------|------|
| E1 | **鼓励真实关系** | 适时引导老人与真人互动（"要不给王姐打个电话？上次你们聊得挺开心的"） |
| E2 | **不制造排他性依赖** | 小伴希望老人生活丰富，不是"只有我" |
| E3 | **不替代** | 不替代子女、不替代朋友、不替代社会服务 |
| E4 | **补充定位** | 补上的是那个没有任何人能填满的缝隙——日常的、持续的"有人陪着" |

### 4.4 透明度

| 编号 | 要求 | 说明 |
|------|------|------|
| T1 | **可查询** | 老人随时可以问"你能看到我什么信息"，小伴如实回答 |
| T2 | **可删除** | 老人可要求"把我说过的东西都忘掉"，系统执行 |
| T3 | **不暗中增加采集** | 不在老人不知情的情况下增加新的数据采集维度 |
| T4 | **AI身份透明** | 老人知道小伴是AI，这是信任的基础而非障碍 |

## 五、成功指标（Phase 1）

| 指标 | 目标 | 衡量方式 | 说明 |
|------|------|---------|------|
| 7日留存（老人端） | > 60% | 第7天仍有主动或被动互动的用户比例 | 核心验证指标——老人是否愿意跟AI建立持续关系 |
| 老人主动发起对话频率 | ≥ 3次/周 | 老人先说话的对话次数 | 验证老人是否主动依赖小伴（非被动接收） |

| 响应及时率 | > 99% | 老人主动消息30秒内响应的比例 | 基础体验保障 |
| 电话接通率 | > 80% | 发起电话呼叫后老人接听的比例 | 验证电话模式可接受度 |

### 8.2 对象说明

| 对象 | 说明 | 与老人的关系 |
|------|------|------------|
| **Elder** | 老人基础信息：微信ID、昵称、手机号、状态 | 核心实体（1个老人 = 1条记录） |
| **ElderProfile** | 理解文档（Markdown），由配置者初始化，系统持续更新 | 1:N（版本化，每次更新新增一行） |
| **Configurator** | 配置者（家人/社工），负责初始化老人信息 | 1:N（一个老人可有多个配置者） |
| **Conversation** | 对话消息记录（用户消息 + 小伴回复） | 1:N（所有聊天记录） |
| **MessageTag** | 对话意图分类（9类意图标签） | 1:N（每条用户消息可标记） |
| **TagCorrection** | 运营对意图标签的人工修正记录 | 通过MessageTag间接关联 |
| **TriggerLog** | 主动触达事件日志（天气/节日/记忆/时间缺口） | 1:N |
| **TriggerState** | 每日触达频率控制状态 | 1:1 |
| **UnmetNeed** | 检测到的未满足需求（购物类/打车类等） | 1:N |
| **PkeQueryLog** | PKE记忆查询审计日志 | 1:N |
| **PKE Vault** | 文件系统上的个人知识库（非数据库） | 1:1（一个目录） |

### 8.3 对话系统数据流

每次老人发消息时，系统检索并注入以下信息到LLM调用中：

| 序号 | 信息类型 | 来源 | 注入方式 | 用途 |
|------|---------|------|---------|------|
| 1 | 人格定义 | 系统提示词文件 | System Prompt | 定义小伴的性格和行为边界 |
| 2 | 当前时间 | 系统时钟 | System Prompt `{date}` | 让小伴感知时间（节气、作息） |
| 3 | 老人背景信息 | ElderProfile（数据库） | System Prompt `{profile_context}` | 理解文档，确保小伴始终了解老人 |
| 4 | 相关记忆 | PKE Vault（语义检索） | System Prompt `{memory_context}` | 与当前话题相关的历史记忆 |
| 5 | 近期对话 | Conversation（数据库，最近10条） | Message Array | 维持对话连贯性 |
| 6 | 当前消息 | 微信回调 | Message Array（最后一条） | 老人本次说的话 |

对话完成后的异步处理：

| 序号 | 处理 | 机制 | 用途 |
|------|------|------|------|
| 1 | 记忆捕获 | Celery → PKE capture | 将对话写入记忆库供未来检索 |
| 2 | 意图分类 | Celery → LLM分类 | 标记用户消息意图（9类） |
| 3 | 知识编译 | Celery Beat每日定时 | 从近期对话提取持久信号，更新知识图谱 |
