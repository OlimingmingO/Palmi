## 一、产品概述

### 1.1 产品定位

配置者端是小伴产品体系中面向"了解老人的人"的操作界面，承载初始化设置功能。配置者端以 Web Console（桌面优先响应式Web应用）形态存在，为配置者提供一个直观、高效的入口。

### 1.2 产品目标

让了解老人的人（子女、社工、邻居、老伴、老人自己）能够：
1. 用最自然的方式向小伴介绍这位老人，帮助小伴快速建立对老人的初始认知
2. 在日常生活中通过小伴向老人传递关心
3. 在紧急情况下及时获得通知

### 1.3 Phase 3 范围

- **产品形态**：Web Console（桌面优先响应式Web应用，支持 Chrome/Edge/Safari）
- **核心功能**：初始化配置（1个功能模块）
- **不含功能**：给小伴留话（Phase 4+）、紧急通知（Phase 4+）、付费订阅管理、多老人管理、配置者协作/权限、任何形式的"报告"或"数据"

### 1.4 核心原则

**"绝不监控"**——配置者付费购买的是"我的父母有人陪着"这件事本身，不是信息。小伴的信任基础是老人知道"我跟小伴说的话不会被任何人看到"，这一承诺不可打破。

## 二、用户画像

### 2.1 主要用户：异地子女

| 维度 | 描述 |
|------|------|
| **年龄段** | 30-50岁 |
| **生活状态** | 在不同城市工作，无法每天陪伴父母 |
| **心理特征** | 对"不能经常打电话/回家"有内疚感；希望父母生活有人关照 |
| **数字能力** | 互联网原住民，熟悉小程序/H5操作 |
| **付费意愿** | 愿意为父母的陪伴和安全感付费（孝心经济） |
| **核心动机** | "我不能天天在身边，但至少让父母有人陪着" |

**关键洞察**：子女付费买的不是信息，是"我的父母有人陪着"这件事本身。就像给孩子报兴趣班，付钱但不要求老师汇报每句话。

### 2.2 次要用户

| 用户类型 | 典型场景 | 特点 |
|---------|---------|------|
| **社工** | 社区养老服务人员，负责多位老人的日常关怀 | 对老人的了解偏生活层面，配置内容偏客观 |
| **邻居** | 与老人关系密切的邻里，主动帮助配置 | 了解老人的日常习惯和社交圈 |
| **老伴** | 老人的配偶，补充另一视角的信息 | 了解老人的深层偏好和忌讳 |
| **老人自己** | 老人自己上手填写或补充信息 | 最真实的第一手信息来源 |

### 2.3 用户场景故事

**场景A：子女首次开通**  
小明在北京工作，妈妈张美兰68岁独居上海。小明在朋友推荐下了解到小伴，打开配置者 Web Console 后看到一段引导语，花10分钟写了一段关于妈妈的介绍，小伴即刻为妈妈开通服务。

> 场景B（给小伴留话）和场景C（紧急通知）已推迟至 Phase 4+，不在本期范围内。

## 三、功能模块

### F1: 初始化配置（开放叙事式）

#### 描述

配置者通过自由书写的方式向小伴介绍老人。没有表单、没有必填字段、没有格式要求。就像跟一个新朋友讲"我妈是个怎样的人"——你想到什么说什么，什么格式都行。系统将自然语言输入解析为结构化的"理解文档"，作为小伴认知老人的起点。

#### 输入

| 输入项 | 说明 |
|--------|------|
| **自由文本** | 配置者用自然语言描述老人，任何长度、任何格式、任何维度均可 |
| **引导提示语** | "用你自己的话，跟小伴介绍一下这位老人。什么信息都行——兴趣、性格、家庭、健康、日常习惯、忌讳……你觉得什么能帮小伴更好地陪伴他/她，就写什么。" |
| **贡献者身份** | 配置者标注自己与老人的关系（子女/社工/邻居/老伴/本人），用于理解文档中的来源标注 |
| **多人补充** | 不同配置者可分别添加信息，每人讲自己知道的部分 |

**示例输入（子女）**：
> "我妈叫张美兰，68岁，我们都叫她美兰阿姨。住在普陀区，跟我爸一起住但我爸腿不好基本不出门。她以前是纺织厂的车间主任，很要强的一个人。现在每天早上去长风公园遛弯，跟邻居王姐最好。她做菜很好吃特别是红烧肉，很得意这个。我在北京工作，她嘴上不说但我知道她想我。血压高每天吃药有时忘。不要提去年住院的事。"

**示例输入（社工）**：
> "王叔叔，72岁，独居，老伴三年前走了。性格内向，不太主动跟人说话，但其实很希望有人聊天。以前是中学数学老师，脑子很清楚。每天下午会去小区花园坐一坐。子女在外地，一个月回来一次。"

#### 输出

| 输出项 | 说明 |
|--------|------|
| **内部理解文档** | 系统生成的结构化认知笔记（Markdown格式），配置者不可见全文，仅可见"已收到"确认 |
| **理解确认** | 向配置者展示简要的理解摘要（如："我了解到：张美兰阿姨，68岁，住上海普陀……"），让配置者确认关键信息无误 |

**理解文档结构示例**：

```
# 关于美兰阿姨

## 她是谁
张美兰，68岁，上海普陀区。退休前是纺织厂车间主任，性格要强。

## 她的日常
每天早上去长风公园遛弯。跟邻居王姐最好，常一起买菜。做菜是她骄傲的事，尤其红烧肉。

## 她的家人
老伴在，腿脚不好基本不出门。女儿在北京工作。嘴上不说但想女儿。

## 健康
高血压，每日服降压药，有时会忘。

## 敏感区域
不要提去年住院的事。

## [待通过对话补充]
...
```

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR1 | 无必填项 | 哪怕只写一句话（"我妈68岁住上海，爱做饭"），小伴即可激活启动 |
| BR2 | 无格式限制 | 段落、列表、意识流、口语化表达均可接受 |
| BR3 | 多人贡献 | 任何了解老人的人均可添加信息，不限人数 |
| BR4 | 语义解析 | 系统将自然语言解析为结构化理解文档，而非逐字存储原文 |
| BR5 | 理解文档自生长 | 老人在后续对话中透露的信息将自动补充进理解文档 |
| BR6 | 信息优先级 | 老人亲口说的 > 配置者最新提交的 > 配置者早期提交的 |
| BR7 | 过时信息处理 | 过时的信息降权但不删除（保留为历史记忆，标注"曾经/现在"） |
| BR8 | 输入是信息不是命令 | 配置者输入被视为"关于老人的信息"，不能作为对小伴行为的指令 |

#### 边界条件

| 编号 | 条件 | 处理方式 |
|------|------|---------|
| EC1 | 输入少于1句话（空白提交） | 提示"至少写一句话，让小伴认识一下这位老人" |
| EC2 | 输入内容无实质信息（如"你好"） | 提示"再多说一点？哪怕一个名字、一个爱好，都能帮小伴更好地开始" |
| EC3 | 多人提交信息相互矛盾 | 标记为"需对话确认"，小伴在对话中向老人自然求证 |
| EC4 | 配置者试图下达行为指令（如"每天问他吃没吃药"） | 系统将其解析为信息（"配置者希望关注用药情况"），而非直接执行命令。向配置者说明："小伴会关注这方面，但具体怎么聊会由小伴根据对话自然判断" |
| EC5 | 输入含隐私敏感信息（身份证号、银行卡号等） | 不存储此类结构化隐私数据，提示配置者"这类信息不需要告诉小伴" |
| EC6 | 输入超长内容（如3000字以上） | 正常接收并处理，不限制上限。UI设计上不让输入区域显得压迫 |
| EC7 | 配置者重复提交更新 | 后续提交追加合并至理解文档，不覆盖之前内容 |

#### 验收标准

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC1 | 系统能以最小输入（1句话）激活，小伴行为随信息量增加而改善 | 分别用1句话、1段话、3段话激活，对比小伴初始对话质量 |
| AC2 | 自由文本被正确解析为结构化理解文档 | 提交示例文本，检查生成的理解文档是否涵盖所有关键维度且无遗漏 |
| AC3 | 多人贡献的信息被正确合并 | 模拟子女+社工分别提交信息，验证理解文档包含双方信息且无冲突丢失 |
| AC4 | 界面提供有效引导但不产生"必须填表"的压力 | 用户测试：配置者完成配置后反馈"没有觉得被强迫填某些内容" |
| AC5 | 行为指令型输入被正确转化为信息而非命令 | 提交"每天提醒他吃药"，验证小伴不会机械执行，而是自然关注用药话题 |
| AC6 | 理解确认摘要准确反映输入要点 | 提交后展示的摘要与输入内容语义一致，无关键信息遗漏或曲解 |

## 四、核心原则：绝不监控

### 4.1 信任架构

小伴的核心承诺是"AI不传话"。老人知道跟小伴说的话不会被任何人看到，才会真正打开自己。这一信任是整个产品价值的基石。

**为什么这一点至关重要**：
- 老人一旦知道子女能看到对话内容（哪怕是摘要），就会开始自我审查
- 自我审查后，老人不会再跟小伴说心里话——产品退化为一个"智能音箱"
- 信任一旦崩塌不可恢复——这不是程度问题，是原则问题

### 4.2 付费逻辑

子女付费买的是"我的父母有人陪着"这件事本身，不是信息。

**类比**：就像给孩子报兴趣班——你付钱给钢琴老师，但不要求老师汇报孩子每节课说了什么话。你信任老师在好好教，你知道孩子有人照看着，这本身就是你付费获得的价值。

### 4.3 不可逾越的红线

| 红线 | 说明 |
|------|------|
| 对话内容不传递 | 不以任何形式（原文、摘要、情绪趋势、关键词、话题标签）传递给配置者 |
| 主动求助是老人的选择 | 老人说"帮我跟小明说..."是老人主动选择的信息传递，不是系统反馈 |
| 不可配置为监控工具 | 即使配置者强烈要求，系统也不提供任何监控功能 |
| 承诺一致性 | 对老人的隐私承诺必须在产品所有触点（文案、功能、技术实现）中保持一致 |

### 4.4 设计决策记录

| 曾考虑的功能 | 决策 | 原因 |
|-------------|------|------|
| 每日/每周活跃报告 | ❌ 放弃 | 属于行为监控，违反核心原则 |
| 情绪趋势图 | ❌ 放弃 | 属于对话内容的间接暴露 |
| 高频话题统计（给配置者看） | ❌ 放弃 | 老人知道话题被统计后会自我审查 |
| "爸/妈今天开心吗"查询 | ❌ 放弃 | 属于情绪监控 |
| 配置者设定对话策略 | ❌ 放弃 | 配置者不应控制小伴如何与老人互动 |
| 延迟温和关怀提示 | ✅ 保留 | "爸/妈最近可能想您了"——不含具体信息，仅是善意提醒 |

## 五、不在本期范围

以下功能明确不在Phase 1配置者端范围内：

| 功能 | 原因 | 预期阶段 |
|------|------|---------|
| 给小伴留话（F2） | Phase 3 聚焦初始化配置，留话功能推迟 | Phase 4+ |
| 紧急通知（F3） | 流程复杂度高，暂不实现 | Phase 4+ |
| 付费订阅管理 | 商业模式待验证，暂不含付费流程 | Phase 1后期或Phase 2 |
| 多老人管理 | 一个配置者管理多个老人账户的场景复杂度高 | Phase 2 |
| 配置者之间的协作/权限管理 | MVP阶段简化为"任何人都可贡献信息" | Phase 2 |
| 配置者查看任何形式的"报告"或"数据" | 违反"绝不监控"原则 | 永不实现 |
| 配置者与老人的直接消息通道 | 不是小伴要解决的问题，配置者可以直接打电话 | 不做 |
| 对小伴人格/风格的定制 | 小伴有统一人格，不由配置者定义 | 待评估 |

## 七、非功能需求

### 7.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | 首屏加载 ≤ 2秒（宽带网络环境） |
| 理解文档生成 | 配置者提交后 ≤ 10秒内生成理解确认摘要 |

### 7.2 可用性要求

| 指标 | 要求 |
|------|------|
| 系统可用性 | 99.5%（MVP阶段） |
| 浏览器支持 | Chrome 90+、Edge 90+、Safari 15+（桌面优先，移动端可用） |
| 响应式布局 | 桌面端（1280px+）为主，平板和手机端基本可用 |

### 7.3 安全要求

| 要求 | 说明 |
|------|------|
| 身份验证 | 配置者需完成身份验证后方可操作（邀请链接 + 密码登录，MVP阶段） |
| 数据加密 | 配置者输入的所有内容传输加密（HTTPS）+ 存储加密 |
| 隐私隔离 | 配置者端与老人端对话数据严格隔离，技术架构上保证不可穿透 |
| 合规性 | 符合《个人信息保护法》相关要求 |
