# 小伴 — 配置者端产品需求文档（PRD）

> 文档版本：v1.0（2026-05-14）  
> 文档性质：产品需求文档（PRD），非技术设计文档  
> 适用范围：Phase 1 配置者端功能定义

---

## 一、产品概述

### 1.1 产品定位

配置者端是小伴产品体系中面向"了解老人的人"的操作界面，承载初始化设置与有限互动功能。配置者端以小程序或H5页面形态存在，为配置者提供一个轻量、低门槛的入口。

### 1.2 产品目标

让了解老人的人（子女、社工、邻居、老伴、老人自己）能够：
1. 用最自然的方式向小伴介绍这位老人，帮助小伴快速建立对老人的初始认知
2. 在日常生活中通过小伴向老人传递关心
3. 在紧急情况下及时获得通知

### 1.3 Phase 1 范围

- **产品形态**：微信小程序 / H5页面
- **核心功能**：初始化配置、给小伴留话、紧急通知（共3个功能模块）
- **不含功能**：付费订阅管理、多老人管理、配置者协作/权限、任何形式的"报告"或"数据"

### 1.4 核心原则

**"绝不监控"**——配置者付费购买的是"我的父母有人陪着"这件事本身，不是信息。小伴的信任基础是老人知道"我跟小伴说的话不会被任何人看到"，这一承诺不可打破。

## 二、用户画像

### 2.1 主要用户：异地子女

| 维度 | 描述 |
|------|------|
| **年龄段** | 30-50岁 |
| **生活状态** | 在不同城市工作，无法每天陪伴父母 |
| **心理特征** | 对"不能经常打电话/回家"有内疚感；希望父母生活有人关照 |
| **数字能力** | 互联网原住民，熟悉小程序/H5操作 |
| **付费意愿** | 愿意为父母的陪伴和安全感付费（孝心经济） |
| **核心动机** | "我不能天天在身边，但至少让父母有人陪着" |

**关键洞察**：子女付费买的不是信息，是"我的父母有人陪着"这件事本身。就像给孩子报兴趣班，付钱但不要求老师汇报每句话。

### 2.2 次要用户

| 用户类型 | 典型场景 | 特点 |
|---------|---------|------|
| **社工** | 社区养老服务人员，负责多位老人的日常关怀 | 对老人的了解偏生活层面，配置内容偏客观 |
| **邻居** | 与老人关系密切的邻里，主动帮助配置 | 了解老人的日常习惯和社交圈 |
| **老伴** | 老人的配偶，补充另一视角的信息 | 了解老人的深层偏好和忌讳 |
| **老人自己** | 老人自己上手填写或补充信息 | 最真实的第一手信息来源 |

### 2.3 用户场景故事

**场景A：子女首次开通**  
小明在北京工作，妈妈张美兰68岁独居上海。小明在朋友推荐下了解到小伴，打开小程序后看到一段引导语，花10分钟写了一段关于妈妈的介绍，小伴即刻为妈妈开通服务。

**场景B：子女日常留话**  
小明周五加班没来得及给妈妈打电话，在小程序里留了一句"跟我妈说我周末回来看她"。第二天小伴在和张阿姨聊天时自然地说："哎对了阿姨，小明说周末要回来看您呢。"

**场景C：紧急通知**  
张阿姨跟小伴聊天时说"帮我打电话给小明，我头有点晕"。小伴立即给小明推送通知："妈妈请您给她打个电话。"

## 三、功能模块

### F1: 初始化配置（开放叙事式）

#### 描述

配置者通过自由书写的方式向小伴介绍老人。没有表单、没有必填字段、没有格式要求。就像跟一个新朋友讲"我妈是个怎样的人"——你想到什么说什么，什么格式都行。系统将自然语言输入解析为结构化的"理解文档"，作为小伴认知老人的起点。

#### 输入

| 输入项 | 说明 |
|--------|------|
| **自由文本** | 配置者用自然语言描述老人，任何长度、任何格式、任何维度均可 |
| **引导提示语** | "用你自己的话，跟小伴介绍一下这位老人。什么信息都行——兴趣、性格、家庭、健康、日常习惯、忌讳……你觉得什么能帮小伴更好地陪伴他/她，就写什么。" |
| **贡献者身份** | 配置者标注自己与老人的关系（子女/社工/邻居/老伴/本人），用于理解文档中的来源标注 |
| **多人补充** | 不同配置者可分别添加信息，每人讲自己知道的部分 |

**示例输入（子女）**：
> "我妈叫张美兰，68岁，我们都叫她美兰阿姨。住在普陀区，跟我爸一起住但我爸腿不好基本不出门。她以前是纺织厂的车间主任，很要强的一个人。现在每天早上去长风公园遛弯，跟邻居王姐最好。她做菜很好吃特别是红烧肉，很得意这个。我在北京工作，她嘴上不说但我知道她想我。血压高每天吃药有时忘。不要提去年住院的事。"

**示例输入（社工）**：
> "王叔叔，72岁，独居，老伴三年前走了。性格内向，不太主动跟人说话，但其实很希望有人聊天。以前是中学数学老师，脑子很清楚。每天下午会去小区花园坐一坐。子女在外地，一个月回来一次。"

#### 输出

| 输出项 | 说明 |
|--------|------|
| **内部理解文档** | 系统生成的结构化认知笔记（Markdown格式），配置者不可见全文，仅可见"已收到"确认 |
| **理解确认** | 向配置者展示简要的理解摘要（如："我了解到：张美兰阿姨，68岁，住上海普陀……"），让配置者确认关键信息无误 |

**理解文档结构示例**：

```
# 关于美兰阿姨

## 她是谁
张美兰，68岁，上海普陀区。退休前是纺织厂车间主任，性格要强。

## 她的日常
每天早上去长风公园遛弯。跟邻居王姐最好，常一起买菜。做菜是她骄傲的事，尤其红烧肉。

## 她的家人
老伴在，腿脚不好基本不出门。女儿在北京工作。嘴上不说但想女儿。

## 健康
高血压，每日服降压药，有时会忘。

## 敏感区域
不要提去年住院的事。

## [待通过对话补充]
...
```

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR1 | 无必填项 | 哪怕只写一句话（"我妈68岁住上海，爱做饭"），小伴即可激活启动 |
| BR2 | 无格式限制 | 段落、列表、意识流、口语化表达均可接受 |
| BR3 | 多人贡献 | 任何了解老人的人均可添加信息，不限人数 |
| BR4 | 语义解析 | 系统将自然语言解析为结构化理解文档，而非逐字存储原文 |
| BR5 | 理解文档自生长 | 老人在后续对话中透露的信息将自动补充进理解文档 |
| BR6 | 信息优先级 | 老人亲口说的 > 配置者最新提交的 > 配置者早期提交的 |
| BR7 | 过时信息处理 | 过时的信息降权但不删除（保留为历史记忆，标注"曾经/现在"） |
| BR8 | 输入是信息不是命令 | 配置者输入被视为"关于老人的信息"，不能作为对小伴行为的指令 |

#### 边界条件

| 编号 | 条件 | 处理方式 |
|------|------|---------|
| EC1 | 输入少于1句话（空白提交） | 提示"至少写一句话，让小伴认识一下这位老人" |
| EC2 | 输入内容无实质信息（如"你好"） | 提示"再多说一点？哪怕一个名字、一个爱好，都能帮小伴更好地开始" |
| EC3 | 多人提交信息相互矛盾 | 标记为"需对话确认"，小伴在对话中向老人自然求证 |
| EC4 | 配置者试图下达行为指令（如"每天问他吃没吃药"） | 系统将其解析为信息（"配置者希望关注用药情况"），而非直接执行命令。向配置者说明："小伴会关注这方面，但具体怎么聊会由小伴根据对话自然判断" |
| EC5 | 输入含隐私敏感信息（身份证号、银行卡号等） | 不存储此类结构化隐私数据，提示配置者"这类信息不需要告诉小伴" |
| EC6 | 输入超长内容（如3000字以上） | 正常接收并处理，不限制上限。UI设计上不让输入区域显得压迫 |
| EC7 | 配置者重复提交更新 | 后续提交追加合并至理解文档，不覆盖之前内容 |

#### 验收标准

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC1 | 系统能以最小输入（1句话）激活，小伴行为随信息量增加而改善 | 分别用1句话、1段话、3段话激活，对比小伴初始对话质量 |
| AC2 | 自由文本被正确解析为结构化理解文档 | 提交示例文本，检查生成的理解文档是否涵盖所有关键维度且无遗漏 |
| AC3 | 多人贡献的信息被正确合并 | 模拟子女+社工分别提交信息，验证理解文档包含双方信息且无冲突丢失 |
| AC4 | 界面提供有效引导但不产生"必须填表"的压力 | 用户测试：配置者完成配置后反馈"没有觉得被强迫填某些内容" |
| AC5 | 行为指令型输入被正确转化为信息而非命令 | 提交"每天提醒他吃药"，验证小伴不会机械执行，而是自然关注用药话题 |
| AC6 | 理解确认摘要准确反映输入要点 | 提交后展示的摘要与输入内容语义一致，无关键信息遗漏或曲解 |

### F2: 给小伴留话

#### 描述

配置者可以向小伴留言，小伴会在与老人的对话中自然地将留言内容传达给老人。这不是消息直接转发，而是由小伴将配置者的心意融入自然对话中，择时、择机、以合适的方式表达。老人的感受应该是"小伴顺便提了一嘴"，而不是"系统转达了一条消息"。

#### 输入

| 输入项 | 是否必填 | 说明 |
|--------|---------|------|
| **留言内容** | 必填 | 配置者想通过小伴传达给老人的话（自然语言） |
| **紧急程度** | 选填 | 普通（默认） / 及时；影响送达时效 |

**输入示例**：
- "跟我妈说我下周回来"
- "告诉爸爸我给他买了新茶叶寄过去了"
- "让妈妈明天记得带伞，有雨"
- "跟爸说我升职了，让他高兴高兴"

#### 输出

| 输出项 | 接收方 | 说明 |
|--------|--------|------|
| **自然化转达** | 老人（通过小伴对话） | 小伴将留言转化为自然对话的一部分，融入恰当的对话时机传达 |
| **送达确认** | 配置者 | 留言被成功传达后，配置者收到"已转达"通知 |

**转达方式示例**：

| 配置者原话 | 小伴对老人的表达方式 |
|-----------|-------------------|
| "跟我妈说我下周回来" | "哎对了阿姨，小明说下周要回来看您呢。" |
| "告诉爸爸我给他买了新茶叶" | "叔叔，听说小明给您寄了新茶叶，是什么茶啊？" |
| "让妈妈明天记得带伞" | "阿姨，明天有雨，出门记得带伞——小明也说怕您淋着。" |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR1 | 非逐字转发 | 小伴必须将留言转化为自然对话语言，禁止机械转发原文 |
| BR2 | 时效性要求 | 普通留言：在24小时内的恰当对话时机传达；及时留言：在4小时内传达 |
| BR3 | 自然时机选择 | 小伴在已有对话中寻找自然的插入时机，不为传达留言强制发起新对话 |
| BR4 | 送达确认 | 传达成功后向配置者推送"已转达"确认（不含对话上下文） |
| BR5 | 队列管理 | 如老人超过24小时未与小伴互动，留言进入队列等待下次互动再传达 |
| BR6 | 留言数量上限 | 最多同时存在3条未送达留言；新增留言排队，不挤掉旧留言 |
| BR7 | 留言内容边界 | 留言必须是"一个关心老人的家人会自然说的话"，不得用于下达指令或索取信息 |

#### 边界条件

| 编号 | 条件 | 处理方式 |
|------|------|---------|
| EC1 | 留言含监控意图（如"问她今天去了哪里"、"看看他有没有按时吃药"） | 系统拒绝并说明："小伴是陪伴者，不是监控工具。如果您担心爸/妈的身体，可以直接打电话关心哦。" |
| EC2 | 留言试图控制小伴行为（如"以后每天提醒她吃药"） | 系统拒绝并说明："这个功能用于给老人传达您的关心，不能设置小伴的行为规则。" |
| EC3 | 留言内容可能引发老人负面情绪（如涉及家庭矛盾） | 系统正常接收但小伴在转达时会自行判断时机和方式，可能延后或柔化表达 |
| EC4 | 已有3条未送达留言时再次提交 | 提示"还有3条留言等待传达，请等前面的到达后再留新的" |
| EC5 | 老人长期未互动（超过72小时） | 向配置者说明"老人最近没有跟小伴互动，留言会在下次对话时传达"；不暴露具体活跃数据 |
| EC6 | 留言内容为空或无实质内容 | 提示"写点什么让小伴转达给爸/妈吧" |
| EC7 | 及时留言但当前时段不适合打扰老人（如深夜） | 延迟至合适时段（次日上午）传达，向配置者说明"考虑到时间，会在明天上午合适的时候转达" |

#### 验收标准

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC1 | 普通留言在24小时内、及时留言在4小时内被传达 | 提交留言后追踪实际送达时间，验证符合时效要求 |
| AC2 | 老人听到转达时感觉自然，不像"系统播报" | 用户测试：老人不会产生"这是系统转发的消息"的感受 |
| AC3 | 配置者在留言送达后收到确认通知 | 模拟完整流程，验证确认通知到达且不含对话内容 |
| AC4 | 监控型、指令型留言被正确拒绝并给出清晰说明 | 提交"问她今天做了什么"等测试用例，验证系统拒绝并给出合理解释 |
| AC5 | 队列满时新留言被正确排队 | 连续提交4条留言，验证第4条被排队且给出提示 |
| AC6 | 送达确认中不泄露任何对话上下文 | 检查确认通知内容，确认仅含"已转达"字样，无对话片段 |

### F3: 紧急通知

#### 描述

紧急通知系统在严格定义的触发条件下向配置者发送通知。这不是监控系统——它不报告老人的日常状态、情绪变化或活动轨迹。它只在老人主动求助或系统检测到潜在紧急情况时发挥作用，确保"关键时刻不缺位"。

#### 输入（触发条件）

系统仅在以下两种情况下触发通知，**无其他条件可触发**：

| 触发类型 | 触发条件 | 示例 |
|---------|---------|------|
| **老人主动求助** | 老人在对话中明确表达需要联系家人的意愿 | "帮我打电话给小明"、"让我女儿来一下"、"叫小明回来"、"帮我联系我儿子" |
| **系统检测紧急状况** | 对话中出现疑似危及人身安全的信号 | 老人描述胸痛/摔倒/突然身体不适；对话中提及"120"、"救护车"；老人表达严重不适且需要立即帮助 |

#### 输出

| 输出项 | 内容格式 | 说明 |
|--------|---------|------|
| **推送通知（主动求助型）** | "妈妈请您给她打个电话" / "爸爸让您有空联系他" | 简洁传达老人的请求，不附带对话内容 |
| **推送通知（紧急检测型）** | "系统检测到可能的紧急情况，建议您联系父母确认安全" | 不透露具体对话内容，仅提示需要关注 |
| **对老人的响应** | 生命威胁场景下建议老人拨打120 | "叔叔，您现在要不要先拨打120？我同时帮您通知小明" |

#### 业务规则

| 编号 | 规则 | 说明 |
|------|------|------|
| BR1 | 触发条件封闭 | 只有上述两种触发条件可以产生通知，任何其他对话内容不得触发 |
| BR2 | 通知时效 | 触发后2分钟内送达配置者 |
| BR3 | 通知内容极简 | 仅传达"需要您关注"的信号，绝不包含对话原文、摘要或情绪分析 |
| BR4 | 紧急检测保守策略 | 宁可漏报非致命情况，不可对非紧急对话误报 |
| BR5 | 生命威胁响应 | 当检测到胸痛+摔倒+提及120等高危信号组合时，同时建议老人拨打120并通知配置者 |
| BR6 | 非紧急情绪关怀 | 老人连续多日情绪低落但无紧急安全风险时，可发送延迟的温和提示："爸/妈最近可能想您了"（非实时，非警报） |
| BR7 | 信息不对称原则 | 通知中绝不包含"老人说了什么"——只告诉配置者"需要做什么" |

#### 边界条件

| 编号 | 条件 | 处理方式 |
|------|------|---------|
| EC1 | 老人说"别告诉我女儿/儿子" | 非生命威胁时尊重老人意愿，不发送通知。生命威胁时仍发送（以安全优先） |
| EC2 | 老人只是随口抱怨不舒服但无紧急信号 | 不触发通知。例如"今天有点头疼"不触发，"我头突然很晕站不稳"触发 |
| EC3 | 配置者主动要求获取信息（如"我妈今天怎么样"） | 系统不提供任何回复，说明："小伴不提供老人的活动或状态信息。如果想知道爸/妈怎么样，最好的方式是直接给他们打个电话。" |
| EC4 | 配置者要求开启更多通知类型（如日报、情绪趋势） | 明确拒绝并说明产品原则："为了让老人能真正信任小伴说心里话，我们承诺不传递任何对话信息给家人。" |
| EC5 | 老人长时间未响应小伴（如超过48小时） | 不自动向配置者报告。但小伴会在下次触达时主动关心老人 |
| EC6 | 误触发紧急通知 | 配置者收到通知后确认老人安全，可标记"误报"，系统学习优化检测模型 |
| EC7 | 通知推送失败（网络问题等） | 系统自动重试3次，间隔30秒。若仍失败则尝试短信通知 |

#### 明确排除项

以下信息**在任何情况下**都不会传递给配置者，即使配置者主动要求：

| 排除项 | 说明 |
|--------|------|
| ❌ 对话内容（原文或摘要） | 老人与小伴的任何对话文字不会被配置者看到 |
| ❌ 情绪/健康趋势图 | 不提供老人情绪变化或健康指标的图表或数据 |
| ❌ 活动状态追踪 | 不提供老人的活跃时间、对话频率等行为数据 |
| ❌ 每日/每周报告 | 不生成任何形式的定期报告给配置者 |
| ❌ 对小伴的行为控制 | 配置者不能设定小伴的对话策略或行为规则 |

#### 验收标准

| 编号 | 验收条件 | 验证方式 |
|------|---------|---------|
| AC1 | 紧急通知在触发后2分钟内送达配置者 | 模拟触发条件，测量从触发到配置者收到通知的时间 |
| AC2 | 非紧急对话不产生任何通知（零误报） | 模拟大量日常对话（含老人抱怨、倾诉等），验证无通知被触发 |
| AC3 | 老人主动求助的请求被正确转发 | 模拟"帮我打电话给XX"等表述，验证通知正确触达 |
| AC4 | 通知内容中不包含任何对话原文、情绪数据或活动数据 | 审查所有通知文案模板，确认无信息泄露 |
| AC5 | 老人的"别告诉XX"请求在非生命威胁时被尊重 | 模拟老人要求保密 + 非紧急场景，验证系统不发通知 |
| AC6 | 生命威胁时即使老人说"别告诉家人"仍触发通知 | 模拟老人描述严重症状并要求保密，验证系统仍通知配置者并建议老人拨打120 |
| AC7 | 配置者无法通过任何操作获取对话内容或活动数据 | 穷举配置者端所有可操作路径，确认无信息获取入口 |

## 四、核心原则：绝不监控

### 4.1 信任架构

小伴的核心承诺是"AI不传话"。老人知道跟小伴说的话不会被任何人看到，才会真正打开自己。这一信任是整个产品价值的基石。

**为什么这一点至关重要**：
- 老人一旦知道子女能看到对话内容（哪怕是摘要），就会开始自我审查
- 自我审查后，老人不会再跟小伴说心里话——产品退化为一个"智能音箱"
- 信任一旦崩塌不可恢复——这不是程度问题，是原则问题

### 4.2 付费逻辑

子女付费买的是"我的父母有人陪着"这件事本身，不是信息。

**类比**：就像给孩子报兴趣班——你付钱给钢琴老师，但不要求老师汇报孩子每节课说了什么话。你信任老师在好好教，你知道孩子有人照看着，这本身就是你付费获得的价值。

### 4.3 不可逾越的红线

| 红线 | 说明 |
|------|------|
| 对话内容不传递 | 不以任何形式（原文、摘要、情绪趋势、关键词、话题标签）传递给配置者 |
| 主动求助是老人的选择 | 老人说"帮我跟小明说..."是老人主动选择的信息传递，不是系统反馈 |
| 不可配置为监控工具 | 即使配置者强烈要求，系统也不提供任何监控功能 |
| 承诺一致性 | 对老人的隐私承诺必须在产品所有触点（文案、功能、技术实现）中保持一致 |

### 4.4 设计决策记录

| 曾考虑的功能 | 决策 | 原因 |
|-------------|------|------|
| 每日/每周活跃报告 | ❌ 放弃 | 属于行为监控，违反核心原则 |
| 情绪趋势图 | ❌ 放弃 | 属于对话内容的间接暴露 |
| 高频话题统计（给配置者看） | ❌ 放弃 | 老人知道话题被统计后会自我审查 |
| "爸/妈今天开心吗"查询 | ❌ 放弃 | 属于情绪监控 |
| 配置者设定对话策略 | ❌ 放弃 | 配置者不应控制小伴如何与老人互动 |
| 延迟温和关怀提示 | ✅ 保留 | "爸/妈最近可能想您了"——不含具体信息，仅是善意提醒 |

## 五、不在本期范围

以下功能明确不在Phase 1配置者端范围内：

| 功能 | 原因 | 预期阶段 |
|------|------|---------|
| 付费订阅管理 | 商业模式待验证，暂不含付费流程 | Phase 1后期或Phase 2 |
| 多老人管理 | 一个配置者管理多个老人账户的场景复杂度高 | Phase 2 |
| 配置者之间的协作/权限管理 | MVP阶段简化为"任何人都可贡献信息" | Phase 2 |
| 配置者查看任何形式的"报告"或"数据" | 违反"绝不监控"原则 | 永不实现 |
| 配置者与老人的直接消息通道 | 不是小伴要解决的问题，配置者可以直接打电话 | 不做 |
| 对小伴人格/风格的定制 | 小伴有统一人格，不由配置者定义 | 待评估 |

## 六、跨端关联

配置者端不是独立系统，其功能与老人端紧密联动：

| 配置者端功能 | 关联的老人端系统 | 关联方式 |
|-------------|----------------|---------|
| F1 初始化配置 | 老人端记忆系统（理解文档） | 配置者输入形成理解文档的种子内容，老人端对话持续生长更新 |
| F2 给小伴留话 | 老人端自然对话系统 | 留言进入小伴的"待传达队列"，在对话中自然择机融入 |
| F3 紧急通知 | 老人端守护系统（后台运行） | 守护系统检测到触发条件后，向配置者端推送通知 |

## 七、非功能需求

### 7.1 性能要求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | 首屏加载 ≤ 2秒（4G网络环境） |
| 留话提交响应 | 提交后 ≤ 1秒内反馈"已收到" |
| 紧急通知时效 | 触发条件满足后 ≤ 2分钟送达 |
| 理解文档生成 | 配置者提交后 ≤ 10秒内生成理解确认摘要 |

### 7.2 可用性要求

| 指标 | 要求 |
|------|------|
| 系统可用性 | 99.9%（紧急通知通道） |
| 多端适配 | 支持iOS/Android微信小程序及主流浏览器H5 |
| 最低系统版本 | 微信 7.0+，iOS 13+，Android 8+ |

### 7.3 安全要求

| 要求 | 说明 |
|------|------|
| 身份验证 | 配置者需完成身份验证后方可操作（微信授权登录） |
| 数据加密 | 配置者输入的所有内容传输加密（HTTPS）+ 存储加密 |
| 隐私隔离 | 配置者端与老人端对话数据严格隔离，技术架构上保证不可穿透 |
| 合规性 | 符合《个人信息保护法》相关要求 |
