## 二、F1: 对话记录存储管线（Conversation Storage Pipeline）

### 2.1 架构设计

F1 不是用户可见的UI模块，而是整个运营后台的**数据基础设施**。它是一条异步数据摄入管道，负责将来自两个渠道的对话数据可靠地写入PostgreSQL。

**核心设计原则**：
- **零数据丢失**：任何情况下对话消息不丢失
- **幂等性**：同一条消息重复投递不产生重复数据
- **异步解耦**：存储异常绝不影响用户侧对话体验
- **有序性**：同一Session内消息按sequence字段严格有序

**组件关系**：

```
生产者                           消费者                        存储
┌─────────────────┐          ┌─────────────────────┐      ┌───────────┐
│ Dialogue Service │─────────→│ Celery Worker         │      │PostgreSQL │
│ (企微文字消息)    │          │ (Storage Task)        │      │           │
└─────────────────┘          │                       │      │ sessions  │
                             │ • Session边界检测      │─────→│ messages  │
                             │ • 幂等写入            │      │           │
                             │ • 自动重试管理        │      └───────────┘
                             │                       │
                             └─────────────────────┘
                                       │
                                       ▼
                              tasks.conversation.store 完成
                              （触发 F4 标签 + 记忆提取）
```

### 2.2 数据流

```
[企业微信文字渠道]
Dialogue Service 处理完对话
  → 发送 Celery 任务到 tasks.conversation.store 队列
    Payload: { elder_id, session_id, messages: [{sender, content, timestamp, channel}] }
  → Celery Worker 执行 store_conversation 任务:
    → 检查 session 是否已存在
      → 不存在: 创建新 Session 记录 (判断是否需要新Session: 距上条消息>30min)
      → 存在: 复用现有 Session
    → 写入 messages 表 (批量insert)
    → 更新 session.turn_count, session.ended_at
    → 写入成功 → 任务正常返回
    → 写入失败 → 抛出异常 → Celery 自动重试（指数退避）
```

### 2.4 核心算法/逻辑

#### 2.4.1 Session边界检测算法

```python
def determine_session(elder_id: str, channel: str, message_timestamp: datetime) -> Session:
    """
    判断新消息属于哪个Session。
    
    规则：
    - 企微渠道：距上条消息 > 30分钟 → 新Session；否则复用当前Session
    """
    if channel == 'phone':
        return None  # 调用方必须提供 session_id

    # 企微渠道：30分钟间隔规则
    last_message = db.query(
        "SELECT session_id, timestamp FROM messages "
        "WHERE elder_id = :elder_id AND channel = 'wechat' "
        "ORDER BY timestamp DESC LIMIT 1",
        elder_id=elder_id
    )

    if last_message is None:
        # 该用户首条消息 → 创建新Session
        return create_session(
            elder_id=elder_id,
            channel='wechat',
            started_at=message_timestamp
        )

    time_gap = message_timestamp - last_message.timestamp

    if time_gap > timedelta(minutes=30):
        # 间隔 > 30分钟 → 结束旧Session，开启新Session
        end_session(last_message.session_id, ended_at=last_message.timestamp)
        return create_session(
            elder_id=elder_id,
            channel='wechat',
            started_at=message_timestamp
        )
    else:
        # 同一Session
        return get_session(last_message.session_id)
```

#### 2.4.2 幂等性写入

```python
def store_messages_batch(session_id: str, messages: list[dict]) -> StoreResult:
    """
    幂等批量写入消息。
    使用 INSERT ON CONFLICT DO NOTHING 防止重复写入。
    """
    stored_count = 0
    duplicates_skipped = 0

    for msg in messages:
        result = db.execute(
            "INSERT INTO messages (id, session_id, elder_id, sender, content, channel, sequence, timestamp) "
            "VALUES (:id, :session_id, :elder_id, :sender, :content, :channel, :sequence, :timestamp) "
            "ON CONFLICT (id) DO NOTHING",
            **msg
        )
        if result.rowcount > 0:
            stored_count += 1
        else:
            duplicates_skipped += 1
            logger.info(f"幂等跳过重复消息: {msg['id']}")

    # 更新Session统计
    if stored_count > 0:
        db.execute(
            "UPDATE sessions SET "
            "turn_count = (SELECT COUNT(*) FROM messages WHERE session_id = :sid AND sender = 'elder'), "
            "ended_at = (SELECT MAX(timestamp) FROM messages WHERE session_id = :sid) "
            "WHERE id = :sid",
            sid=session_id
        )

    return StoreResult(stored_count=stored_count, duplicates_skipped=duplicates_skipped)
```

```
重试与失败处理策略:
├── 最大重试次数: 10（零丢失要求，比普通任务更多）
├── 退避策略: 指数退避 + 抖动（Celery retry_backoff=True）
├── 最大退避时间: 600秒
├── 死信队列: 超过最大重试后路由到 dead_letter 队列
│   └── 监控: 若 dead_letter 队列有任务，企微机器人立即告警技术团队
│
├── 失败分类与处理:
│   ├── DB写入超时 → 正常重试（瞬态故障）
│   ├── DB连接被拒 → 退避重试（DB可能在重启）
│   ├── 数据校验错误 → 立即移入 dead_letter（不重试，需人工修复）
│   └── 幂等冲突（重复消息）→ 任务正常返回（已处理，安全丢弃）
```

### 2.5 错误处理

| 故障场景 | 处理策略 | 影响 |
|---------|---------|------|
| 数据库不可达 | 任务自动重试（指数退避，max_retries=10）；超过后进入 dead_letter 队列 | 对话延迟存储，但零丢失 |
| 消息乱序到达 | 以 `sequence` 字段为准排序，写入时不依赖到达顺序 | 无影响 |
| 时钟偏移 | 对于时间戳极接近的消息（<100ms差异），以服务端接收时间为准 | 无影响 |
| 存储层异常 | 完全异步，绝不阻塞用户侧对话（Dialogue Service fire-and-forget） | 用户无感知 |

### 3.5 核心算法/逻辑

#### 3.5.1 用户列表查询优化

```sql
-- 方案一：直接查询（数据量<50用户时足够快）
SELECT
    e.id,
    e.nickname,
    e.status,
    e.created_at,
    MAX(m.timestamp) AS last_active_at,
    COUNT(DISTINCT s.id) AS total_sessions,
    COUNT(DISTINCT s.id) FILTER (
        WHERE s.started_at > NOW() - INTERVAL '7 days'
    ) AS this_week_sessions
FROM elders e
LEFT JOIN sessions s ON s.elder_id = e.id
LEFT JOIN messages m ON m.elder_id = e.id
WHERE (:search IS NULL OR e.nickname ILIKE '%' || :search || '%')
GROUP BY e.id
ORDER BY last_active_at DESC NULLS LAST
LIMIT :page_size OFFSET (:page - 1) * :page_size;

-- 方案二：物化视图（用户量增长后启用）
CREATE MATERIALIZED VIEW mv_user_stats AS
SELECT
    elder_id,
    MAX(timestamp) AS last_active_at,
    COUNT(DISTINCT session_id) AS total_sessions,
    COUNT(DISTINCT session_id) FILTER (
        WHERE timestamp > NOW() - INTERVAL '7 days'
    ) AS this_week_sessions
FROM messages
GROUP BY elder_id;

-- 定时刷新（每5分钟）
```

#### 3.5.2 时间线生成逻辑

```python
def generate_timeline(elder_id: str, page: int, page_size: int) -> TimelineResponse:
    """
    生成用户时间线，包含有互动日和无互动日。
    从用户首次互动到今天，连续展示每一天。
    """
    # 1. 获取用户首次互动日期
    first_interaction = db.query(
        "SELECT MIN(DATE(timestamp)) FROM messages WHERE elder_id = :elder_id",
        elder_id=elder_id
    )

    if first_interaction is None:
        return TimelineResponse(total_days=0, items=[])

    # 2. 生成日期范围（首次互动日 → 今天）
    today = date.today()
    total_days = (today - first_interaction).days + 1

    # 3. 分页计算日期范围
    start_date = today - timedelta(days=(page - 1) * page_size)
    end_date = start_date - timedelta(days=page_size - 1)
    end_date = max(end_date, first_interaction)

    # 4. 查询该日期范围内的对话统计
    daily_stats = db.query(
        """
        SELECT
            DATE(m.timestamp) AS date,
            COUNT(DISTINCT s.id) AS session_count,
            COUNT(*) FILTER (WHERE m.sender = 'elder') AS turn_count,
            ARRAY_AGG(DISTINCT s.channel) AS channels,
            MIN(m.content) FILTER (WHERE m.sender = 'elder') AS first_message
        FROM messages m
        JOIN sessions s ON m.session_id = s.id
        WHERE m.elder_id = :elder_id
          AND DATE(m.timestamp) BETWEEN :end_date AND :start_date
        GROUP BY DATE(m.timestamp)
        ORDER BY date DESC
        """,
        elder_id=elder_id, start_date=start_date, end_date=end_date
    )

    # 5. 填充无互动日
    stats_by_date = {row.date: row for row in daily_stats}
    items = []
    current = start_date
    while current >= end_date:
        if current in stats_by_date:
            row = stats_by_date[current]
            items.append(TimelineItem(
                date=current,
                session_count=row.session_count,
                total_turns=row.turn_count,
                summary=row.first_message[:20] if row.first_message else None,
                channels=row.channels,
                has_interaction=True
            ))
        else:
            items.append(TimelineItem(
                date=current,
                session_count=0,
                total_turns=0,
                summary=None,
                channels=[],
                has_interaction=False
            ))
        current -= timedelta(days=1)

    return TimelineResponse(total_days=total_days, items=items)
```

#### 3.5.3 用户状态计算

```python
def calculate_user_status(elder: Elder, last_active_at: datetime | None) -> str:
    """
    计算用户活跃状态标签。
    规则：
    - 注册<3天 → 'new'（新手保护期）
    - 最近3天有互动 → 'active'
    - 3-7天无互动 → 'silent'
    - >7天无互动 → 'churn_risk'
    """
    now = datetime.now(tz=timezone.utc)

    # 新手保护期：注册不满3天
    if (now - elder.created_at).days < 3:
        return 'new'

    if last_active_at is None:
        return 'churn_risk'

    days_since_active = (now - last_active_at).days

    if days_since_active <= 3:
        return 'active'
    elif days_since_active <= 7:
        return 'silent'
    else:
        return 'churn_risk'
```

## 四、F4: 意图标签管线（Intent Tagging Pipeline）

### 4.1 架构设计

F4 是一条异步处理管线，在每轮对话存储完成后自动触发LLM进行意图分类，结果写入 `message_tags` 表，并为F5话题统计和F7未满足需求提供数据基础。

**核心设计原则**：
- **异步不阻塞**：标签任务不影响对话存储和用户体验
- **上下文感知**：分类时参考对话上下文（前3轮），避免对短消息误判
- **可修正**：支持人工覆盖AI标注结果
- **可扩展**：标签集可配置，新增/停用标签不影响历史数据

**组件关系**：

```
tasks.conversation.store 完成
       │
       ▼
tasks.tags.classify 任务
       │
       ▼
┌──────────────────────────┐
│ Celery Worker            │
│ (classify_tags task)     │
│                          │
│ 1. 获取对话轮内容         │
│ 2. 获取上下文(前3轮)      │
│ 3. 获取活跃标签集         │
│ 4. 调用LLM分类           │──→ LLM API (DeepSeek/Qwen)
│ 5. 解析输出              │
│ 6. 写入 message_tags     │──→ PostgreSQL
│ 7. 低置信度标记待审核     │
│ 8. 触发F7(任务类标签)     │──→ tasks.unmet.detect 任务
└──────────────────────────┘
```

### 4.2 数据流

```
tasks.conversation.store 任务完成
  → 触发 tasks.tags.classify 任务
    → Celery Worker 执行 classify_tags:
      → 获取该轮对话内容 (elder message + xiaoban reply)
      → 获取对话上下文 (同Session前3轮, 用于理解语境)
      → 获取当前活跃标签集 (intent_tags WHERE is_active=true)
      → 调用LLM进行意图分类
      → 解析LLM输出 → 提取标签 + 置信度
      → 写入 message_tags 表
        → confidence < 0.6: needs_review = true
      → 如果标签含任务类（购物需求/出行需求/任务委托）:
        → 发送 tasks.unmet.detect 任务
      → 任务正常返回
```

#### 4.4.1 LLM分类Prompt设计

```python
INTENT_CLASSIFICATION_PROMPT = """
你是一个对话意图分类器。请分析以下老人与AI伙伴"小伴"的对话，判断老人这轮对话的意图/话题。

可选标签（可多选）：
{available_tags}

对话上下文（用于理解语境，最近3轮）：
{context_messages}

当前轮对话：
老人：{elder_message}
小伴：{xiaoban_reply}

请输出JSON格式：
{{
    "tags": ["标签1", "标签2"],
    "confidence": 0.85,
    "reasoning": "简短理由（20字以内）"
}}

注意：
1. 短消息如"嗯""好的""哈哈"要结合上下文判断，不要轻易标为"其他"
2. 一轮对话可以有多个标签（如"帮我买鸡蛋去公园"→ 购物需求+出行需求）
3. confidence表示你对分类结果的确定程度(0-1)
4. 如果实在无法判断，标为"其他"并将confidence设为<0.5
"""

# 标签集描述模板（动态生成）
TAG_DESCRIPTION_TEMPLATE = """
- {name}：{description}（举例：{examples}）
"""
```

#### 4.4.2 标签处理逻辑

```python
async def process_turn_tags(
    elder_id: str,
    session_id: str,
    turn_elder_message: Message,
    turn_xiaoban_reply: Message
) -> list[MessageTag]:
    """
    对一轮对话进行意图标签分类。
    """
    # 1. 获取对话上下文（前3轮）
    context_messages = await get_context(
        session_id=session_id,
        before_sequence=turn_elder_message.sequence,
        limit=6  # 3轮 = 6条消息
    )

    # 2. 获取活跃标签集
    active_tags = await db.query(
        "SELECT id, name, description FROM intent_tags WHERE is_active = true ORDER BY sort_order"
    )
    tag_names = [t.name for t in active_tags]
    tag_descriptions = format_tag_descriptions(active_tags)

    # 3. 构造Prompt并调用LLM
    prompt = INTENT_CLASSIFICATION_PROMPT.format(
        available_tags=tag_descriptions,
        context_messages=format_context(context_messages),
        elder_message=turn_elder_message.content,
        xiaoban_reply=turn_xiaoban_reply.content
    )

    try:
        result = await call_llm(
            prompt=prompt,
            response_format='json',
            temperature=0.1,  # 低温度，确保稳定分类
            max_tokens=200
        )
        parsed = json.loads(result)
    except (json.JSONDecodeError, LLMError) as e:
        # LLM输出异常：fallback到"其他"
        logger.warning(f"LLM分类输出异常: {e}")
        parsed = {"tags": ["其他"], "confidence": 0.2, "reasoning": "LLM输出解析失败"}

    # 4. 校验标签有效性（必须在活跃标签集中）
    valid_tags = [t for t in parsed.get('tags', []) if t in tag_names]
    if not valid_tags:
        valid_tags = ['其他']
        parsed['confidence'] = min(parsed.get('confidence', 0.3), 0.3)

    confidence = parsed.get('confidence', 0.5)

    # 5. 写入 message_tags 表
    created_tags = []
    for tag_name in valid_tags:
        tag = next(t for t in active_tags if t.name == tag_name)
        message_tag = await create_message_tag(
            session_id=session_id,
            elder_id=elder_id,
            turn_start_message_id=turn_elder_message.id,
            tag_id=tag.id,
            confidence=confidence,
            source='ai',
            needs_review=(confidence < 0.6)
        )
        created_tags.append(message_tag)

    # 6. 如果是任务类标签，触发未满足需求检测
    task_tags = {'购物需求', '出行需求', '任务委托'}
    if any(t in task_tags for t in valid_tags):
        detect_unmet_need.delay({
            'elder_id': elder_id,
            'session_id': session_id,
            'turn_elder_message_id': turn_elder_message.id,
            'turn_xiaoban_reply_id': turn_xiaoban_reply.id,
            'tags': valid_tags,
            'confidence': confidence
        })

    return created_tags
```

#### 4.4.3 人工修正处理

```python
async def correct_tags(
    turn_message_id: str,
    corrected_tags: list[str],
    corrected_by: str,
    reason: str | None
) -> CorrectionResult:
    """
    人工修正标签：
    1. 软删除原有AI标签（标记source='ai_overridden'）
    2. 创建新标签记录（source='manual'）
    3. 写入修正日志（tag_corrections表）
    """
    # 1. 获取当前标签
    current_tags = await db.query(
        "SELECT id, tag_id FROM message_tags "
        "WHERE turn_start_message_id = :msg_id AND source != 'ai_overridden'",
        msg_id=turn_message_id
    )

    # 2. 标记原标签为已覆盖
    for tag in current_tags:
        await db.execute(
            "UPDATE message_tags SET source = 'ai_overridden', updated_at = NOW() "
            "WHERE id = :id",
            id=tag.id
        )

    # 3. 创建新的人工标签
    new_tag_records = []
    for tag_name in corrected_tags:
        tag_def = await db.query_one(
            "SELECT id FROM intent_tags WHERE name = :name", name=tag_name
        )
        new_record = await create_message_tag(
            session_id=current_tags[0].session_id if current_tags else None,
            elder_id=current_tags[0].elder_id if current_tags else None,
            turn_start_message_id=turn_message_id,
            tag_id=tag_def.id,
            confidence=1.0,  # 人工标注置信度为1
            source='manual',
            needs_review=False
        )
        new_tag_records.append(new_record)

    # 4. 写入修正日志
    for original_tag in current_tags:
        for new_tag in new_tag_records:
            await db.execute(
                "INSERT INTO tag_corrections "
                "(id, message_tag_id, original_tag_id, corrected_tag_id, corrected_by, reason) "
                "VALUES (gen_random_uuid(), :msg_tag_id, :orig_id, :new_id, :by, :reason)",
                msg_tag_id=original_tag.id,
                orig_id=original_tag.tag_id,
                new_id=new_tag.tag_id,
                by=corrected_by,
                reason=reason
            )

    return CorrectionResult(
        original_tags=current_tags,
        new_tags=new_tag_records,
        correction_id=new_tag_records[0].id if new_tag_records else None
    )
```

### 4.5 错误处理

| 故障场景 | 处理策略 | 后续动作 |
|---------|---------|---------|
| LLM API调用失败 | 重试3次（指数退避1s/2s/4s） | 仍失败 → 标记该轮为 `tag_status='failed'`，进入hourly批量重试队列 |
| LLM返回非JSON | 尝试正则提取标签名；失败则fallback到"其他"，confidence=0.2 | 标记 `needs_review=true` |
| LLM返回不在标签集中的标签 | 过滤无效标签，保留有效标签；全无效则fallback到"其他" | 标记 `needs_review=true` |
| 队列积压（>1000未处理） | 监控告警技术团队 | 动态增加 Celery Worker 并发数（环境变量控制） |
| 标签集为空（所有标签被停用） | 拒绝处理，日志ERROR | 告警PM "标签集为空，无法执行分类" |
| 熔断器开启（LLM连续失败5次） | 停止消费30秒，半开探测 | 任务在队列中等待，不丢失 |

## 五、F7: 未满足需求识别管线（Unmet Needs Identification）

### 5.1 架构设计

F7 是一条依赖F4标签输出的下游管线。当F4检测到任务类标签（购物需求/出行需求/任务委托）时，将该轮对话推送到 `tasks.unmet.detect` 任务队列，由 Celery Worker 判断小伴是否成功执行了该请求。

**两层检测架构**（快速路径 + LLM精确判断）：

```
tasks.unmet.detect 任务队列
       │
       ▼
┌──────────────────────────────────────┐
│ Celery Worker                          │
│ (detect_unmet_need task)               │
│                                       │
│ Layer 1: 规则匹配（快速路径）           │
│ ├─ 检查小伴回复是否含“成功指标”        │──→ 含成功指标 → 跳过（已满足）
│ ├─ 检查小伴回复是否含“失败指标”        │──→ 含失败指标 → 高置信度未满足
│ └─ 边界情况                           │──→ 进入Layer 2
│                                       │
│ Layer 2: LLM精确判断                   │
│ ├─ 是真正请求 vs 随口感叹?             │
│ ├─ 小伴是否真的没执行?                 │
│ └─ 提取需求描述 + 分类                 │
│                                       │
│ 去重检查                              │
│ ├─ 同一用户同类需求是否已存在?          │
│ └─ 存在 → 添加实例引用（不增加频次）    │
│                                       │
│ 写入 unmet_needs 表                   │
└──────────────────────────────────────┘
```

### 5.2 数据流

```
F4 Tagging Pipeline 检测到任务类标签 (购物需求/出行需求/任务委托)
  → 发送 tasks.unmet.detect 任务
    → Celery Worker 执行 detect_unmet_need:
      → 获取完整对话轮 (elder message + xiaoban reply)
      → 第一层: 规则匹配小伴回复
        → 匹配成功指标 (如“帮您查到”“已经设好了”) → 已满足，任务返回
        → 匹配失败指标 (如“帮不了”“暂时做不到”) → 高置信度未满足
        → 无明确指标 → 进入第二层
      → 第二层: LLM判断
        → 输入: 老人消息 + 小伴回复
        → 输出: is_unmet + is_real_request + category + description
      → 如果确认为未满足需求:
        → 去重检查 (同一用户同一category是否已存在confirmed记录)
          → 已存在: 添加实例引用到 unmet_need_instances 表
          → 不存在: 写入 unmet_needs 表 (status='confirmed')
      → 任务完成
```

### 5.4 核心算法/逻辑

#### 5.4.1 规则匹配检测（Layer 1 — 快速路径）

```python
# 失败指标词库：小伴"做不到"的典型回复模式
FAILURE_INDICATORS = [
    "帮不了", "暂时还不能", "这个我还做不到", "暂时没法",
    "以后可以", "等我学会了", "目前还不支持",
    "您可以自己", "建议您直接",  # 转移/回避模式
    "暂时帮不了", "还做不到这个", "这个功能还没有",
]

# 成功指标词库：小伴成功执行的典型回复模式
SUCCESS_INDICATORS = [
    r"好的.*帮您", r"已经.*设置", r"明天.*天气", r"提醒.*设好了",
    r"帮您查到", r"已经.*下单", r"好的.*记下了.*提醒",
    r"设好了", r"查到了",
]

def rule_based_detection(xiaoban_reply: str, tags: list[str]) -> str:
    """
    规则匹配快速路径。
    返回: 'fulfilled' | 'unmet' | 'uncertain'
    """
    # 前置检查：是否为任务类对话
    task_tags = {'购物需求', '出行需求', '任务委托'}
    if not any(t in task_tags for t in tags):
        return 'fulfilled'  # 非任务类，直接跳过

    # 检查成功指标（优先级高于失败指标）
    for pattern in SUCCESS_INDICATORS:
        if re.search(pattern, xiaoban_reply):
            return 'fulfilled'

    # 检查失败指标
    for indicator in FAILURE_INDICATORS:
        if indicator in xiaoban_reply:
            return 'unmet'

    # 无明确指标 → 交给LLM
    return 'uncertain'
```

#### 5.4.2 LLM精确判断（Layer 2）

```python
UNMET_NEED_PROMPT = """
分析以下对话，判断是否存在"用户提出了请求但小伴没能帮上忙"的情况。

老人说：{elder_message}
小伴回复：{xiaoban_reply}

请判断：
1. 老人是否表达了明确的任务请求？
   - "帮我买鸡蛋" = 明确请求 ✓
   - "要是能帮我买菜就好了" = 假设性感叹，不算 ✗
   - "最近买菜好贵啊" = 陈述/闲聊，不算 ✗
2. 小伴是否成功执行了该请求？

请输出JSON格式：
{{
    "is_unmet": true/false,
    "is_real_request": true/false,
    "need_description": "用一句话描述用户的需求",
    "category": "购物/打车/挂号/缴费/家政/其他",
    "confidence": 0.9,
    "reasoning": "判断理由（20字以内）"
}}

如果不存在未满足需求：
{{
    "is_unmet": false,
    "reason": "简述原因"
}}
"""

async def llm_detect_unmet(elder_message: str, xiaoban_reply: str) -> UnmetDetectionResult:
    """
    使用LLM精确判断是否存在未满足需求。
    """
    prompt = UNMET_NEED_PROMPT.format(
        elder_message=elder_message,
        xiaoban_reply=xiaoban_reply
    )

    try:
        result = await call_llm(
            prompt=prompt,
            response_format='json',
            temperature=0.1,
            max_tokens=200
        )
        parsed = json.loads(result)
    except Exception as e:
        logger.warning(f"未满足需求LLM检测失败: {e}")
        return UnmetDetectionResult(is_unmet=False, needs_manual_review=True)

    if not parsed.get('is_unmet', False):
        return UnmetDetectionResult(is_unmet=False)

    if not parsed.get('is_real_request', False):
        # 随口感叹，不计入
        return UnmetDetectionResult(is_unmet=False, reason="非明确请求")

    return UnmetDetectionResult(
        is_unmet=True,
        need_description=parsed.get('need_description', ''),
        category=parsed.get('category', '其他'),
        confidence=parsed.get('confidence', 0.5)
    )
```

#### 5.4.3 去重逻辑

```python
async def check_and_store_unmet_need(
    elder_id: str,
    session_id: str,
    message_id: str,
    need_description: str,
    category: str,
    confidence: float
) -> bool:
    """
    去重检查并存储未满足需求。
    规则：同一用户同一类别只计1个独立需求（frequency不虚增）。
    重复出现时，添加对话实例引用但不增加频次。
    
    返回: True=新需求已创建, False=已去重（添加了实例引用）
    """
    # 查找同一用户同一分类的已确认需求
    existing = await db.query_one(
        "SELECT id FROM unmet_needs "
        "WHERE elder_id = :elder_id AND category = :category AND status = 'confirmed' "
        "ORDER BY created_at DESC LIMIT 1",
        elder_id=elder_id, category=category
    )

    if existing:
        # 已有同类需求 → 不新建，只添加实例引用
        await db.execute(
            "INSERT INTO unmet_need_instances (id, unmet_need_id, session_id, message_id, created_at) "
            "VALUES (gen_random_uuid(), :need_id, :session_id, :message_id, NOW())",
            need_id=existing.id, session_id=session_id, message_id=message_id
        )
        logger.info(f"去重: 用户{elder_id}已有'{category}'类需求，添加实例引用")
        return False

    # 新需求 → 创建记录
    await db.execute(
        "INSERT INTO unmet_needs (id, elder_id, session_id, message_id, need_description, category, confidence, status) "
        "VALUES (gen_random_uuid(), :elder_id, :session_id, :message_id, :desc, :category, :confidence, 'confirmed')",
        elder_id=elder_id, session_id=session_id, message_id=message_id,
        desc=need_description, category=category, confidence=confidence
    )
    return True
```

#### 5.4.4 聚合统计SQL

```sql
-- 按类别聚合未满足需求（支持时间范围筛选）
SELECT
    un.category,
    COUNT(DISTINCT un.elder_id) AS frequency,      -- 独立用户数
    COUNT(*) + COALESCE(inst_count.cnt, 0) AS instance_count,  -- 总实例数
    MAX(un.created_at) AS latest_at,
    -- 获取最新一条需求描述作为示例
    (SELECT un2.need_description
     FROM unmet_needs un2
     WHERE un2.category = un.category AND un2.status = 'confirmed'
     ORDER BY un2.created_at DESC LIMIT 1
    ) AS latest_example
FROM unmet_needs un
LEFT JOIN (
    SELECT unmet_need_id, COUNT(*) AS cnt
    FROM unmet_need_instances
    GROUP BY unmet_need_id
) inst_count ON inst_count.unmet_need_id = un.id
WHERE un.status = 'confirmed'
  AND un.created_at > NOW() - INTERVAL :time_range
GROUP BY un.category
ORDER BY frequency DESC;

-- 趋势计算（当前周期 vs 上一周期）
WITH current_period AS (
    SELECT category, COUNT(DISTINCT elder_id) AS freq
    FROM unmet_needs
    WHERE status = 'confirmed' AND created_at > NOW() - INTERVAL '7 days'
    GROUP BY category
),
previous_period AS (
    SELECT category, COUNT(DISTINCT elder_id) AS freq
    FROM unmet_needs
    WHERE status = 'confirmed'
      AND created_at BETWEEN NOW() - INTERVAL '14 days' AND NOW() - INTERVAL '7 days'
    GROUP BY category
)
SELECT
    c.category,
    c.freq AS current_freq,
    COALESCE(p.freq, 0) AS previous_freq,
    CASE
        WHEN COALESCE(p.freq, 0) = 0 THEN 'up'
        WHEN c.freq > p.freq * 1.2 THEN 'up'
        WHEN c.freq < p.freq * 0.8 THEN 'down'
        ELSE 'stable'
    END AS trend
FROM current_period c
LEFT JOIN previous_period p ON c.category = p.category;
```

### 5.5 错误处理

| 故障场景 | 处理策略 | 后续动作 |
|---------|---------|---------|
| LLM分类失败 | 标记该条为 `needs_manual_review=true`，不自动分类 | 进入人工审核队列 |
| 误报率过高（>30%） | 监控 `dismissed` 比率 | 如某类别dismissed率>30%，企微通知PM审查检测规则 |
| F4管线积压 | F7自然等待（依赖F4输出） | 级联延迟可接受，不需要额外处理 |
| 去重查询超时 | 按“新需求”处理，允许短期重复 | 下次 dead_letter 清理时合并重复 |
| 类别超出预定义 | LLM可自由输出category字段 | 新类别自动创建，无需预定义 |
