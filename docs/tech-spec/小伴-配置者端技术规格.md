## 一、概述

### 1.1 文档范围

本技术规格覆盖配置者端三大功能模块：

| 模块 | 名称 | 核心能力 |
|------|------|---------|
| F1 | 初始化配置系统 | 自由文本输入 → NLP解析 → 结构化理解文档生成 |
| F2 | 给小伴留话系统 | 留话提交 → 内容审核 → LLM自然化转化 → 择机送达 |
| F3 | 紧急通知系统 | 触发信号消费 → 模板渲染 → 多通道推送 → 降级重试 |

### 1.2 产品形态

- **前端**：微信小程序（Taro 3.6+ 框架，React语法）
- **后端**：FastAPI服务（Python 3.11+），共享PostgreSQL 15+ 数据库
- **异步处理**：Celery + Redis 异步任务队列
- **LLM调用**：通义千问 Qwen-Max（主）/ DeepSeek-V3（备）

### 1.3 共享基础设施引用

本规格中涉及的数据库Schema、消息队列、API规范、错误码、认证方式等共享基础设施，均遵循《小伴-基础架构技术规格》中的定义。以下为直接引用的核心表：

| 表名 | 用途 | 定义位置 |
|------|------|---------|
| `elders` | 老人用户主表 | 基础架构规格 §3.1 |
| `elder_profiles` | 老人理解文档（Markdown，版本化） | 基础架构规格 §3.1 |
| `configurators` | 配置者账户信息 | 基础架构规格 §3.1 |
| `configurator_messages` | 配置者留话队列 | 基础架构规格 §3.8 |
| `emergency_alerts` | 紧急通知记录 | 基础架构规格 §3.9 |

### 1.4 认证方式

配置者端采用微信OAuth登录 → Session Token机制：

```
wx.login() → 获取临时code
  → POST /api/configurator/auth/login { code }
  → 后端调用微信接口换取 openid
  → 查找或创建 configurators 记录
  → 生成 JWT session_token（有效期7天）
  → 返回 { session_token, configurator_info }
  → 后续请求: Authorization: Bearer {session_token}
```

**核心职责**：
- 前端：提供零压力的自由文本输入界面，引导但不强制
- 后端NLP解析：调用LLM将自然语言描述解析为结构化理解文档（Markdown格式）
- 存储：elder_profiles表存储理解文档，支持版本追踪和多人贡献合并

### 2.2 数据流

```
配置者打开小程序
  → 微信OAuth登录 → 获取session_token（JWT，7天有效期）
  → 检查是否已有关联elder
    → 有: 展示当前理解文档摘要 + "补充信息"入口
    → 无: 展示引导语 + 空白输入区域
  → 配置者选择与老人的关系（子女/社工/邻居/老伴/本人）
  → 配置者在自由文本区域书写（任意长度/格式/维度）
  → 点击"让小伴认识TA"
  → POST /api/configurator/profile
    → Step 1: 输入验证
      → 非空检查（EC1）
      → 实质内容检查（EC2，排除纯问候语）
      → 隐私数据过滤（EC5，身份证号/银行卡号正则匹配）
    → Step 2: 记录贡献者信息
      → 关联 configurator_id + relationship 到本次贡献
    → Step 3: 调用LLM解析自由文本
      → 输入: 原始文本 + 贡献者关系 + 现有理解文档（如有）
      → 输出: 结构化理解文档（Markdown）+ 确认摘要
    → Step 4: 存储理解文档
      → 首次: INSERT elder_profiles (version=1)
      → 追加: INSERT elder_profiles (version=N+1)，合并新旧内容
    → Step 5: 返回确认摘要
  → 配置者看到确认: "我了解到: 张美兰阿姨, 68岁, 住上海普陀..."
  → 配置者选择 "确认无误" 或 "修改"
    → 修改: 返回编辑界面，可修改后重新提交
    → 确认: 展示成功状态 "小伴已经开始陪伴了"
```

### 2.3 NLP解析算法

```python
# LLM 模型配置（对齐技术选型决策文档 §六）
LLM_PRIMARY_MODEL = "qwen-max"           # 主力模型：通义千问 Qwen-Max
LLM_FALLBACK_MODEL = "deepseek-v3"       # 备用模型：DeepSeek-V3


def parse_profile_text(input_text: str, contributor_relationship: str, existing_profile: str | None = None) -> tuple[str, str]:
    """
    将配置者的自由文本解析为结构化理解文档

    Args:
        input_text: 配置者原始输入文本
        contributor_relationship: 配置者与老人关系（子女/社工/邻居/老伴/本人）
        existing_profile: 已有的理解文档内容（用于合并场景）

    Returns:
        (understanding_doc, summary): 结构化理解文档 + 确认摘要
    """
    prompt = f"""
    你是小伴的初始化系统。一位{contributor_relationship}向你介绍了一位老人。
    请将以下自然语言描述解析为结构化的"理解文档"。

    规则：
    1. 按以下维度组织（有信息才填，没有的维度不要猜测）：
       - 基本信息（姓名、年龄、地址、称呼方式）
       - 日常生活（作息、活动、兴趣爱好）
       - 家人关系（家庭成员、关系状态）
       - 健康状况（已知疾病、用药、身体状况）
       - 性格特点（个性、社交偏好）
       - 敏感区域（不要提的话题、忌讳）
       - 其他备注
    2. 用自然语言描述，不要用表格或字段形式
    3. 保留原始信息的语气和情感色彩
    4. 如果有现有理解文档，将新信息合并进去：
       - 新信息补充到对应维度
       - 矛盾处标注"需确认"（如"此前说住浦东，新信息说住普陀，需确认"）
       - 不删除旧信息，仅追加或标注
    5. 信息优先级: 对话透露 > 最新配置者提交 > 早期提交

    {'现有理解文档：\n' + existing_profile if existing_profile else '（首次创建）'}

    配置者输入：
    {input_text}

    请输出Markdown格式的理解文档：
    """

    understanding_doc = call_llm(
        prompt=prompt,
        model="qwen-max",       # 主力模型（通义千问 Qwen-Max）
        temperature=0.3,         # 低温度保证解析准确性
        max_tokens=2000,
        timeout=15               # 超时15秒
    )

    # 生成确认摘要（用于前端展示，让配置者确认关键信息）
    summary_prompt = f"""
    用1-2句话概括以下理解文档的核心信息，让配置者确认是否正确。
    格式："我了解到：[关键信息摘要]"
    要求：只提关键事实（姓名、年龄、地点、最突出的特点），不要太长。

    理解文档：
    {understanding_doc}
    """
    summary = call_llm(
        prompt=summary_prompt,
        model="qwen-max",      # 主力模型（通义千问 Qwen-Max）
        temperature=0.2,
        max_tokens=200,
        timeout=5
    )

    return understanding_doc, summary
```

### 2.5 输入验证规则

```python
import re

# 隐私数据正则模式
SENSITIVE_PATTERNS = [
    r'\d{17}[\dXx]',                    # 身份证号（18位）
    r'\d{15}',                           # 旧版身份证号（15位）
    r'\d{16,19}',                        # 银行卡号（16-19位）
    r'\d{3}-?\d{4}-?\d{4}',             # 电话号码格式（非手机号）
]

# 纯问候语/无实质内容模式
GREETING_PATTERNS = [
    r'^(你好|您好|hi|hello|嗨|哈喽|在吗)\s*[!！。.？?]*$',
    r'^(好的|ok|行|嗯|哦)\s*[!！。.？?]*$',
]

def validate_input(text: str) -> tuple[str, str | None]:
    """
    验证配置者输入内容

    Returns:
        (validated_text, error_message):
        - 验证通过: (cleaned_text, None)
        - 验证失败: (None, error_message)
    """
    # EC1: 空白提交
    if not text or text.strip() == "":
        return None, "至少写一句话，让小伴认识一下这位老人"

    cleaned = text.strip()

    # EC2: 无实质内容（纯问候语或过短）
    if len(cleaned) < 5:
        return None, "再多说一点？哪怕一个名字、一个爱好，都能帮小伴更好地开始"

    for pattern in GREETING_PATTERNS:
        if re.match(pattern, cleaned, re.IGNORECASE):
            return None, "再多说一点？哪怕一个名字、一个爱好，都能帮小伴更好地开始"

    # EC5: 结构化隐私数据过滤
    has_sensitive = False
    for pattern in SENSITIVE_PATTERNS:
        matches = re.findall(pattern, cleaned)
        if matches:
            has_sensitive = True
            for match in matches:
                cleaned = cleaned.replace(match, "[已过滤]")

    if has_sensitive:
        # 返回过滤后的文本 + 警告（非阻断）
        # 前端额外展示提示: "这类信息不需要告诉小伴，已自动过滤"
        pass

    # EC4: 行为指令型输入 → 不阻断，但标注供LLM解析时处理
    # 由 parse_profile_text 中的 prompt 规则处理
    # "每天提醒他吃药" → 解析为信息 "配置者希望关注用药情况"

    # EC6: 超长内容（不限制上限，正常接收）
    # 无需处理

    return cleaned, None
```

### 2.8 错误处理

| 错误场景 | 处理策略 | 用户感知 |
|---------|---------|---------|
| LLM解析超时（>15秒） | 重试1次（备用LLM供应商）；仍失败则存储原始文本，标记 `last_updated_by='pending_parse'`，异步重新解析并通知团队 | 展示"已收到，小伴正在消化中"，不阻断流程 |
| 理解文档版本冲突（并发写入） | 使用乐观锁：检测version冲突后，获取最新版本重新合并提交，最多重试3次 | 用户无感知（后台自动处理） |
| 网络中断（提交失败） | 前端本地存储草稿（localStorage），网络恢复后自动重试提交 | 提示"网络不太好，已保存草稿，稍后自动重试" |
| LLM返回格式异常 | 校验Markdown结构，异常则使用原始文本作为理解文档内容（降级存储） | 用户无感知，但确认摘要可能较简略 |
| elder_id为null且系统创建elder失败 | 数据库事务回滚，返回5001错误 | 提示"系统暂时出了点问题，请稍后再试" |
| 隐私数据过滤后文本过短 | 过滤后重新执行EC2检查 | 提示"过滤后信息太少，请补充其他信息" |

### 3.2 数据流

```
配置者提交留话
  → POST /api/configurator/message
    → Step 1: 内容审核
      → 监控意图检测 (正则 + LLM双重检查)
      → 行为控制意图检测

      → SELECT COUNT(*) FROM configurator_messages

    → Step 3: 写入数据库
      → INSERT INTO configurator_messages (
          configurator_id, elder_id, original_content, urgency,
          status='pending',
          scheduled_deliver_by = NOW() + interval(urgency)
        )
      → urgency='normal': scheduled_deliver_by = NOW() + 24h
      → urgency='timely': scheduled_deliver_by = NOW() + 4h
    → Step 4: 推入异步转化任务
      → tasks.configurator.transform.delay(message_id)
    → 返回确认给配置者

老人与小伴对话中
  → 对话引擎检查: SELECT * FROM configurator_messages
```

### 3.3 留话内容审核算法

```python
import re

# 监控意图匹配模式
SURVEILLANCE_PATTERNS = [
    r"问[她他].*去了哪",
    r"看看[她他].*有没有",
    r"[她他]今天做了什么",
    r"[她他]最近.*怎么样",
    r"[她他]有没有按时",
    r"跟踪",
    r"监控",
    r"问问.*在干[什嘛]",
    r"[她他].*是不是.*了",
    r"看[她他].*情况",
]

# 行为控制意图匹配模式
CONTROL_PATTERNS = [
    r"以后每天.*提醒",
    r"每天[问让]",
    r"不[要许].*跟",
    r"设置.*规则",
    r"修改.*方式",
    r"让小伴.*不要",
    r"命令",
    r"强制",
    r"必须.*每天",
]

def validate_message_content(content: str) -> tuple[bool, str | None]:
    """
    判断留话内容是否合规（非监控型、非指令型）

    Returns:
        (approved, rejection_reason): 
        - 通过: (True, None)
        - 拒绝: (False, rejection_message)
    """
    # EC6: 空内容或无实质内容
    if not content or len(content.strip()) < 2:
        return False, "写点什么让小伴转达给爸/妈吧"

    # Rule 1: 监控意图检测（正则快速匹配）
    for pattern in SURVEILLANCE_PATTERNS:
        if re.search(pattern, content):
            return False, "小伴是陪伴者，不是监控工具。如果您担心爸/妈的身体，可以直接打电话关心哦。"

    # Rule 2: 行为控制意图检测（正则快速匹配）
    for pattern in CONTROL_PATTERNS:
        if re.search(pattern, content):
            return False, "这个功能用于给老人传达您的关心，不能设置小伴的行为规则。"

    # Rule 3: LLM二次校验（处理正则难以覆盖的隐晦表达）
    if needs_llm_check(content):
        llm_verdict = llm_check_message_intent(content)
        if llm_verdict['intent'] == 'surveillance':
            return False, "小伴是陪伴者，不是监控工具。如果您担心爸/妈的身体，可以直接打电话关心哦。"
        elif llm_verdict['intent'] == 'control':
            return False, "这个功能用于给老人传达您的关心，不能设置小伴的行为规则。"

    return True, None


def needs_llm_check(content: str) -> bool:
    """
    判断是否需要LLM二次校验
    当正则未匹配但内容包含可疑关键词时触发
    """
    AMBIGUOUS_KEYWORDS = ["问", "看", "检查", "确认", "提醒", "告诉", "让"]
    return any(kw in content for kw in AMBIGUOUS_KEYWORDS)


def llm_check_message_intent(content: str) -> dict:
    """
    使用LLM判断留话意图

    Returns: { "intent": "caring" | "surveillance" | "control", "confidence": 0.0-1.0 }
    """
    prompt = f"""
    判断以下留言的意图类型：
    - caring: 关心老人的正常留言（如"跟妈妈说我想她"）
    - surveillance: 监控意图（试图了解老人行为/状态）
    - control: 行为控制（试图给小伴下达行为指令）

    留言内容："{content}"

    请返回JSON: {{"intent": "...", "confidence": 0.0-1.0}}
    """
    return call_llm_json(prompt, temperature=0.1)
```

### 3.4 留话转化算法（配置者原话 → 自然对话）

```python
def transform_message(message_id: str) -> str:
    """
    将配置者留言转化为小伴在对话中自然提及的表达
    由 Celery task tasks.configurator.transform 调用
    """
    message = get_configurator_message(message_id)
    elder_profile = get_latest_elder_profile(message.elder_id)
    configurator = get_configurator(message.configurator_id)

    prompt = f"""
    你是小伴，需要在和老人的对话中自然地传达一条来自{configurator.relationship}的消息。

    规则：
    1. 不能逐字转发原文，必须用自然对话的方式表达
    2. 表达方式应该像"顺便提起"，不像"宣读通知"
    3. 称呼要符合老人的习惯（参考理解文档中的家人称呼）
    4. 控制在1-2句话以内
    5. 语气温暖自然，像朋友聊天时随口说的

    老人信息：{elder_profile.nickname if elder_profile else '老人'}
    {configurator.relationship}名字叫{configurator.nickname}
    配置者原话："{message.original_content}"

    请输出小伴的自然表达版本（仅输出转化后的话，不要其他内容）：
    """

    transformed = call_llm(
        prompt=prompt,
        model="qwen-max",          # 主力模型（通义千问 Qwen-Max）
        temperature=0.7,            # 适度温度保证表达自然多样
        max_tokens=200,
        timeout=10
    )

    # 更新数据库
    update_message_transformed(message_id, transformed)

    return transformed

# 转化示例:
# "跟我妈说我下周回来"        → "哎对了阿姨，小明说下周要回来看您呢。"

# "让妈妈明天记得带伞，有雨"    → "阿姨，明天有雨，出门记得带伞——小明也说怕您淋着。"
```

### 3.5 送达时机判断

```python
def should_deliver_now(elder_id: str, message) -> bool:
    """
    判断当前对话是否适合传达留话
    由老人端 Dialogue Engine 在对话处理过程中调用
    """
    # 1. 检查截止时间紧迫性
    if message.urgency == 'timely' and approaching_deadline(message, buffer_minutes=30):
        return True  # 紧迫消息强制送达，即使时机不完美

    # 2. 深度情感对话中不送达
    current_emotion = get_current_conversation_emotion(elder_id)
    if current_emotion < -0.5:  # 老人正处于悲伤/不安状态
        return False  # 等待更好的时机

    # 3. 对话至少已进行2轮才送达（确保已暖场）
    turn_count = get_current_session_turns(elder_id)
    if turn_count < 2:
        return False

    # 4. 老人正在进行具体任务请求时不打断
    current_intent = get_last_turn_intent(elder_id)
    if current_intent in ['任务委托', '购物需求', '出行需求', '信息查询']:
        return False  # 让老人先完成任务

    # 5. 话题相关性检查（最佳时机）
    current_topic = get_current_conversation_topic(elder_id)
    if topic_relevance(current_topic, message.transformed_content) > 0.5:
        return True  # 话题相关，自然过渡的好时机

    # 6. 对话自然停顿点
    if is_conversation_pause_point(elder_id):
        return True  # 如老人说"嗯嗯"、"好的"等过渡性回复

    return False


def approaching_deadline(message, buffer_minutes: int = 30) -> bool:
    """检查是否接近送达截止时间"""
    remaining = message.scheduled_deliver_by - datetime.now(tz=timezone.utc)
    return remaining.total_seconds() < buffer_minutes * 60
```

### 3.8 错误处理

| 错误场景 | 处理策略 | 用户感知 |
|---------|---------|---------|
| LLM转化失败 | 保留原始内容作为降级fallback；对话中使用最小化转化（仅调整称呼和语气） | 用户无感知 |
| 送达截止时间超时（24h/4h） | 定时任务检测超时留言 → 强制标记为优先送达 → 下次对话无论时机直接送达 | 配置者可能感知到延迟 |
| 配置者确认通知推送失败 | 微信模板消息失败 → 重试3次（间隔30s）→ 全部失败则在下次打开小程序时展示"已转达"状态 | 延迟感知送达确认 |
| 老人长期未互动（>72h） | 留言保持transformed状态等待 → 向配置者推送提示"老人最近没有跟小伴互动，留言会在下次对话时传达" → 不暴露活跃数据 | 配置者了解情况但不获取隐私 |
| 并发提交超过队列上限 | 数据库层面使用 `SELECT FOR UPDATE` 锁检查队列数，确保不超过3条 | 第4条返回2003错误 |
| 深夜及时留话 | EC7处理: 延迟至次日上午9点传达 → 向配置者说明"考虑到时间，会在明天上午合适的时候转达" | 配置者立即收到延迟说明 |

```python
import time

def send_notification(alert_id: str):
    """
    发送紧急通知给配置者
    支持微信模板消息（主通道）→ SMS短信（降级通道）
    """
    alert = get_alert(alert_id)
    configurator = get_configurator(alert.configurator_id)
    template = NOTIFICATION_TEMPLATES.get(alert.trigger_type)

    # 渲染模板内容
    elder = get_elder(alert.elder_id)
    pronoun = "她" if guess_gender(elder.nickname) == "female" else "他"
    content = template["content"].format(
        elder_nickname=elder.nickname or "老人",
        pronoun=pronoun
    )

    # 尝试1-3: 微信模板消息
    for attempt in range(1, 4):
        try:
            result = wechat_send_template_message(
                openid=configurator.wechat_openid,
                template_id=template["wechat_template_id"],
                title=template["title"],
                content=content,
                url=""  # 不跳转任何页面，避免暴露信息入口
            )
            if result.success:
                update_alert(
                    alert.id,
                    status='sent',
                    delivery_channel='wechat',
                    sent_at=now(),
                    retry_count=attempt
                )
                log_notification_success(alert.id, 'wechat', attempt)
                return
        except Exception as e:
            log_notification_retry(alert.id, 'wechat', attempt, str(e))

        if attempt < 3:
            time.sleep(30)  # 30秒间隔重试

    # 降级: SMS短信
    if configurator.phone:
        sms_content = template["sms_content"].format(
            elder_nickname=elder.nickname or "老人",
            pronoun=pronoun
        )
        try:
            sms_result = send_sms(
                phone=configurator.phone,
                content=sms_content
            )
            if sms_result.success:
                update_alert(
                    alert.id,
                    status='sent',
                    delivery_channel='sms',
                    sent_at=now(),
                    retry_count=4
                )
                log_notification_success(alert.id, 'sms', 4)
                return
        except Exception as e:
            log_notification_retry(alert.id, 'sms', 4, str(e))

    # 全部失败
    update_alert(alert.id, status='failed', retry_count=4)
    notify_tech_team(f"Alert {alert.id} delivery failed after all retries (wechat x3 + sms x1)")
```

### 4.5 延迟通知调度（情绪关怀类）

```python
from datetime import datetime, timedelta, time as dt_time

# 复用 Hermes Agent 的 cron/scheduler.py（77KB 生产级调度器）
# 延迟通知通过 Celery task 的 eta 参数实现延迟投递


def schedule_emotional_concern(alert_id: str):
    """
    非紧急的情绪关怀通知，延迟到次日上午9点发送
    避免深夜或不合适的时间打扰配置者

    调度方式：通过 Celery task 的 eta 参数实现延迟投递，
    由 Hermes cron/scheduler.py 统一管理定时任务生命周期。
    """
    alert = get_alert(alert_id)

    # 检查是否已有该老人的待发送情绪关怀通知
    existing = find_pending_emotional_alert(alert.elder_id, alert.configurator_id)
    if existing:
        # 不堆积多条情绪关怀通知 — 一条就够
        update_alert(alert.id, status='suppressed')
        return

    # 计算次日上午9点
    now = datetime.now()
    tomorrow_9am = datetime.combine(
        now.date() + timedelta(days=1),
        dt_time(hour=9, minute=0)
    )

    # 如果当前时间在上午9点之前，则今天9点就发
    if now.hour < 9:
        tomorrow_9am = datetime.combine(now.date(), dt_time(hour=9, minute=0))

    # 更新通知状态
    update_alert(alert.id, status='scheduled', scheduled_send_at=tomorrow_9am)

    # 通过 Celery eta 延迟投递（替代 APScheduler）
    # Hermes cron/scheduler.py 负责服务启动时补发遗漏的 scheduled 通知
    tasks.alert.send.apply_async(
```

```python
# ===========================
# 架构级信息隔离设计
# ===========================
#
# 核心原则: 对话数据绝不流向配置者
# 这不是靠"小心编码"保障的，而是架构层面的物理隔离

# 1. 数据库层隔离
# ─────────────────
# emergency_alerts 表没有指向 messages 表内容的外键
# trigger_source_message_id 仅供内部追踪，API层绝不暴露

# GRANT SELECT, INSERT, UPDATE ON configurator_messages TO configurator_svc;
# GRANT SELECT, UPDATE ON emergency_alerts TO configurator_svc;
# GRANT SELECT ON elders TO configurator_svc;
# -- 注意: 没有 messages 表的任何权限
# -- 注意: 没有 sessions 表的读权限
# -- 注意: 没有 memory_entries 表的读权限


# 2. API层过滤 (最终安全网)
# ─────────────────
def filter_configurator_response(response_data: dict) -> dict:
    """
    所有配置者端API响应的最终安全过滤
    防止任何意外的对话内容泄露
    """
    FORBIDDEN_FIELDS = [
        'message_content',
        'conversation',
        'dialogue',
        'elder_said',
        'xiaoban_said',
        'session_content',
        'trigger_source_message_id',  # 内部追踪ID也不暴露
        'emotion_score',
        'emotion_detail',
        'conversation_summary',
    ]

    def recursive_filter(data):
        if isinstance(data, dict):
            for field in FORBIDDEN_FIELDS:
                if field in data:
                    del data[field]
                    log_security_warning(
                        f"SECURITY: Attempted to expose '{field}' to configurator. "
                        f"Data stripped. Investigate source."
                    )
            for key, value in data.items():
                recursive_filter(value)
        elif isinstance(data, list):
            for item in data:
                recursive_filter(item)

    recursive_filter(response_data)
    return response_data
```

### 4.8 错误处理

| 错误场景 | 处理策略 | 用户感知 |
|---------|---------|----------|
| 微信模板消息被拒（内容违规） | 降级使用通用模板文本（"请联系家人确认安全"）；同时通知团队修订模板 | 配置者收到通用提醒 |
| SMS网关故障 | 切换备用SMS供应商重试 | 可能延迟收到通知 |
| 定时通知遗漏（服务器重启） | 启动时扫描: `SELECT * FROM emergency_alerts WHERE status='scheduled' AND scheduled_send_at < NOW()`，立即补发 | 可能延迟但不丢失 |
| 重复通知防护 | 幂等键: `elder_id + trigger_type + 1小时时间窗口`。同一老人同类触发在1小时内不重复发送 | 不会收到重复通知 |
| 配置者OpenID失效 | 微信返回无效用户错误 → 标记该配置者需重新授权 → 降级SMS | SMS替代通知 |
| Guardian System误触发大量告警 | 熔断保护: 同一老人1小时内最多2条紧急通知；超出的告警记录但不推送 | 避免通知轰炸 |
| 网络完全不可用 | 告警写入本地日志 + 磁盘文件；网络恢复后批量补发 | 延迟收到通知 |
