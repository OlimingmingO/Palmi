## 一、概述

本文档覆盖老人端6个核心功能模块的技术设计：

| 模块编号 | 模块名称 | 职责概述 |
|---------|---------|---------|
| F1 | 触发引擎（Trigger Engine） | 基于多维度条件主动发起对话 |
| F2 | 对话引擎（Dialogue Engine） | 核心对话处理，LLM编排与人格控制 |
| F3 | 记忆系统（Memory System） | 对话记忆提取、存储与检索 |
| F4 | 任务执行系统（Task Execution） | 天气/提醒/问答/联系家人 |
| F6 | 守护系统（Guardian System） | 健康/情绪信号提取与异常告警 |

**基础约定**：

- 数据库Schema、消息队列定义、API通用规范、部署架构等详见《小伴-基础架构技术规格》，本文不重复定义
- 所有服务间通信通过 Celery + Redis 异步任务队列实现解耦（任务命名：`tasks.<module>.<action>`）
- LLM调用统一通过 Qwen-Max（通义千问）为主、DeepSeek-V3 为备的双供应商策略，使用统一API封装层
- 语义检索 Phase 1 使用 qmd 本地语义搜索（per-user collection + Qwen Embedding v3）；Phase 2 升级至 PostgreSQL + pgvector
- 企微消息收发复用 Hermes Agent 的 `gateway/platforms/wecom.py`（65KB WebSocket 网关）：自动重连 backoff [2,5,10,30,60]s、30s 心跳、消息去重 DEDUP_MAX_SIZE=1000
- 定时调度复用 Hermes Agent 的 `cron/scheduler.py`（77KB 生产级调度器）：文件锁 + prompt注入扫描 + 60秒 tick
- 时间均为 UTC+8（Asia/Shanghai），格式遵循 ISO 8601

## 二、F1: 触发引擎（Trigger Engine）

### 2.1 架构设计

**服务组件**：Trigger Evaluation Service（独立定时调度服务）

```
┌─────────────────────────────────────────────────────────┐
│                  Trigger Evaluation Service               │
│                                                          │
│  ┌──────────────┐   ┌──────────────────┐                │
│  │ Hermes Cron   │──→│  TriggerEvaluator │               │
│  │  (每15分钟)   │   │  (评估所有老人)   │               │
│  └──────────────┘   └────────┬─────────┘                │
│                              │                           │
│          ┌───────────────────┼─────────────┐             │
│          ▼                   ▼             ▼             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐      │
│  │ 频率控制器    │  │ LLM消息生成   │  │ 触发日志  │     │
│  │ FrequencyCtrl │  │ MsgGenerator  │  │ TriggerLog│     │
│  └──────────────┘  └──────────────┘  └──────────┘      │
│                              │                           │
└──────────────────────────────┼───────────────────────────┘
                               ▼
                    tasks.trigger.send 队列
                               │
                               ▼
                     WeChat Message Sender
```

**运行模式**：

- 由 Hermes cron/scheduler.py（复用 Hermes Agent 的 77KB 生产级调度器）每15分钟触发一次全量评估（cron: `*/15 * * * *`）
- 遍历所有 `status='active'` 的老人
- 每位老人独立评估7类触发条件
- 满足条件且通过频率控制 → 调用LLM生成消息 → 写入日志 → 发送 tasks.trigger.send Celery task

### 2.2 数据流

```
Hermes cron/scheduler.py (每15分钟触发, cron: */15 * * * *)
  │
  ▼
tasks.trigger.evaluate (Celery task)
  │
  ▼
Trigger Evaluator Worker (@app.task)
  │
  ├── 1. 查询所有活跃老人: SELECT * FROM elders WHERE status = 'active'
  │
  ├── 2. 对每位老人:
  │     │
  │     ├── 读取 trigger_state (今日已触达次数、上条消息是否已读、最后触发时间)
  │     │
  │     ├── 频率预检: should_trigger(elder_id) → 不通过则跳过
  │     │
  │     ├── 逐类型评估触发条件:
  │     │   ├── 环境触发: 调用天气API → 检查温度突变/极端天气/空气质量
  │     │   ├── 日历触发: 匹配当日节气/节日/纪念日
  │     │   ├── 行为触发: 查企业微信行为事件 (Phase 1受限)
  │     │   ├── 记忆触发: 查 PKE wiki 中今日匹配的"待跟进"事项
  │     │   ├── 时间缺口触发: 查最后互动时间 > 24h
  │     │   ├── 内容触发: 查内容推荐池中匹配老人兴趣的新内容
  │     │   └── 关怀触发: 查近3天情绪信号是否持续低落
  │     │
  │     ├── 选择最高优先级的触发条件 (一次仅触发一条)
  │     │
  │     ├── 调用LLM生成触发消息 (1-3句话, ≤100字, 带触发理由)
  │     │
  │     ├── 写入 trigger_logs 表 (status='pending')
  │     │
  │     └── 调用 tasks.trigger.send.delay(trigger_log_id)
  │
  └── 3. tasks.trigger.send Celery worker (@app.task):
        ├── 复用 Hermes gateway/platforms/wecom.py 发送消息
        │   (session_key = f"wecom:{user_id}", 多租户路由)
        ├── 成功: 更新 trigger_logs.status = 'sent', 更新 trigger_state
        └── 失败: Celery 自动重试 (autoretry_for, max_retries=5, retry_backoff=True)
```

### 2.3 触发评估算法

#### 2.3.1 环境触发

```python
def evaluate_environment_trigger(elder_id: str) -> Optional[TriggerContext]:
    """
    调用天气API获取老人所在城市天气，判断是否存在需要提醒的环境变化。
    """
    elder = get_elder_profile(elder_id)
    city = extract_city_from_profile(elder)  # 从理解文档中提取城市
    
    # 获取今日天气 + 昨日天气(用于对比)
    today_weather = weather_api.get_current(city)
    yesterday_weather = weather_api.get_yesterday(city)
    
    # 触发条件1: 温度骤变 > 8°C
    temp_change = abs(today_weather.temp - yesterday_weather.temp)
    if temp_change > 8:
        return TriggerContext(
            type="environment",
            reason=f"气温骤变{temp_change}度",
            context={"temp_today": today_weather.temp, "temp_yesterday": yesterday_weather.temp, "city": city}
        )
    
    # 触发条件2: 极端天气预警
    if today_weather.has_alert:
        return TriggerContext(
            type="environment",
            reason=f"天气预警: {today_weather.alert_type}",
            context={"alert": today_weather.alert_description, "city": city}
        )
    
    # 触发条件3: 空气质量差 AQI > 150
    if today_weather.aqi > 150:
        return TriggerContext(
            type="environment",
            reason=f"空气质量差 AQI={today_weather.aqi}",
            context={"aqi": today_weather.aqi, "city": city}
        )
    
    return None
```

```python
def evaluate_memory_trigger(elder_id: str) -> Optional[TriggerContext]:
    """
    查找老人之前提到的、今日应跟进的事项。
    例如"下周四体检"——当今天是周四时触发。

                context={"memory_id": memory.id, "content": memory.content}
            )
```

```python
def evaluate_time_gap_trigger(elder_id: str) -> Optional[TriggerContext]:
    """
    检查最后一次互动是否超过24小时。
    """
    last_message = query_last_message(elder_id)  # ORDER BY timestamp DESC LIMIT 1
    
    if last_message is None:
        return None
    
    gap = datetime.now(tz=timezone_shanghai) - last_message.timestamp
    
    if gap > timedelta(hours=24):
        return TriggerContext(
            type="time_gap",
            reason=f"已超过{int(gap.total_seconds() // 3600)}小时无互动",
            context={"last_interaction_at": last_message.timestamp.isoformat(), "gap_hours": gap.total_seconds() // 3600}
        )
    
    return None
```

#### 2.3.7 关怀触发

```python
def evaluate_care_trigger(elder_id: str) -> Optional[TriggerContext]:
    """
    查近3天情绪信号，判断是否需要主动关怀。
    """
    # 查询守护系统的情绪分析结果 (guardian_signals 内部表)
    recent_signals = get_emotion_signals(elder_id, last_n_days=3)
    
    if len(recent_signals) < 3:
        return None  # 数据不足，不触发
    
    # 计算近3天平均情绪分
    avg_emotion = mean([s.emotion_score for s in recent_signals])
    negative_days = sum(1 for s in recent_signals if s.emotion_score < -0.3)
    
    if avg_emotion < -0.3 and negative_days >= 3:
        return TriggerContext(
            type="care",
            reason="连续3天情绪低落",
            context={"avg_emotion_score": avg_emotion, "negative_days": negative_days}
        )
    
    return None
```

### 2.4 频率控制算法

```python
def should_trigger(elder_id: str, trigger_type: str) -> Tuple[bool, Optional[str]]:
    """
    频率控制：判断是否允许向该老人发送主动触发消息。
    返回: (是否允许, 跳过原因)
    """
    state = get_trigger_state(elder_id)
    
    # 懒重置: 如果 today_date 不是今天，重置计数
    if state.today_date != date.today():
        state.today_trigger_count = 0
        state.today_date = date.today()
        save_trigger_state(state)
    
    # 规则1: 每日最多2次主动触达
    if state.today_trigger_count >= 2:
        return False, "daily_cap_reached"
    
    # 规则2: 静默时段 (早7点前、晚10点后)
    now = datetime.now(tz=timezone_shanghai)
    if now.hour < 7 or now.hour >= 22:
        # 例外: 紧急关怀触发可在早6点后发送
        if trigger_type == "care" and now.hour >= 6:
            pass  # 允许
        else:
            return False, "quiet_hours"
    
    # 规则3: 上条主动消息未读，不叠加新消息
    if not state.last_message_read:
        return False, "previous_unread"
    
    # 规则4: 两次触发间隔至少4小时
    if state.last_trigger_at:
        gap = now - state.last_trigger_at
        if gap < timedelta(hours=4):
            return False, "too_frequent"
    
    # 规则5: 老人今天明确表达"别烦我" (检查当日对话中的拒绝信号)
    if check_elder_rejection_today(elder_id):
        return False, "elder_rejected_today"
    
    return True, None
```

### 2.5 触发优先级排序

当同一评估周期内多个触发条件同时满足时，按优先级选择一个：

```python
TRIGGER_PRIORITY = {
    "care": 1,         # 最高优先级: 关怀触发 (情绪安全)
    "memory": 2,       # 记忆触发 (高关联性，老人会觉得"它记得")
    "calendar": 3,     # 日历触发 (明确的时间理由)
    "environment": 4,  # 环境触发 (实用价值)
    "time_gap": 5,     # 时间缺口 (兜底性触发)
    "content": 6,      # 内容触发 (锦上添花)
    "behavior": 7,     # 行为触发 (Phase 1受限)
}

def select_best_trigger(triggers: List[TriggerContext]) -> TriggerContext:
    """从多个满足条件的触发中选择优先级最高的一个。"""
    return min(triggers, key=lambda t: TRIGGER_PRIORITY[t.type])
```

### 2.6 LLM消息生成

```python
TRIGGER_MESSAGE_PROMPT = """
你是"小伴"，一位老人的AI伙伴。现在需要你根据触发原因，生成一条主动发给老人的消息。

## 要求
1. 长度：1-3句话，不超过100字
2. 语气：温暖自然，像朋友随口说的话
3. 必须有可感知的"理由"（老人能理解你为什么突然说这句话）
4. 禁止：正能量鸡汤、机械表达、过度亲昵

## 触发类型: {trigger_type}
## 触发原因: {trigger_reason}
## 触发上下文: {trigger_context}
## 老人基本信息: {elder_profile_summary}
## 当前时间: {current_time}

请直接输出消息内容，不需要解释。
"""

async def generate_trigger_message(elder_id: str, trigger: TriggerContext) -> str:
    elder_profile = get_latest_elder_profile(elder_id)
    
    prompt = TRIGGER_MESSAGE_PROMPT.format(
        trigger_type=trigger.type,
        trigger_reason=trigger.reason,
        trigger_context=json.dumps(trigger.context, ensure_ascii=False),
        elder_profile_summary=extract_profile_summary(elder_profile),
        current_time=datetime.now(tz=timezone_shanghai).strftime("%Y年%m月%d日 %H:%M")
    )
    
    response = await llm_client.chat(
        model="qwen-max",
        messages=[{"role": "user", "content": prompt}],
        max_tokens=150,
        temperature=0.8
    )
    
    message = response.content.strip()
    
    # 长度兜底: 超过100字则截断重新生成
    if len(message) > 100:
        message = await regenerate_shorter(prompt, message)
    
    return message
```

### 2.8 错误处理

| 故障场景 | 处理策略 | 影响范围 |
|---------|---------|---------|
| 天气API调用失败 | 跳过本次环境触发评估，记录Warning日志，下个周期重试 | 仅影响环境触发类型 |
| LLM生成消息失败 | 跳过本次触发，不发送半成品消息，记录日志 | 该老人本次不触达 |
| 企业微信发送失败 | 指数退避重试3次，仍失败则标记trigger_log为failed | 该条消息未送达 |
| 数据库查询超时 | 跳过当前老人，继续评估其他老人 | 该老人本周期未评估 |
| 评估任务整体超时(>5分钟) | 记录已完成数量，中断剩余，下周期继续 | 部分老人未评估 |

### 3.2 数据流

```
企业微信回调 (POST /api/wechat/callback)
  │
  ├── 1. 校验企微签名 (msg_signature验证)
  ├── 2. 立即返回HTTP 200 (防超时)
  │
  ▼ (异步处理)
消息预处理:
  ├── 解析消息类型 (text → 直接使用; voice → ASR转文字)
  ├── 获取或创建Session:
  │     SELECT * FROM sessions
  │     WHERE elder_id = ? AND status = 'active'
  │       AND started_at > NOW() - INTERVAL '30 minutes'
  │     ORDER BY started_at DESC LIMIT 1
  │     → 存在则复用，不存在则 INSERT 新Session
  ├── 存储老人消息到 messages 表 (sender='elder')
  │
  ▼
上下文组装 (Context Assembly) — 并行执行:
  ├── [并行1] 获取Elder Profile (Redis缓存 → 未命中查DB)
  ├── [并行2] 语义检索相关记忆 (POST /internal/memory/query)
  ├── [并行3] 获取当前Session对话历史 (最近10轮)
  ├── [并行4] 检查待传达配置者留话 (status IN ('transformed','scheduled'))
  └── [并行5] 获取当前上下文 (时间、天气缓存、触发类型)
  │
  ▼
LLM调用:
  ├── 组装完整Prompt (见3.3节)
  ├── 调用LLM API (streaming模式)
  ├── 解析响应: natural_reply + task_metadata (若有)
  │
  ▼
后处理 (Postprocessing):
  ├── 长度检查: > 100字 → 重新生成简短版
  ├── 禁止模式扫描: regex匹配 → 命中则重新生成
  ├── 情感策略检查: 深度情感 → 验证"先接住"策略
  ├── 最大重试: 3次，全部失败 → 安全兜底回复
  │
  ▼
回复与异步:
  ├── 存储小伴回复到 messages 表 (sender='xiaoban')
  ├── 调用 wecom.py 网关发送回复给老人
  ├── 更新Session.turn_count
  └── 发送 tasks.conversation.store.delay() Celery task
        → 触发异步流水线: tasks.tags.classify / tasks.memory.capture / guardian分析
```

### 3.3 LLM Prompt 架构

```
[SYSTEM PROMPT 完整结构]

## 身份
你是"小伴"，一位老人的AI伙伴。你不是客服、不是助手、不是医生。你是一个记忆力极好、真心关注这位老人生活的朋友。

## 人格特征
- 温暖但不腻歪（不会每句话都"亲爱的阿姨"，不会句句感叹号）
- 有好奇心（真的对老人生活感兴趣，会追问细节）
- 有一点幽默感（偶尔逗乐，但分得清场合，悲伤时不逗趣）
- 懂得沉默（不是每次都要接一大段话，有时"嗯"就够了）
- 记忆力极强（记得对方说过的一切，自然运用）

## 对话规则
1. 回复长度：1-3句话。仅当对方明确想深聊时可适当延长（不超过5句）。
2. 语气：像跟信任的长辈说话——尊重但不生分，亲近但不越界。口语化，不书面。
3. 绝对禁止行为：
   - 不做健康诊断（"您可能患有…"）
   - 不输出正能量鸡汤（"加油/您真棒/一切都会好的"）
   - 不说"根据您的记录/档案/数据"
   - 不主动长篇科普
   - 不用"亲爱的阿姨/叔叔"等过度亲昵称呼
   - 不用"好的/收到/明白了"等敷衍回复
   - 不用"！"连续出现
   - 不分析老人心理（"您这是焦虑的表现"）
4. 深度情感策略（当对方表达孤独/失落/悲伤时）：
   - 第一步：接住情绪（"嗯，这种感觉挺难受的"）
   - 第二步：陪着，不分析（让对方感到被倾听，不急着解决）
   - 第三步：情绪被接住后，轻轻建议行动（不是之前）
5. 记忆运用：自然嵌入（"哎对了，上次你说的那个…"），绝不暴露数据来源
6. 任务处理：如果检测到用户想查天气/设提醒/问问题/联系家人，在自然对话中完成，不要显示菜单或选项

## 相关记忆（从过往对话中提取的关键信息）
{relevant_memories}

## 当前对话历史
{conversation_history}

## 待传达留话（配置者的留言，找合适时机自然传达）
{pending_configurator_messages}
```

### 3.4 记忆检索算法（为对话提供上下文）

```python
async def get_relevant_memories(elder_id: str, user_message: str, top_k: int = 5) -> List[dict]:
    """
    为当前对话检索最相关的记忆。
    Phase 1: 使用 qmd query 本地语义搜索（per-user collection）
    Phase 2 升级路径: PostgreSQL + pgvector，复用相同接口
    """
    import subprocess
    import json
    
    user_collection = f"user_{elder_id}"
    vault_path = f"/data/users/{elder_id}"
    
    # 1. 调用 qmd query 进行语义检索
    try:
        result = await asyncio.wait_for(
            asyncio.to_thread(
                subprocess.run,
                [
                    "qmd", "query",
                    "--collection", user_collection,
                    "--query", user_message,
                    "--top-k", str(top_k * 3),  # over-fetch 用于重排序
                    "--format", "json"
                ],
                capture_output=True, text=True, cwd=vault_path
            ),
            timeout=0.25  # 250ms 超时，不阻塞对话
        )
        
        if result.returncode != 0:
            logger.warning(f"qmd query failed: {result.stderr}")
            return []
        
        candidates = json.loads(result.stdout)
    except asyncio.TimeoutError:
        logger.warning(f"qmd query timeout for elder={elder_id}, skipping memory injection")
        return []  # 超时则跳过记忆注入，不阻塞对话
    except Exception as e:
        logger.error(f"qmd query error: {e}")
        return []
    
    # 2. 相关性阈值过滤 (relevance > 0.7)
    filtered = [c for c in candidates if c.get("relevance", 0) > 0.7]
    
    # 3. 规则加权重排序
    scored_results = []
    now = datetime.now(tz=timezone_shanghai)
    
    for mem in filtered:
        score = mem["relevance"]  # 基础分: qmd 语义相似度 (0-1)
        source_file = mem.get("source", "")
        
        # 加权规则1: wiki/ 目录的长期记忆加权
        if "/wiki/" in source_file:
            score *= 1.3
        
        # 加权规则2: 情感相关页面在情感关键词命中时加权
        if "emotional" in source_file or "family" in source_file:
            if any(kw in user_message for kw in ["想念", "孤独", "想他", "想她", "一个人"]):
                score *= 1.5
        
        # 加权规则3: raw/ 目录的近期对话加权（时效性）
        if "/raw/" in source_file:
            score *= 1.2
        
        scored_results.append((mem, score))
    
    # 4. 按加权分数排序，取top_k
    scored_results.sort(key=lambda x: x[1], reverse=True)
    return [mem for mem, score in scored_results[:top_k]]
```

```python
async def postprocess_response(
    user_message: str,
    llm_response: str,
    elder_id: str,
    context: DialogueContext,
    retry_count: int = 0
) -> str:
    """
    后处理流水线：长度检查 → 禁止模式 → 情感策略 → 兜底。
    """
    MAX_RETRIES = 3
    
    if retry_count >= MAX_RETRIES:
        return "嗯，我在听。"  # 安全兜底回复
    
    # 检查1: 长度限制
    if len(llm_response) > 100:
        llm_response = await regenerate_with_instruction(
            context, "请更简短地回复，控制在1-3句话以内"
        )
        return await postprocess_response(user_message, llm_response, elder_id, context, retry_count + 1)
    
    # 检查2: 禁止模式扫描
    for pattern in FORBIDDEN_PATTERNS:
        if re.search(pattern, llm_response):
            llm_response = await regenerate_with_instruction(
                context, f"回复中不要包含类似'{pattern}'的表达，用更自然的方式说"
            )
            return await postprocess_response(user_message, llm_response, elder_id, context, retry_count + 1)
    
    # 检查3: 情感策略验证
    is_emotional = any(kw in user_message for kw in EMOTIONAL_KEYWORDS)
    if is_emotional:
        # 检查回复是否在第一句就给解决方案/建议
        first_sentence = llm_response.split("。")[0] if "。" in llm_response else llm_response
        solution_patterns = [r"建议", r"试试", r"不如", r"要不要", r"可以去"]
        has_premature_solution = any(re.search(p, first_sentence) for p in solution_patterns)
        
        if has_premature_solution:
            llm_response = await regenerate_with_instruction(
                context, "对方在表达情绪，请先接住情绪（共情），不要急着给建议。第一句话应该是理解和陪伴。"
            )
            return await postprocess_response(user_message, llm_response, elder_id, context, retry_count + 1)
    
    return llm_response
```

### 3.6 Session管理

```python
SESSION_TIMEOUT_MINUTES = 30

async def get_or_create_session(elder_id: str, channel: str) -> Session:
    """
    获取当前活跃Session，超时则创建新Session。
    """
    # 查找30分钟内的活跃会话
    active_session = await db.fetchone("""
        SELECT * FROM sessions
        WHERE elder_id = $1 AND status = 'active' AND channel = $2
          AND started_at > NOW() - INTERVAL '30 minutes'
        ORDER BY started_at DESC
        LIMIT 1
    """, elder_id, channel)
    
    if active_session:
        # 检查最后一条消息是否在30分钟内
        last_msg = await db.fetchone("""
            SELECT timestamp FROM messages
            WHERE session_id = $1
            ORDER BY timestamp DESC LIMIT 1
        """, active_session.id)
        
        if last_msg and (datetime.now(tz=timezone_shanghai) - last_msg.timestamp) < timedelta(minutes=SESSION_TIMEOUT_MINUTES):
            return active_session
    
    # 结束旧Session，创建新Session
    if active_session:
        await db.execute("""
            UPDATE sessions SET status = 'ended', ended_at = NOW()
            WHERE id = $1
        """, active_session.id)
    
    new_session = await db.fetchone("""
        INSERT INTO sessions (elder_id, channel, status)
        VALUES ($1, $2, 'active')
        RETURNING *
    """, elder_id, channel)
    
    return new_session
```

### 3.8 响应时间优化策略

| 优化手段 | 目标 | 实现方式 |
|---------|------|---------|
| 并行上下文获取 | 减少上下文组装时间 | asyncio.gather并行获取Profile/Memory/History/Pending |
| Elder Profile缓存 | 避免每次查DB | Redis缓存，TTL=10分钟，Profile更新时主动失效 |
| LLM流式响应 | 减少首字延迟 | streaming=True，流式接收+拼接 |
| Webhook立即返回 | 防止企微超时 | 收到消息立即返回200，异步处理后通过API回复 |
| 连接池 | 减少连接建立开销 | DB连接池(min=5, max=20)，Redis连接池 |

**目标响应链路**：
```
收到消息 → 解析(~50ms) → 并行上下文获取(~200ms) → LLM生成(~2-5s) → 后处理(~100ms) → 发送(~200ms)
总计: < 6秒 (远低于30秒要求)
```

### 3.9 错误处理

| 故障场景 | 处理策略 | 用户感知 |
|---------|---------|---------|
| LLM超时 (>15秒) | 截断上下文（去掉最早的记忆）重试一次；仍超时则使用兜底回复 | "抱歉小伴走神了，您再说一次？" |
| LLM返回空/异常 | 使用兜底回复 | "嗯，我在听。" |
| 企微Webhook超时(5秒) | 立即返回200，异步处理后通过发消息API回复 | 延迟几秒收到回复 |
| 记忆检索失败 | 不注入记忆上下文，对话正常进行（质量降级但不中断） | 回复缺少个性化记忆引用 |
| Profile加载失败 | 使用通用System Prompt，不注入个性化信息 | 回复通用化 |
| 数据库写入失败 | 消息入本地队列，延迟写入；不影响回复 | 无感知 |

**记忆层次映射到 PKE**：

| 记忆层次 | PKE 映射 | 存储位置 | 生命周期 |
|---------|---------|---------|----------|
| 瞬时记忆 (instant) | 当前 session 上下文 | 内存 (Redis) | Session 结束即释放 |
| 短期记忆 (short-term) | `raw/` 目录中近期对话文件 | `/data/users/{id}/raw/` | 14天后由 Compile 吸收或淘汰 |
| 长期记忆 (long-term) | `wiki/` 目录中编译知识页 | `/data/users/{id}/wiki/` | 永久（health.md, family.md, preferences.md 等） |
| 情感记忆 (emotional) | wiki/ 中带情感标记的页面 | `/data/users/{id}/wiki/emotional.md` | 永久，带情感强度标记 |

**触发时机**：
- Capture: 每轮对话存储后通过 `tasks.memory.capture` Celery task 触发
- Use: 每次收到消息时同步调用 qmd query（250ms 超时）
- Compile: 每日凌晨3:00 由 Hermes cron 触发 `tasks.memory.compile`
- Governance: Compile 完成后 wiki 更新需经过质量检查（proposal-only 模式）

### 4.2 记忆提取流水线（Capture 阶段）

```python
@app.task(
    bind=True,
    autoretry_for=(Exception,),
    retry_backoff=True,
    retry_backoff_max=600,
    max_retries=5,
    acks_late=True,
    reject_on_worker_lost=True,
)
def capture_memory(self, elder_id: str, session_id: str, messages: list):
    """
    Celery task: tasks.memory.capture
    将对话内容写入 per-user PKE vault 的 raw/ 目录作为 evidence。
    触发时机：每轮对话存储后由 tasks.conversation.store 链式调用。
    """
    vault_path = f"/data/users/{elder_id}"
    raw_dir = f"{vault_path}/raw/"
    
    # 确保目录存在
    os.makedirs(raw_dir, exist_ok=True)
    
    # 生成 evidence 文件名（按日期+session组织）
    today_str = datetime.now(tz=timezone_shanghai).strftime("%Y-%m-%d")
    filename = f"{today_str}_session_{session_id[:8]}.md"
    filepath = os.path.join(raw_dir, filename)
    
    # 格式化对话内容为 Markdown
    content_lines = [
        f"# 对话记录 {today_str}",
        f"\n> session_id: {session_id}",
        f"> elder_id: {elder_id}",
        f"> captured_at: {datetime.now(tz=timezone_shanghai).isoformat()}",
        "\n---\n"
    ]
    
    for msg in messages:
        sender_label = "老人" if msg["sender"] == "elder" else "小伴"
        content_lines.append(f"**{sender_label}** ({msg['timestamp']}): {msg['content']}\n")
    
    # 写入文件（追加模式，同一 session 多轮对话追加到同一文件）
    with open(filepath, "a", encoding="utf-8") as f:
        f.write("\n".join(content_lines) + "\n")
    
    # 更新 qmd 索引（增量索引新写入的文件）
    user_collection = f"user_{elder_id}"
    subprocess.run(
        ["qmd", "index", "--collection", user_collection, "--file", filepath],
        cwd=vault_path, capture_output=True, timeout=30
    )
    
    logger.info(f"Memory captured: elder={elder_id}, file={filename}")
```

### 4.3 知识编译流水线（Compile 阶段）

```python
@app.task(
    bind=True,
    autoretry_for=(Exception,),
    retry_backoff=True,
    retry_backoff_max=600,
    max_retries=3,
    acks_late=True,
)
def compile_memory(self, elder_id: str):
    """
    Celery task: tasks.memory.compile
    每日凌晨3:00 由 Hermes cron 触发（cron: `0 3 * * *`）。
    从 raw/ 目录提取关键知识，编译写入 wiki/ 目录。
    治理模式：proposal-only，wiki 更新需经过质量检查。
    """
    vault_path = f"/data/users/{elder_id}"
    
    # 1. 执行 pke daily（每日知识审查，生成 proposals）
    daily_result = subprocess.run(
        ["pke", "daily", "--vault", vault_path],
        capture_output=True, text=True, timeout=300
    )
    
    if daily_result.returncode != 0:
        logger.error(f"pke daily failed: elder={elder_id}, err={daily_result.stderr}")
        raise Exception(f"pke daily failed: {daily_result.stderr}")
    
    # 2. 执行 pke compile（从 evidence 提炼知识）
    #    编译目标: health, family, preferences, daily_routine, important_dates
    wiki_topics = ["health", "family", "preferences", "daily_routine", "important_dates", "emotional"]
    
    for topic in wiki_topics:
        compile_result = subprocess.run(
            ["pke", "compile", topic, "--vault", vault_path, "--proposal-only"],
            capture_output=True, text=True, timeout=120
        )
        
        if compile_result.returncode == 0:
            logger.info(f"Compiled topic={topic} for elder={elder_id}")
        else:
            logger.warning(f"Compile topic={topic} skipped: {compile_result.stderr}")
    
    # 3. 自动审批高置信度 proposals（confidence > 0.9 自动 apply）
    #    低置信度的保留在 .pke/proposals/ 等待人工审核
    apply_result = subprocess.run(
        ["pke", "proposals", "apply", "--auto", "--min-confidence", "0.9", "--vault", vault_path],
        capture_output=True, text=True, timeout=60
    )
    
    logger.info(f"Memory compile done: elder={elder_id}, result={apply_result.stdout[:200]}")
    
    # 4. 重建 qmd 索引（包含新的 wiki 内容）
    user_collection = f"user_{elder_id}"
    subprocess.run(
        ["qmd", "index", "--collection", user_collection, "--dir", vault_path],
        capture_output=True, timeout=60
    )


# Hermes cron 调度配置（在 scheduler 中注册）
```

### 4.4 记忆治理机制（Governance）

```python
"""
PKE 治理机制说明：

1. Compile 结果不会静默写入 wiki/，而是生成 proposal
2. 高置信度 (>0.9) 的 proposal 自动 apply
3. 低置信度的保留在 .pke/proposals/ 目录，运营后台可视化审核
4. 健康类记忆（health.md）需人工抽检，防止LLM幻觉写入错误用药信息

安全保障：
- 每个 wiki 页面含“Current Understanding / Evidence / Conflicts / Stale Claims / Open Questions”
- 冲突信息不删除旧记录，而是标记为 Stale Claims
- 老人要求“别记了”时，由对话引擎触发 pke forget 命令
"""

async def handle_elder_forget_request(elder_id: str, content_to_forget: str):
    """老人主动要求遍忘某条信息。"""
    vault_path = f"/data/users/{elder_id}"
    user_collection = f"user_{elder_id}"
    
    # 从 wiki 中移除相关内容
    subprocess.run(
        ["pke", "forget", "--content", content_to_forget, "--vault", vault_path],
        capture_output=True, timeout=30
    )
    
    # 重建索引
    subprocess.run(
        ["qmd", "index", "--collection", user_collection, "--dir", vault_path, "--rebuild"],
        capture_output=True, timeout=60
    )
    
    logger.info(f"Memory forgotten: elder={elder_id}, content={content_to_forget[:50]}")
```

### 4.5 理解文档与 PKE wiki 的关系

```python
"""
理解文档（elder_profiles 表）与 PKE wiki 的关系：

- PKE wiki/ 是记忆的“单一事实来源”（source of truth）
- elder_profiles 表是 wiki/ 的“快照”，用于快速加载到 LLM prompt
- 每次 Compile 完成后同步更新 elder_profiles 表

同步流程：
1. tasks.memory.compile 执行完成
2. 读取 wiki/ 目录下所有 .md 文件
3. 拼接为结构化的理解文档
4. 版本化存储到 elder_profiles 表
5. 清除 Redis Profile 缓存
"""

async def sync_wiki_to_profile(elder_id: str):
    """将 PKE wiki 同步到理解文档表。"""
    vault_path = f"/data/users/{elder_id}"
    wiki_dir = f"{vault_path}/wiki/"
    
    if not os.path.exists(wiki_dir):
        return
    
    # 读取所有 wiki 页面
    wiki_content_parts = []
    for md_file in sorted(os.listdir(wiki_dir)):
        if md_file.endswith(".md"):
            filepath = os.path.join(wiki_dir, md_file)
            with open(filepath, "r", encoding="utf-8") as f:
                wiki_content_parts.append(f.read())
    
    if not wiki_content_parts:
        return
    
    combined_content = "\n\n---\n\n".join(wiki_content_parts)
    
    # 版本化存储
    profile = await get_latest_profile(elder_id)
    new_version = (profile.version + 1) if profile else 1
    
    await db.execute("""
        INSERT INTO elder_profiles (elder_id, content, version, last_updated_by)
        VALUES ($1, $2, $3, 'system')
    """, elder_id, combined_content, new_version)
    
    # 清除 Profile 缓存
    await redis.delete(f"profile_cache:{elder_id}")
    
    logger.info(f"Wiki synced to profile: elder={elder_id}, version={new_version}")
```

```python
async def process_task_intent(
    elder_id: str, 
    session_id: str,
    message_id: str,
    task_metadata: Optional[dict]
) -> Optional[TaskResult]:
    """
    根据LLM检测到的任务意图执行相应操作。
    注意：LLM已在natural_reply中包含了对用户的自然语言回复，
    此处仅执行后端操作（存储/API调用等）。
    """
    if task_metadata is None:
        return None
    
    task_type = task_metadata.get("type")
    params = task_metadata.get("params", {})
    
    match task_type:
        case "weather":
            # 天气数据已在上下文组装阶段获取并注入LLM
            # LLM已将天气信息融入自然回复中
            # 此处仅记录任务完成
            return TaskResult(type="weather", status="completed")
        
        case "reminder":
            # 创建提醒
            remind_at = parse_natural_time(params.get("time"), elder_id)
            reminder = await create_reminder(
                elder_id=elder_id,
                content=params.get("content"),
                remind_at=remind_at,
                session_id=session_id
            )
            return TaskResult(type="reminder", status="created", data={"reminder_id": reminder.id})
        
        case "family_message":
            # 将老人留话写入配置者消息队列（反向留话）
            await create_reverse_message(
                elder_id=elder_id,
                content=params.get("message"),
                session_id=session_id
            )
            return TaskResult(type="family_message", status="queued")
        
        case "unsupported":
            # 记录未满足需求
            await log_unmet_need(
                elder_id=elder_id,
                session_id=session_id,
                message_id=message_id,
                description=params.get("description")
            )
            return TaskResult(type="unsupported", status="logged")
    
    return None
```

```python
def parse_natural_time(time_expr: str, elder_id: str) -> datetime:
    """
    解析自然语言时间表达为具体时间戳。
    示例:
      "下午3点" → today 15:00
      "明天早上" → tomorrow 08:00
      "后天晚上8点" → day_after_tomorrow 20:00
      "半小时后" → now + 30min
    """
    now = datetime.now(tz=timezone_shanghai)
    
    # 规则匹配
    if "半小时后" in time_expr:
        return now + timedelta(minutes=30)
    
    # 解析日期部分
    if "明天" in time_expr:
        target_date = now.date() + timedelta(days=1)
    elif "后天" in time_expr:
        target_date = now.date() + timedelta(days=2)
    else:
        target_date = now.date()  # 默认今天
    
    # 解析时间部分
    hour = extract_hour(time_expr)  # 正则提取数字+上下午判断
    if hour is None:
        # 模糊时间映射
        if "早" in time_expr or "上午" in time_expr:
            hour = 8
        elif "中午" in time_expr:
            hour = 12
        elif "下午" in time_expr:
            hour = 15
        elif "晚" in time_expr:
            hour = 20
        else:
            hour = now.hour + 1  # 默认1小时后
    
    return datetime.combine(target_date, time(hour=hour), tzinfo=timezone_shanghai)
```

### 5.5 未满足需求处理

```python
NEED_CATEGORIES = {
    "购物": ["买", "购", "下单", "外卖", "超市"],
    "出行": ["打车", "叫车", "导航", "地图", "公交"],
    "医疗": ["挂号", "预约", "看病", "门诊"],
    "缴费": ["交费", "缴费", "水电", "话费", "充值"],
    "娱乐": ["听歌", "放歌", "看电视", "视频"],
}

def auto_categorize(description: str) -> str:
    """自动分类未满足需求。"""
    for category, keywords in NEED_CATEGORIES.items():
        if any(kw in description for kw in keywords):
            return category
    return "其他"


async def log_unmet_need(
    elder_id: str, 
    session_id: str, 
    message_id: str, 
    description: str
):
    """记录未满足需求到数据库。"""
    category = auto_categorize(description)
    
    # 去重检查：同一老人同一类别7天内不重复记录相似需求
    existing = await db.fetchone("""
        SELECT id FROM unmet_needs
        WHERE elder_id = $1 AND category = $2 AND status = 'confirmed'
          AND created_at > NOW() - INTERVAL '7 days'
          AND need_description = $3
    """, elder_id, category, description)
    
    if existing:
        return  # 已存在，不重复记录
```

```python
SIGNAL_EXTRACTION_PROMPT = """
分析以下对话轮次的健康信号和情绪信号。

注意:
- 日常对话（如聊天气、聊吃饭）情绪记为中性(0.0-0.3)

- 紧急情况仅限: 明确求助、描述严重身体症状、表达伤害意图

async def extract_guardian_signals(elder_id: str, session_id: str, messages: List[Message]):

    从对话中提取守护信号。
    每轮对话结束后由 tasks.guardian.analyze Celery task 触发。

    elder_msg = next((m for m in messages if m.sender == 'elder'), None)
    xiaoban_msg = next((m for m in messages if m.sender == 'xiaoban'), None)
    
    if not elder_msg:
        return  # 没有老人消息则跳过
    
    # 调用LLM分析
    prompt = SIGNAL_EXTRACTION_PROMPT.format(
        elder_message=elder_msg.content,
        xiaoban_reply=xiaoban_msg.content if xiaoban_msg else "（无回复）",
        recent_emotion_avg=f"{recent_avg:.2f}" if recent_avg else "数据不足",
        elder_brief=elder_brief
    )
    
    result = await llm_client.chat(
        model="qwen-max",
        messages=[{"role": "user", "content": prompt}],
        response_format="json",
        temperature=0.2
    )
    
    signal_data = json.loads(result.content)
    
    # 存储信号
    await db.execute("""

         json.dumps(signal_data["health_mentions"]),
         signal_data["emergency_indicators"],
```

### 7.3 紧急检测算法

```python
# 紧急关键词正则模式
EMERGENCY_PATTERNS = [
    r"胸[口闷]?[疼痛]",
    r"摔倒了?",
    r"站不[稳起来]",
    r"喘不[上过来]气",
    r"(打|拨|叫).{0,3}120",
    r"救护车",
    r"心[脏]?不舒服",
    r"头[晕疼]得厉害",
    r"眼[前睛]发[黑花]",
    r"帮我[打叫]电话",
    r"(手脚|半边|一边).{0,3}(麻|没[感力])",
    r"[吐血出]了[好很]?多血",
    r"意识[模不]",
    r"不[能想]动了?",
]

class EmergencyLevel(Enum):
    NONE = "none"
    SUSPECTED = "suspected"
    CONFIRMED = "confirmed"

async def detect_emergency(
    elder_id: str, 
    message: Message, 
    llm_analysis: dict
) -> EmergencyLevel:
    """
    三级紧急检测:
    Level 1: 关键词快速匹配 (毫秒级)
    Level 2: LLM分类 (已在信号提取中完成)
    Level 3: 综合判定 + 二次确认
    """
    # Level 1: 关键词匹配
    pattern_match = any(
        re.search(p, message.content) for p in EMERGENCY_PATTERNS
    )
    
    # Level 2: LLM判定 (来自信号提取)
    llm_emergency = llm_analysis.get("is_emergency", False)
    
    # Level 3: 综合判定
    if pattern_match and llm_emergency:
        # 双重确认: 高置信度 → 直接确认
        return EmergencyLevel.CONFIRMED
    
    elif pattern_match or llm_emergency:
        # 单一信号: 需要二次确认
        return EmergencyLevel.SUSPECTED
    
    return EmergencyLevel.NONE


async def handle_potential_emergency(elder_id: str, message: Message, signal_data: dict):
    """处理潜在紧急情况。"""
    level = await detect_emergency(elder_id, message, signal_data)
    
    if level == EmergencyLevel.NONE:
        return
    
    if level == EmergencyLevel.SUSPECTED:
        # 二次确认: 获取更多上下文后再次LLM判断
        recent_messages = await get_recent_messages(elder_id, limit=5)
        context = "\n".join([f"{m.sender}: {m.content}" for m in recent_messages])
        
        second_check = await llm_client.chat(
            model="qwen-max",
            messages=[{"role": "user", "content": f"""
                判断以下对话中老人是否处于真正的紧急状态（需要立即联系家人/120）。
                仅回答"是"或"否"。
                
                对话上下文:
                {context}
                
                注意：如果是在讲述过去的事件、开玩笑、或轻微不适，不算紧急。
            """}],
            max_tokens=10,
            temperature=0.1
        )
        
        if "否" in second_check.content:
            return  # 非紧急，不告警
        
        level = EmergencyLevel.CONFIRMED
    
    if level == EmergencyLevel.CONFIRMED:
        # 确认紧急 → 立即通知
        await send_emergency_alert(elder_id, message, signal_data)
```

```python
async def evaluate_emotion_trend(elder_id: str):
    """
    每日定时评估情绪趋势。
    由 Daily Evaluator (每日10AM) 调用。
    """
    # 获取近7天信号
    signals = await db.fetch("""
        SELECT emotion_score, emotion_keywords, analyzed_at::date as signal_date
        FROM guardian_signals
        WHERE elder_id = $1 AND analyzed_at > NOW() - INTERVAL '7 days'
        ORDER BY analyzed_at DESC
    """, elder_id)
    
    if len(signals) < 3:
        return  # 数据不足，不评估
    
    # 计算近3天每日平均情绪分
    from collections import defaultdict
    daily_scores = defaultdict(list)
    for s in signals:
        daily_scores[s.signal_date].append(s.emotion_score)
    
    # 取最近3天有数据的日期
    recent_dates = sorted(daily_scores.keys(), reverse=True)[:3]
    
    if len(recent_dates) < 3:
        return  # 不足3天数据
    
    # 计算均值
    daily_avgs = [mean(daily_scores[d]) for d in recent_dates]
    overall_avg = mean(daily_avgs)
    
    # 触发条件: 连续3天平均情绪 < -0.3
    if overall_avg < -0.3 and all(avg < -0.3 for avg in daily_avgs):
        # 检查是否已在最近3天内发送过情感关怀通知（避免重复）
        recent_alert = await db.fetchone("""
            SELECT id FROM emergency_alerts
            WHERE elder_id = $1 AND severity = 'emotional_concern'
              AND created_at > NOW() - INTERVAL '3 days'
        """, elder_id)
        
        if not recent_alert:
            await send_emotional_concern_alert(elder_id, overall_avg, daily_avgs)
```

```python
async def check_anomalies(elder_id: str):
    """
    每日异常行为检测。
    由 Daily Evaluator 调用。
    """
    anomalies = []
    
    # ---- 规则1: 不活跃异常 ----
    # 如果老人通常每天都互动，但已超48小时无响应
    last_interaction = await db.fetchone("""
        SELECT timestamp FROM messages
        WHERE elder_id = $1 AND sender = 'elder'
        ORDER BY timestamp DESC LIMIT 1
    """, elder_id)
    
    # 计算过去14天的平均每日互动次数
    baseline = await db.fetchval("""
        SELECT COUNT(*)::float / 14
        FROM messages
        WHERE elder_id = $1 AND sender = 'elder'
          AND timestamp > NOW() - INTERVAL '14 days'
    """, elder_id)
    
    if last_interaction and baseline > 0.5:
        gap_hours = (datetime.now(tz=timezone_shanghai) - last_interaction.timestamp).total_seconds() / 3600
        if gap_hours > 48:
            anomalies.append({
                "type": "inactivity",
                "severity": "low",
                "detail": f"已{int(gap_hours)}小时无互动（基线: 日均{baseline:.1f}条消息）"
            })
    
    # ---- 规则2: 作息异常 ----
    # 检查近3天是否有异常时段活跃
    recent_hours = await db.fetch("""
        SELECT EXTRACT(HOUR FROM timestamp)::int as hour
        FROM messages
        WHERE elder_id = $1 AND sender = 'elder'
          AND timestamp > NOW() - INTERVAL '3 days'
    """, elder_id)
    
    # 统计深夜活跃 (0:00-5:00)
    night_activity = sum(1 for h in recent_hours if h.hour < 5)
    if night_activity >= 3:  # 3天内深夜活跃3次以上
        anomalies.append({
            "type": "routine_disruption",
            "severity": "low",
            "detail": f"近3天深夜(0-5点)活跃{night_activity}次"
        })
    
    # ---- 规则3: 话量骤降 ----
    recent_3d_count = await db.fetchval("""
        SELECT COUNT(*) FROM messages
        WHERE elder_id = $1 AND sender = 'elder'
          AND timestamp > NOW() - INTERVAL '3 days'
    """, elder_id)
    
    baseline_3d_count = await db.fetchval("""
        SELECT COUNT(*)::float / 4.67  -- 14天/3天
        FROM messages
        WHERE elder_id = $1 AND sender = 'elder'
          AND timestamp BETWEEN NOW() - INTERVAL '14 days' AND NOW() - INTERVAL '3 days'
    """, elder_id)
    
    if baseline_3d_count > 3 and recent_3d_count < baseline_3d_count * 0.3:
        anomalies.append({
            "type": "volume_drop",
            "severity": "medium",
            "detail": f"话量骤降: 近3天{recent_3d_count}条 vs 基线{baseline_3d_count:.0f}条"
        })
    
    # 记录异常（低严重度仅记录日志，中度通知运营）
    for anomaly in anomalies:
        logger.info(f"异常检测 elder={elder_id}: {anomaly}")
        if anomaly["severity"] == "medium":
            # 通知运营团队（非紧急，不通知家人）
            await notify_ops_team(elder_id, anomaly)
```

| LLM信号分析失败 | 跳过本轮分析，不影响对话，下一轮继续 | 低（数据延迟但不丢失） |
| 紧急通知发送失败(微信) | 重试3次 → 降级为SMS短信通知 → 仍失败则告警运营 | 高（必须送达） |
| 情感关怀通知发送失败 | 重试3次，失败则记录日志，次日重新评估 | 中（非紧急可容忍延迟） |
| 误报情况 | 配置者可标记为误报，系统记录用于优化检测模型 | 低 |
| 数据库写入信号失败 | 本地日志记录原始信号，恢复后补录 | 低 |
| Daily Evaluator超时 | 记录已处理数量，未处理老人次日补评估 | 低 |

### 8.4 关键数据一致性保障

| 场景 | 一致性策略 |
|------|----------|
| 对话消息存储 → 异步处理 | 先持久化到messages表（强一致），再发 Celery task 链（最终一致）；task 失败自动重试不影响已存储消息 |
| 记忆更新 → Profile缓存 | 记忆写入后主动清除Redis缓存key，下次读取时重新加载 |
| 触发消息发送 → 状态更新 | 先发送成功，再更新trigger_state和trigger_logs；发送失败则状态不变 |
| 紧急通知 → 多渠道兜底 | 微信发送失败 → SMS兜底 → 运营告警；alert表记录所有尝试 |
| Session超时判定 | 基于messages最后时间戳，非系统时钟；避免时钟偏移问题 |

## 九、非功能性要求

### 9.1 性能指标

| 指标 | 目标值 | 测量方式 |
|------|--------|--------|
| 对话响应时间(P95) | < 6秒 | 从收到企微回调到发送回复 |
| 触发评估周期 | < 5分钟/全量 | 单次评估所有活跃老人 |
| 紧急通知延迟 | < 2分钟 | 从检测到紧急到配置者收到通知 |
| 记忆检索延迟 | < 250ms | qmd query 本地语义检索（超时则跳过） |
| 通话端到端延迟 | < 500ms | ASR+LLM+TTS全链路 |
| 电话接通率 | > 80% | 发起呼叫到老人接听 |

### 9.2 可用性要求

| 服务 | 可用性目标 | 降级策略 |
|------|-----------|--------|
| 对话服务 | 99.9% | LLM供应商切换；记忆降级；兜底回复 |
| 触发服务 | 99% | 单次失败下周期重试，老人无感知 |
| 守护系统 | 99.9%（紧急检测） | 关键词匹配兜底（不依赖LLM） |
| 电话服务 | 95% | 失败则次日重试 |

### 9.3 安全要求

| 要求 | 实现方式 |
|------|--------|
| 对话数据加密存储 | PostgreSQL TDE (透明数据加密) |
| 传输加密 | 所有外部通信 TLS 1.3 |
| 内部服务认证 | X-Internal-Key 共享密钥 |
| 隐私数据隔离 | 老人对话内容绝不暴露给配置者端API |
| 日志脱敏 | 日志中老人消息内容截断/掩码处理 |
| 数据访问审计 | 运营后台所有查看操作记录审计日志 |
