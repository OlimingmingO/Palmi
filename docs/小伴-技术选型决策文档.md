## 一、选型背景与约束

### 1.1 产品技术需求概要

小伴是一款面向60-75岁居家老人的AI陪伴助手，Phase 1通过企业微信提供服务。技术选型需满足以下核心需求：

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 企业微信消息收发 | P0（生死线） | 主动发消息+接收回复，WebSocket或回调方式 |
| 长期记忆系统 | P0（核心差异化） | 记住每个老人说过的一切，对话中自然调用 |
| 多租户支持 | P0 | 同时服务20→1000+个老人，每人独立记忆/上下文 |
| 主动触达（"猫式存在"） | P0 | 基于触发器（天气/记忆/时间缺口等）主动发消息 |
| 运营后台 | P0 | 对话浏览、意图标签、未满足需求识别 |
| 中文大模型接入 | P0 | 通义千问/DeepSeek，中文对话质量第一 |

### 1.2 团队资源约束

- 纯起步阶段，1个强全栈工程师（CEO本人）
- 无外部资金，需控制开发周期在可接受范围内
- Python为首选语言（CEO技术背景+中国AI生态Python优先）

## 三、源码级对比分析（Hermes Agent vs OpenClaw）

### 3.1 项目基本面

| 维度 | Hermes Agent | OpenClaw |
|------|-------------|----------|
| 语言 | **Python** | TypeScript/Node.js + Swift + Kotlin |
| 仓库大小 | 212MB | 465MB |
| 文件总数 | 3,346（Python 1,629个） | 13,897（TS/JS 11,340个） |
| 架构模式 | Gateway + Platform Adapters | Plugin SDK + Extensions |
| 存储方案 | SQLite(WAL) + FTS5全文搜索 + 文件 | 文件 + LanceDB向量库 + 嵌入式引擎 |

| 设计目标 | 个人AI助手（多平台消息网关） | 个人AI操作系统（本地优先，原生桌面/移动端） |

### 3.2 企业微信适配器（MVP生死线）

**Hermes Agent：✅ 完整就绪，配置即用**

源码审查确认的文件清单：
```
gateway/platforms/wecom.py           — 65KB，WebSocket网关完整实现
gateway/platforms/wecom_callback.py  — 16KB，HTTP回调模式
gateway/platforms/wecom_crypto.py    — 5KB，消息加解密
gateway/platforms/weixin.py          — 82KB，微信个人号适配器（额外彩蛋）
tests/gateway/test_wecom.py          — 单元测试覆盖
website/docs/user-guide/messaging/wecom.md — 完整用户文档
```

配置极简（两个环境变量即可启用）：
```yaml
platforms:
  wecom:
    enabled: true
    extra:
      bot_id: "your-bot-id"    # 或 WECOM_BOT_ID 环境变量
      secret: "your-secret"    # 或 WECOM_SECRET 环境变量
```

已实现能力：WebSocket长连接、自动重连（backoff [2,5,10,30,60]秒）、30秒心跳保活、消息去重（DEDUP_MAX_SIZE=1000）、图片/文件/语音上传（分chunk，512KB/chunk）、DM策略（open/allowlist/pairing）、群聊策略。

**OpenClaw：❌ 不存在**

extensions目录中有飞书(feishu)和QQ(qqbot)，但无wecom/wechat扩展。从零开发需3-4周。

**结论：仅此一项即可决策。企微适配器是MVP的生死线，Hermes直接省掉3-4周。**

### 3.4 记忆系统

**Hermes Agent — 基础但可扩展：**

```
tools/memory_tool.py     — MEMORY.md + USER.md 双存储
agent/memory_manager.py  — 记忆管理器（MemoryProvider调度）
agent/memory_provider.py — 抽象基类（预留插件扩展点）
plugins/memory/honcho/   — Honcho用户建模插件示例
```

设计特点：
- 文件级持久化（`§`分隔符分条目）
- 字符级容量限制（MEMORY.md约2,200字符，USER.md约1,375字符）
- 会话内冻结：运行时更新落盘但不刷新system prompt（保持prefix cache）
- MemoryProvider抽象基类预留了清晰的扩展点

**限制：容量太小，不满足"记住一切"需求。但有明确的扩展路径。**

**OpenClaw — 极其强大的四层记忆架构：**

```
extensions/memory-core/      — 核心记忆引擎（Dreaming机制）
extensions/memory-lancedb/   — LanceDB向量存储
extensions/memory-wiki/      — Wiki式结构化记忆
extensions/active-memory/    — 主动记忆检索注入
packages/memory-host-sdk/    — 嵌入式记忆SDK（60+文件）
```

**Dreaming机制（类人睡眠记忆巩固）：**
- Light Sleep：轻度记忆整理，从当天对话提取关键信号
- REM Sleep：深度语义关联，跨session建立连接
- Deep Sleep：长期记忆巩固，去重/合并/淘汰
- 可配置cron频率（如`0 3 * * *`每天凌晨3点执行）

**Active Memory（主动记忆注入）：**
- 每次对话前，自动语义搜索相关记忆注入prompt context
- 支持多种query模式（message/recent/full）
- 支持多种prompt风格（balanced/strict/recall-heavy/precision-heavy等）
- 可配置timeout（250ms-120s）、缓存TTL

**结论：OpenClaw记忆系统设计远超Hermes。这是它最大的优势，也是我们决定使用自研PKE的参照系。**

### 3.5 主动消息/定时触发

**Hermes Agent：✅ 完整Cron系统**
```
cron/scheduler.py      — 77KB，定时调度器（文件锁+prompt注入扫描）
cron/jobs.py           — 41KB，Job管理
tools/cronjob_tools.py — 32KB，Agent可通过tool call自主创建Cron任务
```

- 支持deliver到指定平台/频道
- 每60秒tick检查due jobs
- 支持per-job toolset配置
- Agent运行时可自主创建/管理定时任务

直接对应小伴需求：早安问候、用药提醒、主动关心、定时记忆巩固（Dreaming）。

**OpenClaw：** Dreaming有内置cron（记忆巩固用），但无独立的用户级定时消息系统。

**结论：Hermes的Cron系统几乎为"主动关心老人"量身定做。**

### 3.6 多租户/多用户

两者都是单用户设计，但改造难度不同：

**Hermes Agent改造路径（更清晰）：**
- Gateway架构天然支持多平台多session
- 已有`group_sessions_per_user: true`按用户隔离session的概念
- SQLite WAL模式支持并发读写
- 改造方向：session_key加入user_id维度→per-user记忆空间→per-user system prompt

**OpenClaw改造路径（代价更高）：**
- 本地优先设计（Bonjour设备发现、桌面进程管理）
- 完全不具备多租户概念
- 改成云端服务需去掉整个桌面/移动端壳 + 重建session管理

## 四、自研PKE作为记忆系统底座

### 4.1 Personal Knowledge Engine简介

CEO自研的本地知识引擎，已实现MVP并在个人使用中验证。

**核心架构（三段循环）：**
```
Capture（捕获证据） → Compile（编译知识） → Use（自然使用）
```

**已实现能力：**
- `pke use "question"` — 语义检索知识库回答问题
- `pke capture` — 将新信息作为evidence存入raw/目录
- `pke compile "topic"` — 从evidence提炼知识写入wiki/
- `pke daily` — 每日知识审查（proposal-only模式）
- `pke monitor --watch` — 文件变更实时监测
- `pke proposals/apply/reject` — 知识更新治理流程
- 语义检索（qmd query）— 基于embedding的向量搜索

**技术栈：**
- Node.js CLI（`/bin/pke`）
- qmd（MinerU Document Explorer）— 本地语义索引和检索
- Vault结构：`raw/`（证据）+ `wiki/`（知识）+ `.pke/`（状态）
- 每个wiki页面含：Current Understanding / Evidence / Conflicts / Stale Claims / Open Questions

### 4.2 PKE与小伴记忆需求的映射

| 小伴记忆需求 | PKE对应能力 | 映射方式 |
|-------------|-----------|---------|
| 对话后自动提取记忆 | `pke capture` | 每轮对话结束→自动将对话写入per-user的raw/目录 |
| 长期记忆巩固 | `pke daily` + `pke compile` | 每日凌晨对每个活跃用户执行，从对话中提取持久知识到wiki/ |
| 对话前语义检索（Active Memory） | `pke use` / `qmd query` | 收到消息→qmd query检索相关记忆→注入prompt |
| 记忆不被静默污染 | PKE治理机制（proposal→approval→apply） | wiki更新需要明确触发，不会因为LLM幻觉写入错误记忆 |
| 老人信息优先级管理 | wiki结构化页面（Current Understanding / Stale Claims） | 最新信息更新Current Understanding，旧信息降级不删除 |

### 4.3 PKE相比直接复用OpenClaw记忆的优势

| 维度 | 自研PKE | 直接用OpenClaw记忆模块 |
|------|---------|---------------------|
| 技术掌控 | 完全掌控每行代码，出问题能修 | 依赖TS生态，调试成本高 |
| 治理机制 | 内置wiki写入gate（proposal-only模式），不会silently污染知识库 | Dreaming机制自动写入，无人工审批环节 |
| 多租户适配 | 每人一个vault（文件级隔离），PKE逻辑不用改 | 需要从单用户本地改为多用户云端 |
| 可审阅性 | Markdown文件，人类可读可编辑 | LanceDB向量库，不透明 |
| 语言一致性 | Node.js（可通过子进程调用或改写为Python） | TypeScript，需要在Python架构中嵌入Node运行时 |
| 改造工作量 | 5-6周 | 8-12周（含TypeScript学习+架构重构） |

### 4.4 PKE多租户改造方案

**方案：文件级隔离（MVP阶段首选）**

```
/data/users/
├── user_001/
│   ├── raw/           ← 该老人的对话日志（evidence）
│   ├── wiki/          ← 该老人的持久知识（理解文档）
│   └── .pke/         ← 该老人的PKE状态
├── user_002/
│   ├── raw/
│   ├── wiki/
│   └── .pke/
└── ...
```

每个用户一个独立vault + 独立qmd collection。PKE的所有命令在user目录scope内执行，逻辑几乎不需要修改。外层套一个调度器遍历活跃用户即可。

**100用户以内完全撑得住。** 千级以上再考虑迁移到数据库（PostgreSQL+pgvector）。

### 4.5 PKE改造工作量拆解

| 改造项 | 工作量 | 说明 |
|--------|--------|------|
| Per-user vault管理 | 1周 | 用户注册时自动创建vault，qmd init collection |
| 对话自动Capture | 1周 | 每轮对话结束→异步写入用户raw/目录 |
| 定时Compile调度 | 1-2周 | Cron系统每日凌晨对每个活跃用户执行pke daily + compile，需控制并发 |
| 对话前Use注入（Active Memory） | 1周 | 收到消息→qmd query搜索→Top 5结果注入prompt |
| 运营可视化 | 1周 | 在运营后台展示每个用户的wiki/状态、compile结果 |
| **合计** | **5-6周** | |

### 5.2 决策理由

**为什么选Hermes Agent做底座（而非OpenClaw）：**

1. **企微适配器直接可用** — 65KB的wecom.py包含WebSocket长连接、消息加解密、图片语音上传、重连机制，配置两个环境变量即可跑通。OpenClaw没有，从零写要3-4周。
2. **Python语言** — 中国AI/NLP生态Python优先，CEO技术栈匹配，招人容易。
3. **复杂度可控** — 3,346文件 vs 13,897文件。Hermes"刚好够用"，OpenClaw"远远过剩"。
4. **架构匹配度高** — Gateway(消息通道) + Cron(主动触发) + Session(多用户) + SQLite(存储)，四个模块直接对应小伴核心需求。
5. **改造成本更低** — 14-20周 vs 18-26周（OpenClaw还需要学TypeScript生态+去掉桌面壳+重建云端架构）。

**为什么用自研PKE做记忆（而非Hermes原生记忆或OpenClaw记忆）：**

1. **容量突破** — Hermes原生记忆仅2,200字符，连一个老人的基本画像都存不下。PKE无硬性上限。
2. **治理机制** — PKE的wiki更新需要明确触发（proposal-only模式），不会因LLM幻觉静默写入错误记忆。这在老年人健康场景尤为重要——错误记忆可能导致错误的用药提醒。
3. **技术壁垒** — 自研技术是竞争壁垒。完全掌控代码，出问题能修，能按产品需求自由演进。
4. **多租户天然适配** — 文件级隔离（每人一个vault），PKE逻辑几乎不需要修改。
5. **Markdown可审阅** — 运营团队可以直接阅读任何用户的记忆wiki，不需要专门工具。
6. **三段循环精确匹配** — Capture(对话存证) → Compile(知识提炼) → Use(对话注入)，与小伴记忆需求一一对应。

**为什么不完全自建（方案C）：**

- Hermes的企微适配器+Cron系统直接省掉5-6周工作量
- Gateway的多平台抽象、Session管理、消息路由都是成熟代码，自己写容易出bug
- 不需要维护WebSocket重连、心跳、消息去重这些底层通信逻辑

## 八、技术风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 企微WebSocket连接不稳定 | 中 | 高 | Hermes已有reconnect backoff([2,5,10,30,60]秒)+30秒心跳；监控断连频率 |
| 通义千问API延迟波动 | 中 | 高 | 流式输出+DeepSeek备用+预设快速回复模板（"让我想想"占位） |
| qmd语义检索中文效果不佳 | 中 | 中 | 配置中文embedding模型（Qwen embedding优于默认英文模型）；Phase 2可升级pgvector |
| PKE per-user vault文件数过多（>500用户） | 低 | 中 | 文件系统层面用user_id前两位做目录分片；千级以上迁移数据库 |
| 大模型幻觉写入错误记忆 | 高 | 高 | PKE治理机制（Compile结果需经过质量检查）；健康类记忆人工抽检 |
| Hermes Agent上游更新导致Fork冲突 | 低 | 低 | 只改造必要模块，保持与上游目录结构一致便于cherry-pick |
| 单人维护代码量大 | 高 | 中 | 严格控制Phase 0-1 scope；不做不必要的重构；优先验证假设 |
